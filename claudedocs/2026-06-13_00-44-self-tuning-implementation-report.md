Дата: 2026-06-13. Выполнен основной план (шаги 1–10 из §11) документа
`claudedocs/self-tuning-infrastructure.md`. Работа велась в worktree, ветка `worktree-tuning-infra`,
коммит `1a30501` (версия пакета поднята pre-commit-хуком до 1.0.108).

Проверки: oxlint чистый; полный `npm test` — 176 из 176; профильные наборы `npm run test:llm-log`
и `npm run test:llm-log-db` зелёные; каждый компонент дополнительно проверен живым смоук-прогоном.

## Что построено

### 1. Экспорт лога LLM — `scripts/llm-log-export.js`

Послойный доступ к `log.llm_request` по принципу экономии контекста (§2a): без флагов выводится только
компактная таблица метаданных (id, время, вид, модель, токены, цена, статус); полное содержимое — строго
адресно по `--id`, с выбором частей записи (`--fields payload.messages,response`) и обрезкой длинных
значений (`--max-chars`). Поиск по содержимому (`--grep`) выполняется внутри PostgreSQL и возвращает только
идентификаторы и фрагмент совпадения — сами payload в вывод не попадают. Флаг `--full` без `--id` требует
явного `--limit` (защита от «оптовой» выгрузки).

### 2. Привязка записей лога к версии кода и промпта

Миграция `migrations-log/002_log_tuning_meta.sql` добавила в `log.llm_request` колонки `git_commit`
и `prompt_hash`. Обе заполняются централизованно в `buildRecord` (`src/pipeline/llm-log.js`):
короткий хеш HEAD берётся один раз при старте процесса через `getBotBuildInfo`, а `prompt_hash` —
SHA-256 (первые 16 hex-символов) конкатенации system-сообщений запроса. Места вызовов менять не пришлось.
Заполнение проверено на живых записях.

### 3. Неинтерактивный прогон сценариев

`scripts/run-scenario.js` плюс общий модуль `scripts/lib/scenario-runner.js` (его же использует `eval.js`).
Реплики сценария идут через тот же `handleMessage`, что и Telegram, — полный пайплайн: классификация,
память, компрессия истории, инструменты, извлечение фактов. Поддерживается ход `scheduler_tick`.
Жёстко требуется `NODE_ENV=test` (скрипт выставляет его сам): тестовый пользователь и записи лога получают
флаг `is_test`. Артефакты двухслойные: `transcript.summary.md` (короткая сводка для чтения в контекст)
и `transcript.full.json` (полные ответы и список LLM-вызовов каждого хода). Флаг `--keep-user` оставляет
пользователя для ручного дослеживания. Сценарии лежат в `tests/scenarios/*.json`; созданы два:
`facts-and-recall` и `style-preference`.

### 4. Эталонные наборы (детерминированные)

- `tests/eval/classify_cases.json` — 12 кейсов классификатора намерений: проверка выбранного навыка
  (`expect_skill`), областей памяти (`expect_scopes_include`), флага `expect_needs_memory`; есть кейс
  с follow-up по контексту диалога.
- Факты — канонический `tests/memory_cases.json` (тот же, что использует `npm test`); логика вердикта
  совпадает с `layerExtraction` из `tests/run.js`.

### 5. Критерии качества — `tests/eval/criteria.yaml`

Единый источник истины для прогонщика, судьи и отчёта сравнения: пороги детерминированных наборов,
оси/шкалы/минимумы/веса судьи, допуски приёмки (`acceptance`), стоп-лимит стоимости прогона (`budget`).
Формулировки осей — в отдельных файлах рубрик `tests/eval/rubrics/<ось>.md` (relevance, memory_usage,
persona, format, no_hallucinations).

### 6. LLM-судья — `scripts/eval/judge.js`

Промпт судьи собирается из файлов рубрик; оси и шкалы приходят из критериев — в коде ничего не зашито.
Вызовы идут через штатный `chatJSON` со строгой JSON-схемой и логируются отдельным видом запроса
`eval_judge` (добавлен в `REQUEST_KINDS`), чтобы их стоимость была видна отдельно от боевых запросов.
Промпт зарегистрирован в каноническом реестре `docs/prompt-inventory.md`.

### 7. Прогонщик оценок — `scripts/eval.js`

Наборы classify / facts / dialog (`--suite all`), повторы `--repeat N` (детерминированные кейсы — по
большинству, баллы судьи — медиана), ограничение `--scenario`, метка эксперимента `--label`.
Стоп-лимит стоимости из `criteria.yaml` проверяется после каждого кейса; при превышении частичные
результаты сохраняются. Результаты — `claudedocs/experiments/<дата>-<метка>/`: `summary.json`
(агрегаты и строка на кейс, без содержимого) и `cases/<id>.json` (детали адресно). В stdout — только
агрегаты и список провалившихся кейсов. Флаг `--deterministic` включает `EVAL_TEMPERATURE=0`.

### 8. Сравнение прогонов — `scripts/eval-compare.js`

Отчёт «было/стало» от компактного к детальному: агрегатные дельты (доли пройденных, баллы осей, цена хода,
p95 задержки), таблица только изменившихся кейсов, рекомендация по порогам приёмки из `criteria.yaml`.
Диффы фактических выходов — в отдельном `diffs.md`, читается адресно. Файлы пишутся в каталог
прогона-кандидата.

### 9. Регламент — раздел в `AGENTS.md`

Короткая карта тюнинга: шпаргалка команд, ссылка на методику и жёсткие правила (только `NODE_ENV=test`,
один эксперимент — одно изменение, критерии и эталоны меняются отдельной веткой, слияние — решение
человека, дисциплина экономии контекста).

### 10. Skill `tune-bot` — `.claude/skills/tune-bot/SKILL.md`

Вся методика в одном месте (создан через протокол `/edit-claude-files`): протокол возобновления
(«сначала прочитай `state.md`»), полный цикл итерации с командами, протокол чекпоинтов, обязательные
правила экономии контекста, ограничители, типовые итерации («улучшить промпт одного request_kind»,
«снизить цену цикла», «разобрать регрессию по жалобе»), карта файлов.

## Решения и отклонения от плана

- **Температура 0 — опционально.** Мера §4.2 реализована флагом `--deterministic`
  (переменная `EVAL_TEMPERATURE`, читается в `chatJSON`), но по умолчанию выключена: часть
  провайдеров/моделей отвергает нестандартную температуру, и дефолт «включено» валил бы прогон целиком.
- **`prompt_hash` считается централизованно** в `buildRecord` из system-сообщений payload, а не «в месте
  вызова», как в эскизе §3.2: одно место вместо шестнадцати, ни один вызов менять не пришлось.
- **Наборы topic_cases и compress_cases не создавались** — §11 ограничивает шаг 4 классификатором
  и фактами; архитектура `eval.js` позволяет добавить их как новые наборы в `criteria.yaml`.
- **Попутно починены три поломки, существовавшие в master до этих правок:**
  `tests/log-analysis.test.mjs` ожидал русские сообщения об ошибках, ранее переведённые на английский
  (две проверки), а `tests/llm-log-integration.test.mjs` ссылался на удалённый скрипт
  `scripts/migrate-llm-log-db.js` — заменено на идемпотентный прогон `src/migrate.js`.
- **В worktree скопированы локальные секреты** (`config/local.yaml`, `.env`, `.mcp.json`) — git их
  не отслеживает, в коммит они не попали; нужны были для живых прогонов.
