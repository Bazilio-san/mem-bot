# Предложение: агентский инструментарий создания и редактирования Skills (только для администраторов)

## 1. Цель

Сейчас skill — это файловый пакет (`skills/<name>/SKILL.md` плюс `domain-schema.json` и `references/`), который человек
создаёт и правит вручную в редакторе. Это надёжно для ревью и git, но медленно и требует знания формата: структуры
фронтматтера, обязательных блоков, мета-схемы определения домена, правил канонизации `entity_key`.

Предлагается агентский инструментарий, который делает создание и редактирование skills интеллектуальной операцией под
управлением модели. Администратор описывает на естественном языке, какой нужен навык или какую часть существующего навыка
изменить, а инструментарий с помощью LLM генерирует или правит нужные части пакета, проверяет их и сохраняет на диск с
горячей перезагрузкой реестра. Любой предметный кусок навыка — признаки классификации, prompt ответа, prompt извлечения,
закрытая схема доменной памяти, словари `entity_key`, список инструментов и справочники — становится редактируемым через
диалог, а не через ручную правку файла.

Весь инструментарий доступен **только администраторам** (пользователь с `is_admin = true` в таблице `mem.users`).

## 2. Текущее состояние проекта

Опорные точки, на которых строится инструментарий, уже есть:

- Реестр skills `src/pipeline/skills/registry.js`: `loadSkills`, `listSkillRoutes`, `getSkill`, `getSkillByDomain`,
  `getDomainSchema`, `getReference`, `invalidateSkillsCache`. Это чтение и кэш; записи в нём пока нет.
- Разбор файла `src/pipeline/skills/parse.js`: `splitSkillFile`, `parseFrontmatter`, `extractSection`,
  `extractJsonBlock`. Обратной операции (сборки `SKILL.md` из частей) пока нет.
- Мета-валидатор определения домена `src/schema/meta.js` (`validateDefinition`) и применение схемы при записи факта
  `src/schema/validate.js` (`validateAndCanonicalize`).
- Командная строка `src/pipeline/skills/cli.js` (`validate | list | sync`) — ручные операции вне диалога.
- Реестр инструментов агента `src/pipeline/tools.js` с гарантированной проверкой прав: `executeTool` отклоняет вызов
  инструмента с `requiresAdmin = true`, если `ctx.isAdmin` ложно, и пишет в журнал `tool_calls` статус `blocked`.
- Образец админ-инструмента `src/pipeline/agent-tools/global-knowledge/global-knowledge-add.js`: поля
  `requiresAdmin: true` и `isEnabled: (ctx, config) => <флаг> && ctx.isAdmin`.

Чего не хватает: модуля генерации частей навыка моделью, модуля безопасной записи и перезагрузки навыка, и набора
admin-инструментов, через которые модель управляет этими операциями из диалога.

## 3. Базовый принцип

Инструментарий состоит из трёх слоёв:

1. **Слой генерации (LLM под капотом)** — `src/pipeline/skills/author.js`. Чистые функции, которые по описанию или
   текущему содержимому возвращают строгий JSON: черновик целого навыка, переписанный prompt-блок, определение схемы
   домена, предложение точечной правки. Каждая возвращённая схема проходит мета-валидацию до записи.
2. **Слой записи и перезагрузки** — `src/pipeline/skills/writer.js`. Собирает `SKILL.md` из частей, проверяет инвенты
   (имя, уникальность, существование инструментов, безопасность пути), атомарно пишет файлы в каталог `skills/<name>/`,
   делает резервную копию прежней версии, перезагружает реестр и заводит строку домена в `mem.agent_domains`.
3. **Слой инструментов агента** — каталог `src/pipeline/agent-tools/skill-authoring/`. Набор admin-only инструментов,
   через которые модель в диалоге выполняет операции слоёв 1 и 2. Это и есть «агентский инструментарий»: администратор
   разговаривает с ботом, а бот вызывает эти инструменты.

Сам инструментарий доставляется как **meta-skill** `skill-author` (домен `skill_author`): его блок `# Skill Prompt`
объясняет модели анатомию навыков, назначение каждой части и порядок работы, а его `tools.allowed` перечисляет
инструменты слоя 3. Благодаря этому модель понимает, что такое навыки и какие операции над ними бывают, без специального
кода — тем же механизмом подмешивания, что и у обычных навыков (см. раздел 5). Получается самоприменимая конструкция:
навык, который редактирует навыки.

Жёсткое ограничение области: слой записи пишет и удаляет **только внутри каталога `config.skills.dir`**. Любой путь за
его пределами, абсолютный путь или выход через `..` отклоняется. Навык не может править исходный код, конфигурацию или
чужие каталоги.

## 4. Состав инструментов агента

Все инструменты ниже имеют `requiresAdmin: true` и `isEnabled: (ctx, config) => config.skills.authoring.enabled &&
ctx.isAdmin`. Имена и описания инструментов — на английском (как требует реестр), заголовки `title` — на русском.

### 4.1. Чтение и навигация

- `skill_author_list` — перечислить все навыки: имя, `domain_key`, краткое описание, набор инструментов, наличие схемы и
  справочников. Опирается на `listSkillRoutes` и `getSkill`.
- `skill_author_read` — показать целиком разобранный навык или одну его часть (`frontmatter`, `skill_prompt`,
  `fact_extraction_prompt`, `domain_schema`, список `references`). Нужен модели, чтобы редактировать осмысленно.

### 4.2. Создание

- `skill_author_create` — создать новый навык по описанию на естественном языке. Под капотом вызывает
  `generateSkillDraft` (слой 1): модель предлагает имя, `domain_key`, описание, `when_to_use`, сигналы классификации,
  `# Skill Prompt`, `## Fact Extraction Prompt` и при необходимости определение схемы домена. Возвращает **черновик и
  результат валидации**, но по умолчанию не пишет на диск (см. `apply` ниже).

### 4.3. Точечное редактирование частей

- `skill_author_set_field` — изменить машинное поле фронтматтера: `title`, `description`, `enabled`, `when_to_use`,
  `positive_signals`, `negative_signals`, `tools.allowed`, `tools.base`, `model.main`, `model.extract`,
  `references.allowed`. Значение задаётся прямо или генерируется моделью из инструкции.
- `skill_author_write_prompt` — переписать или улучшить блок `# Skill Prompt` по инструкции («сделай тон строже»,
  «добавь правило про уточняющие вопросы»). Под капотом — `refineBlock`.
- `skill_author_write_extraction` — то же для блока `## Fact Extraction Prompt`.
- `skill_author_schema_generate` — сгенерировать или дополнить определение схемы домена (`domain-schema.json`):
  сущности, закрытые схемы `data`, режимы и словари `entity_key`. Под капотом — `generateDomainSchema`; результат
  обязательно проходит `validateDefinition`.
- `skill_author_schema_edit` — точечно поправить схему: добавить или убрать сущность, поле, значение словаря, синоним,
  поменять режим ключа. Каждое изменение валидируется до применения.
- `skill_author_add_reference` — создать или обновить файл `references/<path>.md` по инструкции (модель пишет
  содержимое), включить `references.allowed`. Путь проверяется на принадлежность каталогу `references/**`.
- `skill_author_remove_reference` — удалить файл справочника (с подтверждением).

### 4.4. Проверка, применение, жизненный цикл

- `skill_author_validate` — проверить навык (черновик или сохранённый) без записи: форма фронтматтера, наличие
  обязательных блоков, валидность схемы по мета-схеме, существование всех `tools.allowed` в реестре инструментов,
  уникальность `name` и `domain_key`. Возвращает список замечаний.
- `skill_author_apply` — записать подготовленный черновик или правку на диск, сделать резервную копию прежней версии,
  перезагрузить реестр и при новом `domain_key` завести строку в `mem.agent_domains`. Разрушающее действие, требует
  `confirm = true`.
- `skill_author_enable` / `skill_author_disable` — переключить поле `enabled` навыка (мягкое отключение без удаления).
- `skill_author_delete` — удалить каталог навыка целиком. Самое разрушающее действие: требует `confirm = true` и
  отдельного подтверждения администратора; `general` удалить нельзя.
- `skill_author_reload` — перечитать реестр с диска (после правок в редакторе вне бота).

Большинство порождающих инструментов принимают флаг `apply` (по умолчанию `false`): без него инструмент возвращает
предложенный результат и замечания валидации для предпросмотра, а запись делает только при `apply = true` (или отдельным
вызовом `skill_author_apply`). Это даёт схему «сначала покажи, потом применяй».

## 5. Как модель понимает, что и когда делать

Модель не знает «из коробки», что такое skill и зачем нужна та или иная операция над ним. Понимание задаётся явно
четырьмя механизмами; вместе они переводят размытую просьбу администратора в выбор конкретного инструмента и конкретной
запчасти навыка.

### 5.1. Описания инструментов как триггеры выбора

Модель выбирает инструмент по полю `description` его function-definition. Поэтому описание пишется не как справка для
человека, а как контракт-триггер: что делает, **когда вызывать**, чем отличается от похожих инструментов и что на входе.
Похожие инструменты явно разводятся в тексте, иначе модель будет путать, например, правку текста и правку схемы:

```text
skill_author_schema_edit:
  "Edit the domain memory schema of an existing skill: add or remove an entity, a data field, a vocabulary
   value, a synonym, or change the entity_key mode. Use when the admin asks to change WHAT structured facts a
   skill remembers or HOW its keys are normalized. Do NOT use for prompt wording — use
   skill_author_write_prompt for that. Returns a preview; writes only when apply=true."
```

Это первичный и самый сильный сигнал маршрутизации между инструментами инструментария.

### 5.2. Системный блок-грунтинг: что такое skill и как его править

Когда инструментарий активен у администратора, в системный промпт добавляется блок `SKILL_AUTHORING_CONTEXT` (аналог
`ACTIVE_SKILL_CONTEXT`, но про сам процесс редактирования). Он сообщает модели три вещи:

- **Анатомию навыка** — из каких частей он состоит: namespace памяти (`domain_key`), признаки классификации
  (`when_to_use`, сигналы), `# Skill Prompt`, `## Fact Extraction Prompt`, закрытая схема `data` со словарями
  `entity_key`, список `tools.allowed`, справочники. Без этого модель не знает, какая запчасть за что отвечает.
- **Назначение каждой части** — зачем её менять: `when_to_use` влияет на маршрутизацию запросов; `data_schema`
  определяет, какие предметные факты запоминаются и как дедуплицируются по `entity_key`; `tools.allowed` ограничивает
  предметные инструменты домена; `# Skill Prompt` задаёт поведение ответа. Это и есть понимание цели операции.
- **Рабочий порядок** — правила процесса: сначала прочитать текущее состояние навыка, затем показать предпросмотр, и
  только после подтверждения применять (`apply = true`); на замечания валидатора — исправляться и повторять; крупные
  правки схемы выполнять осознанно, помня, что ранее сохранённые факты не переписываются.

В этот же блок входит компактная карта «намерение администратора → инструмент» — та же таблица, что в разделе 10, но
сформулированная как инструкция модели.

### 5.3. Инструментарий как meta-skill (`skill-author`)

Самый чистый способ доставить грунтинг из раздела 5.2 — оформить инструментарий **как отдельный навык** `skill-author`
(домен `skill_author`), доступный только администраторам. Тогда его блок `# Skill Prompt` и есть это знание об анатомии,
целях частей и порядке работы, а его `tools.allowed` перечисляет инструменты инструментария. Подмешивается он ровно тем
же механизмом `ACTIVE_SKILL_CONTEXT`, что и любой другой навык: специального кода не требуется. Маршрутизатор выбирает
`skill-author`, когда администратор говорит «заведи навык», «поправь навык такой-то», по его `when_to_use`. Получается
самоприменимая конструкция — «навык, который редактирует навыки», — и она же служит живым примером формата для модели.

Доступ к `skill-author` дополнительно закрыт правами: его инструменты имеют `requiresAdmin: true` и `isEnabled` с
проверкой `ctx.isAdmin`, поэтому не-администратор этот навык фактически не может использовать, даже если запрос в него
попадёт.

### 5.4. Грунтинг текущим состоянием и петля валидации

Модель сопоставляет размытую просьбу с конкретной запчастью не угадыванием, а через данные:

- **Сначала чтение.** Инструмент `skill_author_read` возвращает реальную структуру навыка (поля фронтматтера, тексты
  блоков, сущности схемы, список справочников). Модель правит то, что видит, а не то, что предполагает.
- **Валидация как обратная связь.** Каждый порождающий инструмент возвращает результат `validateSkill` со списком
  замечаний. Модель использует их, чтобы исправиться до применения, а не записать сломанный навык.

Вместе эти четыре механизма и дают модели и выбор нужного инструмента, и понимание цели операции, и знание формата
навыка.

## 6. LLM под капотом (слой генерации)

Модуль `src/pipeline/skills/author.js` повторяет приём строгого структурированного вывода из `src/llm.js` (`chatJSON`):
модель возвращает только JSON по описанной схеме, ответ разбирается и валидируется кодом. Ключевые функции:

- `generateSkillDraft({ description, hints })` → объект навыка целиком (поля фронтматтера + тексты блоков + определение
  схемы). Аналог прежнего генератора схемы, но для всего пакета. Результат прогоняется через `validateDefinition` и
  проверку обязательных полей; замечания возвращаются вместе с черновиком.
- `refineBlock({ kind, current, instruction, skillContext })` → новый текст блока `# Skill Prompt` или
  `## Fact Extraction Prompt`. `skillContext` (имя, домен, описание) даёт модели рамку, чтобы текст оставался в теме.
- `generateDomainSchema({ domainKey, title, description, samples })` → определение схемы домена (сущности, закрытые
  `data_schema`, режимы и словари `entity_key`). Обязательно валидируется мета-схемой; невалидное определение не
  возвращается как готовое, а сопровождается списком замечаний для повторной генерации.
- `proposeSchemaEdit({ definition, instruction })` → изменённое определение плюс краткое человекочитаемое описание
  правки. Применяется только после повторной мета-валидации.

Модель для этих операций берётся из `config.llm.mainModel` (качество важнее скорости), с возможностью переопределить
переменной окружения. Промпты живут в `author.js` и требуют от модели соблюдать формат skills: закрытые схемы
(`additionalProperties: false`, непустой `required`), допустимые режимы `entity_key` (`fixed_vocab` со словарём или
`slug`), английские описания инструментов.

## 7. Слой записи и перезагрузки

Модуль `src/pipeline/skills/writer.js`:

- `composeSkillFile(parts)` — собрать текст `SKILL.md` из фронтматтера и блоков (обратная операция к `splitSkillFile`).
  Фронтматтер сериализуется детерминированно, блоки расставляются в стабильном порядке.
- `validateSkill(parts)` — единая проверка перед записью: форма фронтматтера, обязательные блоки, мета-валидация схемы,
  существование `tools.allowed` в реестре инструментов, kebab-case имени, совпадение `name` с именем каталога,
  уникальность `name` и `domain_key` среди остальных навыков, безопасность всех путей.
- `writeSkill(name, parts, { backup = true })` — атомарная запись: новые файлы пишутся во временные имена и
  переименовываются; прежняя версия сохраняется рядом как резервная копия (`.bak` с отметкой времени) или, если каталог
  под git, фиксация изменений оставляется человеку. После записи вызывается `invalidateSkillsCache()` и
  `loadSkills({ force: true })`, а для нового `domain_key` — заведение строки в `mem.agent_domains` (та же логика, что в
  команде `skills sync`).
- `deleteSkill(name, { confirm })` — удаление каталога навыка с теми же проверками области и резервной копией.

Горячая перезагрузка означает, что новый или изменённый навык начинает действовать без перезапуска процесса: следующий
запрос пользователя уже классифицируется с учётом обновлённого реестра.

## 8. Доступ только для администраторов

Гейтинг строится на уже существующем механизме прав, поэтому отдельной системы ролей не вводится:

- Каждый инструмент инструментария имеет `requiresAdmin: true`. Обёртка `executeTool` (`src/pipeline/tools.js`)
  отклоняет вызов от не-администратора ещё до обработчика и пишет в `tool_calls` статус `blocked`. Это страховка на
  уровне исполнения, а не только на уровне видимости.
- Каждый инструмент имеет `isEnabled: (ctx, config) => config.skills.authoring.enabled && ctx.isAdmin`, поэтому
  не-администратору инструменты вообще не попадают в набор `buildToolDefs(ctx)` и не видны модели.
- Признак администратора берётся из контекста запроса `ctx.isAdmin`, который ядро вычисляет как `user.is_admin === true`
  по таблице `mem.users` (см. `src/agent.js`). Назначение администратора остаётся ручной операцией в базе.
- Все операции создания, правки и удаления навыков журналируются в `tool_calls` (имя инструмента, вход, статус,
  задержка) — это аудиторский след изменений конфигурации поведения бота.

Дополнительный флаг `config.skills.authoring.enabled` (переменная окружения `SKILL_AUTHORING_ENABLED`, по умолчанию
выключено) позволяет держать инструментарий выключенным в обычной эксплуатации и включать только на время
редактирования. При выключенном флаге инструментов нет даже у администратора.

## 9. Поток работы

**Создание нового навыка.**

1. Администратор: «Заведи навык про подбор рецептов: учитывает аллергии и нелюбимые продукты, помнит кухни, которые
   нравятся».
2. Модель вызывает `skill_author_create` с описанием. Инструментарий генерирует черновик (имя `recipe-helper`, домен
   `recipe_helper`, признаки, prompt-блоки, схему с сущностями `dietary_restriction`, `cuisine_preference`) и
   возвращает его вместе с замечаниями валидации, ничего не записывая.
3. Модель показывает администратору сводку. Администратор просит поправку: «убери ограничение по бюджету, добавь любимые
   кухни словарём». Модель вызывает `skill_author_schema_edit`.
4. Администратор подтверждает. Модель вызывает `skill_author_apply` с `confirm = true`. Навык записан, реестр
   перезагружен, строка домена заведена. Следующий запрос про рецепты уже маршрутизируется в новый навык.

**Редактирование части существующего навыка.**

1. Администратор: «В навыке про авиабилеты сделай prompt строже про уточняющие вопросы и добавь синоним „пересадки“ к
   ключу багажа».
2. Модель вызывает `skill_author_write_prompt` (правка `# Skill Prompt`) и `skill_author_schema_edit` (правка словаря
   синонимов). Оба возвращают предпросмотр.
3. После подтверждения — `skill_author_apply`.

## 10. Интерфейс редактирования всех частей навыка

| Часть навыка | Инструмент | LLM-операция под капотом |
|--------------|------------|--------------------------|
| Имя, домен, описание, признаки классификации | `skill_author_set_field` | по инструкции или прямое значение |
| `# Skill Prompt` | `skill_author_write_prompt` | `refineBlock` |
| `## Fact Extraction Prompt` | `skill_author_write_extraction` | `refineBlock` |
| Схема домена (сущности, поля, словари) | `skill_author_schema_generate` / `skill_author_schema_edit` | `generateDomainSchema` / `proposeSchemaEdit` |
| Список инструментов `tools.allowed` | `skill_author_set_field` | проверка существования в реестре инструментов |
| Переопределение моделей `model.*` | `skill_author_set_field` | прямое значение |
| Справочники `references/**` | `skill_author_add_reference` / `skill_author_remove_reference` | генерация содержимого моделью |
| Включение/выключение навыка | `skill_author_enable` / `skill_author_disable` | переключение поля `enabled` |
| Целый навык | `skill_author_create` / `skill_author_delete` | `generateSkillDraft` / удаление каталога |

## 11. Предлагаемые изменения в коде

- `src/config.js` — добавить `skills.authoring`: `enabled` (флаг `SKILL_AUTHORING_ENABLED`, по умолчанию выключено) и при
  необходимости модель генерации.
- `src/pipeline/skills/author.js` — новый модуль: `generateSkillDraft`, `refineBlock`, `generateDomainSchema`,
  `proposeSchemaEdit`. Чистые функции поверх `chatJSON`, каждый результат со схемой проходит `validateDefinition`.
- `src/pipeline/skills/writer.js` — новый модуль: `composeSkillFile`, `validateSkill`, `writeSkill`, `deleteSkill`,
  заведение строки домена в `mem.agent_domains`, вызов `invalidateSkillsCache` и `loadSkills({ force: true })`.
- `src/pipeline/skills/registry.js` — при необходимости вынести разбор одного каталога в экспортируемую функцию, чтобы
  `writer.js` мог пере-разбирать только изменённый навык.
- `src/pipeline/agent-tools/skill-authoring/` — по модулю на инструмент из раздела 4, все с `requiresAdmin: true` и
  `isEnabled` с проверкой `ctx.isAdmin` и флага; регистрация в `src/pipeline/agent-tools/index.js`.
- `skills/skill-author/SKILL.md` — meta-skill `skill-author` (домен `skill_author`): его `# Skill Prompt` содержит
  грунтинг из раздела 5 (анатомия навыка, назначение частей, порядок работы и карта «намерение → инструмент»), а
  `tools.allowed` перечисляет инструменты инструментария. Этот навык доставляет понимание модели тем же механизмом
  `ACTIVE_SKILL_CONTEXT`, что и обычные навыки; отдельного спецкода для системного блока не требуется.
- Тесты `tests/skill-authoring.test.mjs` (без сети): сборка `SKILL.md` из частей и обратный разбор дают эквивалентный
  навык; `validateSkill` ловит дубликаты имени и домена, неизвестные инструменты, небезопасные пути; запись и удаление
  ограничены каталогом `skills/`; гейтинг (`requiresAdmin`, `isEnabled`) для не-администратора прячет инструменты;
  meta-skill `skill-author` проходит обычную валидацию реестра и его `tools.allowed` ссылается на существующие
  инструменты инструментария.

## 12. Безопасность и гарантии

- Область записи ограничена каталогом `config.skills.dir`; абсолютные пути и `..` отклоняются на уровне `writer.js`.
- Запись только после прохождения `validateSkill`: сломанный навык на диск не попадает.
- Все порождающие операции по умолчанию возвращают предпросмотр; запись и удаление требуют `confirm = true`.
- Перед перезаписью делается резервная копия прежней версии; история навыков ведётся в системе контроля версий.
- Атомарная запись через временный файл и переименование исключает частично записанный `SKILL.md`.
- Доступ только администраторам на двух уровнях (видимость через `isEnabled`, исполнение через `executeTool`), плюс
  журналирование каждой операции в `tool_calls`.
- Глобальный флаг `SKILL_AUTHORING_ENABLED` держит инструментарий выключенным вне сессий редактирования.

## 13. Этапы

1. Слой генерации `author.js` и слой записи `writer.js` с тестами без сети (сборка/разбор, валидация, ограничение
   области). Инструменты ещё не подключены.
2. Инструменты чтения и создания (`skill_author_list`, `skill_author_read`, `skill_author_create`,
   `skill_author_validate`, `skill_author_apply`) под флагом и правами администратора, плюс meta-skill `skill-author`
   с грунтинг-промптом (раздел 5), чтобы модель понимала анатомию навыков и выбирала нужные инструменты.
3. Инструменты точечного редактирования частей и работы со справочниками.
4. Инструменты жизненного цикла (`enable`/`disable`/`delete`/`reload`) и аудиторские отчёты по журналу `tool_calls`.

## 14. Критерии готовности

- Администратор в диалоге создаёт новый рабочий навык из описания на естественном языке; после подтверждения навык
  записан, реестр перезагружен, строка домена заведена, и следующий запрос маршрутизируется в новый навык.
- Администратор правит любую отдельную часть навыка (поле, prompt, схему, словарь, справочник) через диалог, с
  предпросмотром до применения.
- Невалидный навык не записывается; замечания валидации понятны и возвращаются модели.
- Из размытой просьбы администратора модель выбирает правильный инструмент и правильную часть навыка, опираясь на
  описания-триггеры, грунтинг meta-skill `skill-author`, прочитанное текущее состояние и замечания валидации.
- Запись и удаление ограничены каталогом `skills/`; выход за его пределы невозможен.
- Не-администратор не видит инструментов и не может их выполнить; попытка вызова отклоняется и журналируется.
- При выключенном `SKILL_AUTHORING_ENABLED` инструментов нет ни у кого.

## 15. Риски

- Качество сгенерированной схемы и prompt-блоков требует ревью администратором; предпросмотр и обязательная мета-
  валидация снижают риск, но не отменяют человеческий контроль.
- Горячая правка активного навыка меняет поведение бота для всех пользователей этого домена немедленно; смягчается
  резервными копиями, журналом и возможностью быстро откатить через `enable`/`disable` или восстановление из git.
- Изменение `data_schema` не переписывает ранее сохранённые факты: они остаются в прежней форме, поэтому крупные правки
  схемы стоит сопровождать осознанным решением о судьбе старых записей.

## 16. План реализации для Claude Code

Этот раздел — исполнимый план. Он самодостаточен: по нему реализацию можно выполнить с нуля, а при сбое — продолжить с

### 16.0. Соглашения исполнения (прочитать перед началом)

- **Ветка.** Создать `feature/skill-authoring` от `master`. Не работать на `master` напрямую.
- **Порядок.** Выполнять этапы строго по очереди (16.1 → 16.6). Внутри этапа — пункты чек-листа сверху вниз.

  файле (раздел 17) и сделать коммит-чекпойнт `checkpoint(skill-authoring): <ID пункта>`. При возобновлении после сбоя:
  прочитать раздел 17, найти первый `[ ]` и продолжить с него. Незавершённый пункт оставлять `[ ]`.
- **Тесты без сети** (БД и LLM не нужны): `node tests/skill-authoring.test.mjs`, `npm run test:skills`.
- **Тесты со схемой** (нужна БД; LLM можно пропустить): `SCHEMA_SKIP_LLM=1 npm run test:schema`.
- **Полный прогон** (нужны БД и LLM-прокси): `npm test`. Запускать в конце этапов, меняющих контур ответа.
- **Ручная проверка инструментов** требует `SKILL_AUTHORING_ENABLED=on` в `.env` и пользователя с `is_admin = true`.
- **Язык.** Описания инструментов (`description`) и значения `enum` — на английском (требование реестра инструментов и
  тест на английские описания в `tests/run.js`). Заголовки `title` и любой текст, адресованный пользователю, — на
  русском развёрнутыми предложениями.
- **Запрет области.** Любой код записи трогает только каталог `config.skills.dir`. Никаких записей в `src/`, `migrations/`
  или вне `skills/`.

### 16.1. Этап A — конфигурация и слой генерации (`author.js`)

Файлы: `src/config.js`, `src/pipeline/skills/author.js`, тесты в `tests/skill-authoring.test.mjs` (заготовка).

1. В `src/config.js` в объект `skills` добавить подраздел:
   ```js
   authoring: {
     enabled: flag(env.SKILL_AUTHORING_ENABLED, false),
     // Модель генерации частей навыка; пусто → берётся config.llm.mainModel в author.js.
     model: env.SKILL_AUTHORING_MODEL || null,
   },
   ```
2. Создать `src/pipeline/skills/author.js` — чистые функции поверх `chatJSON` из `src/llm.js`. Модель:
   `config.skills.authoring.model || config.llm.mainModel`. Контракты (вход → выход):
   - `generateSkillDraft({ description, hints })` →
     `{ name, domain_key, title, description, when_to_use, positive_signals, negative_signals, skill_prompt,
     fact_extraction_prompt, definition, issues }`. `definition` — объект схемы домена или `null` (если предметных
     сущностей нет). `issues` — замечания `validateDefinition` по `definition` (пустой массив, если всё чисто).
   - `refineBlock({ kind, current, instruction, skillContext })` → `{ text }`. `kind` ∈ `'skill_prompt' |
     'fact_extraction_prompt'`. `skillContext` = `{ name, domain_key, title, description }`.
   - `generateDomainSchema({ domain_key, title, description, samples })` → `{ definition, issues }`. `definition`
     обязан проходить `validateDefinition`; если нет — вернуть его вместе с `issues`, не бросая исключение.
   - `proposeSchemaEdit({ definition, instruction })` → `{ definition, summary, issues }`. `summary` — короткое
     человекочитаемое описание правки на русском.
   - Каждый промпт требует от модели соблюдать формат: закрытые `data_schema` (`additionalProperties:false`, непустой
     `required`), режимы `entity_key` только `fixed_vocab` (с непустым `vocabulary`) или `slug`, английские описания.
3. Проверка этапа: модульные функции импортируются без сети; ошибки модели не роняют процесс (как в существующих
   генераторах — при сбое возвращать пустой/частичный результат с `issues`).

### 16.2. Этап B — слой записи и перезагрузки (`writer.js`)

Файлы: `src/pipeline/skills/writer.js`, при необходимости экспорт из `src/pipeline/skills/registry.js`, тесты.

1. В `registry.js` сделать переиспользуемым разбор одного каталога: экспортировать функцию вида
   `parseSkillDir(dir, name)` (вынести из приватного загрузчика) и `getAllSkills()` → массив полных объектов навыков.
   Это нужно `writer.js` для проверки уникальности и пере-разбора одного навыка.
2. Создать `src/pipeline/skills/writer.js`. `parts` — объект:
   `{ name, frontmatter, skill_prompt, fact_extraction_prompt, definition }`, где `frontmatter` содержит `domain_key`,
   `title`, `description`, `enabled`, `classification{ when_to_use, positive_signals, negative_signals }`,
   `memory{ scopes, schema }`, `tools{ allowed, base }`, `model{ main, extract }`, `references{ allowed }`.
   Контракты:
   - `composeSkillFile(parts)` → строка `SKILL.md`. Фронтматтер сериализуется в стабильном порядке полей; затем блок
     `# Skill Prompt`, затем `## Fact Extraction Prompt`. Схема в файл `SKILL.md` не встраивается (хранится в
     `domain-schema.json`), `memory.schema` ставится в `domain-schema.json`.
   - `validateSkill(parts)` → `{ ok, issues }`. Правила (все обязательны):
     - `name` соответствует `^[a-z0-9]+(-[a-z0-9]+)*$` и совпадает с именем каталога;
     - `domain_key` непустой, нижний регистр, `^[a-z0-9_]+$`;
     - задан `classification.when_to_use` и непустой `skill_prompt`;
     - если есть `definition` с непустыми `entities` — он проходит `validateDefinition`, и `definition.domain_key`
       совпадает с `frontmatter.domain_key`;
     - каждый инструмент из `tools.allowed` существует в реестре инструментов (`getTool(name)` не `null`);
     - `name` и `domain_key` уникальны среди прочих навыков (при редактировании текущий навык исключается из проверки);
     - все пути справочников лежат внутри `references/**`, без абсолютных путей и `..`.
   - `writeSkill(name, parts, { backup = true })` → `{ path, reloaded: true }`. Порядок: `validateSkill` (бросить с
     понятным текстом при `!ok`) → резервная копия прежнего каталога (если был) → атомарная запись `SKILL.md` и
     `domain-schema.json` (через временный файл и переименование) → `invalidateSkillsCache()` → `loadSkills({ force:
     true })` → `ensureDomainRow(domain_key, title, description)`.
   - `writeReference(name, relPath, content)` → `{ path }`; путь проверяется на принадлежность `references/**`.
   - `removeReference(name, relPath, { confirm })` и `deleteSkill(name, { confirm })` → выполняются только при
     `confirm === true`; `deleteSkill('general')` и `deleteSkill('skill-author')` запрещены.
   - `ensureDomainRow(domainKey, title, description)` → `INSERT ... ON CONFLICT (domain_key) DO NOTHING` в
     `mem.agent_domains` (та же логика, что в `skills:sync`).
3. Проверка: `tests/skill-authoring.test.mjs` (без сети) покрывает compose↔parse round-trip, все правила `validateSkill`,
   ограничение области записи/удаления и запрет удаления `general`/`skill-author`.

### 16.3. Этап C — инструменты чтения и создания + meta-skill `skill-author`

Файлы: `src/pipeline/agent-tools/skill-authoring/*`, `src/pipeline/agent-tools/index.js`, `skills/skill-author/`.

1. Создать инструменты (каждый: `requiresAdmin: true`, `isEnabled: (ctx, config) => config.skills.authoring.enabled &&
   ctx.isAdmin`, английское `description`-триггер, русский `title`):
   - `skill_author_list` → перечень навыков (имя, домен, описание, инструменты, наличие схемы/справочников).
   - `skill_author_read` (`{ name, part? }`) → разобранный навык целиком или одна часть.
   - `skill_author_create` (`{ description, apply? }`) → черновик (через `generateSkillDraft`) + замечания; пишет на
     диск только при `apply === true` (через `writeSkill`).
   - `skill_author_validate` (`{ name }` или `{ parts }`) → результат `validateSkill`.
   - `skill_author_apply` (`{ name, parts, confirm }`) → запись подготовленного навыка; требует `confirm === true`.
2. Зарегистрировать инструменты в `src/pipeline/agent-tools/index.js`.
3. Создать meta-skill `skills/skill-author/SKILL.md` (домен `skill_author`, без предметной схемы) с грунтингом из
   раздела 5. Каркас:
   ```markdown
   ---
   name: skill-author
   domain_key: skill_author
   title: Редактор навыков
   description: Создание и редактирование навыков (skills) бота администратором через диалог.
   enabled: true
   classification:
     when_to_use: >
       Администратор просит создать новый навык, изменить существующий навык или любую его часть —
       признаки классификации, prompt ответа, prompt извлечения, схему памяти, инструменты или справочники.
     positive_signals: [создай навык, заведи навык, поправь навык, измени промпт навыка, добавь сущность в схему]
     negative_signals: [обычный пользовательский запрос по предметной области]
   memory:
     scopes: [dialog]
   tools:
     allowed: [skill_author_list, skill_author_read, skill_author_create, skill_author_validate,
       skill_author_apply, skill_author_set_field, skill_author_write_prompt, skill_author_write_extraction,
       skill_author_schema_generate, skill_author_schema_edit, skill_author_add_reference,
       skill_author_remove_reference, skill_author_enable, skill_author_disable, skill_author_delete,
       skill_author_reload]
     base: true
   references:
     allowed: false
   ---

   # Skill Prompt

   <Грунтинг из раздела 5.2: анатомия навыка; назначение каждой части; порядок read → preview → confirm → apply;
   карта «намерение администратора → инструмент» из раздела 10; правило не записывать невалидный навык и исправляться
   по замечаниям валидатора.>

   ## Fact Extraction Prompt

   Не извлекай предметных фактов из служебного диалога редактирования навыков.
   ```
4. Завести домен: `npm run skills:sync` (или довериться `ensureDomainRow` при первой записи). Проверка:
   `npm run skills:validate` принимает `skill-author`; `npm run test:skills` зелёный (meta-skill не ломает реестр).
5. Замечание о видимости: инструменты инструментария зарегистрированы локально и потому считаются базовыми; их защищают
   `requiresAdmin` и `isEnabled` (админ + флаг). Грунтинг-промпт активируется, когда маршрутизатор выбирает
   `skill-author` по `when_to_use`.

### 16.4. Этап D — инструменты точечного редактирования

Файлы: `src/pipeline/agent-tools/skill-authoring/*`, регистрация в `index.js`.
