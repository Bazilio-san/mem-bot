# План-промпт: стриминг ответа LLM и статусов инструментов

## 1. Роль и цель

Ты Codex в репозитории `D:\DEV\SAND\inter-2-prep`. Твоя задача — реализовать потоковую обратную связь для
ИИ-бота с памятью: стриминг финального текста LLM, события этапов агентского цикла (включая показ пользователю
«Вызываю инструмент <человеческое имя инструмента>» при каждом вызове инструмента) и поверх этого устойчивый
Telegram UX как одну из возможных реализаций потребителя событий.

Главный архитектурный принцип этой задачи: ИИ-бот не должен зависеть от наличия или функциональности Telegram-бота.
Ядро агента (`src/agent.js`, `src/llm.js`, `src/pipeline/**`) обязано полностью работать без Telegram: оно умеет
стримить текст и испускать абстрактные события через `onEvent`, ничего не зная о мессенджере. Telegram-адаптер
(`src/telegram.js`) — лишь один потребитель этих событий; его отсутствие или поломка не должны влиять на работу
ядра. Любой другой канал (командная строка, тесты, будущий веб-интерфейс) подписывается на тот же контракт событий.

Сначала прочитай актуальные файлы `docs/ai-bot-with-memory/**`, `docs/telegram/**`, `src/agent.js`,
`src/llm.js`, `src/pipeline/tools.js`, `src/pipeline/agent-tools/**`, `src/telegram.js`, `tests/run.js`,
`tests/reactions.test.mjs` и релевантные тесты Telegram/voice. После чтения действуй по этому плану, сохраняя
текущие границы архитектуры: ядро агента не знает о Telegram, Telegram-адаптер только отображает абстрактные
события ядра в возможности конкретного канала.

## 2. Текущее состояние, от которого надо отталкиваться

`src/agent.js` сейчас экспортирует `handleMessage({ externalId, userMessage, domainKey, extractSync })`, выполняет
классификацию, выборку памяти, сборку сообщений, цикл инструментов до пяти шагов, сохраняет финальный `answer` и
возвращает диагностический объект. Внутри цикла он вызывает `chat(...)` из `src/llm.js`; этот вызов не потоковый и
возвращает готовое сообщение модели с `content` и `tool_calls`.

`src/llm.js` работает через OpenAI-совместимый Chat Completions API прокси. Это правильный слой для добавления
потокового варианта `chatStream`, потому что именно здесь уже изолированы модель, baseURL, ключ и отладка. Важно не
ломать `chatJSON` и `embed`: структурированные вспомогательные этапы должны остаться непотоковыми.

`src/pipeline/tools.js` — это реестр-агрегатор: он собирает инструменты из `src/pipeline/agent-tools/**` (через
`agent-tools/index.js`, экспортирующий массив `allTools`). Каждый инструмент лежит в своём модуле и уже экспортирует
объект с полями `name`, `title` (человеческое имя на русском), `definition` (OpenAI function definition), при
необходимости `requiresAdmin` и `isEnabled(ctx, config)`, а также `handler(ctx, args)`. Поэтому `src/pipeline/tools.js`
уже предоставляет всё нужное для UI-событий: `toolDefs`, производный `toolMeta` вида `{ [name]: { title } }`, функцию
`toolTitle(name)` (возвращает `toolMeta[name]?.title || name`), `buildToolDefs(ctx)` (фильтрует по `isEnabled`),
`getTool(name)` и `executeTool(ctx, name, args)`. Отдельный реестр названий заводить не нужно — `title` уже является
единственным источником правды; для UI-событий, журналов и тестов используй существующую `toolTitle(name)`.

`src/telegram.js` уже умеет отправлять `sendChatAction` с `typing`, держит `startTypingLoop(chatId)` во время долгого
распознавания речи, отправляет финальный результат через `deliverAgentResult`, разбивает длинные сообщения и сохраняет
внешние Telegram `message_id` через `saveSentRefs`. Это хорошая база, но для стриминга текста нельзя слать каждый токен
отдельным сообщением: нужно редактировать одно черновое сообщение с троттлингом, а финал сохранить как одну реплику
ассистента.

Официальная Telegram Bot API документация описывает `sendChatAction` как способ показать, что на стороне бота что-то
происходит; статус держится около пяти секунд или меньше и сбрасывается при сообщении от бота. Поэтому `typing` нужно
переотправлять таймером до появления первого видимого текста или до завершения обработки. OpenAI Chat Completions
стримит `ChatCompletionChunk`, где текст приходит в `choices[].delta.content`, а вызовы инструментов — в
`choices[].delta.tool_calls` с частями имени и аргументов. Аргументы инструментов всё равно нужно валидировать и
разбирать только после полного завершения соответствующего assistant-сообщения.

## 3. Дизайн: общий событийный контракт ядра

Добавь в `handleMessage` необязательный параметр `onEvent`, например:

```js
export async function handleMessage({
  externalId,
  userMessage,
  domainKey = 'general',
  extractSync = false,
  onEvent = null,
  stream = false,
}) { ... }
```

`onEvent` — асинхронный или синхронный callback и единственная точка связи ядра с любым каналом вывода. Ядро вызывает
его best-effort: ошибка callback не должна ломать агентский ответ, а отсутствие callback (`onEvent = null`) — это
нормальный режим работы без какого-либо адаптера. Сделай маленький helper `emit(event)` в `src/agent.js`, который
добавляет `conversationId`, `userId`, `domainKey` после их появления и ловит ошибки UI-слоя. События должны быть
абстрактными, без Telegram-полей и без любых упоминаний конкретного мессенджера — так контракт остаётся пригодным для
командной строки, тестов и других каналов:

```js
{ type: 'agent.started' }
{ type: 'stage.started', stage: 'classify', title: 'Определяю намерение' }
{ type: 'stage.started', stage: 'memory', title: 'Ищу релевантную память' }
{ type: 'stage.started', stage: 'llm', title: 'Готовлю ответ' }
{ type: 'assistant.delta', text: 'часть ответа' }
{ type: 'assistant.completed', text: 'полный финальный ответ' }
{ type: 'tool.started', toolName: 'memory_search', toolTitle: 'Поиск в памяти' }
{ type: 'tool.completed', toolName: 'memory_search', toolTitle: 'Поиск в памяти', ok: true }
{ type: 'tool.completed', toolName: 'memory_search', toolTitle: 'Поиск в памяти', ok: false, error: '...' }
{ type: 'agent.completed' }
{ type: 'agent.failed', error: '...' }
```

Минимальный обязательный набор для реализации: `assistant.delta`, `assistant.completed`, `tool.started`,
`tool.completed`, `agent.completed`, `agent.failed`. События `stage.started` полезны для красивого UX, но не должны
заставлять Telegram писать много служебных сообщений. В Telegram их лучше сводить к `typing`, а не к видимому тексту,
кроме инструментов.

Событие `tool.started` испускается после того, как модель полностью решила вызвать инструмент и имя инструмента
известно, но до `executeTool`. Не показывай пользователю аргументы инструмента: в них могут быть приватные данные или
внутренние
идентификаторы. Человеческое имя инструмента должно быть безопасной короткой строкой, заданной разработчиком.

## 4. Человеческие имена инструментов: уже реализовано

Реестр человеческих имён уже существует и заводить его заново не нужно. Каждый модуль инструмента в
`src/pipeline/agent-tools/**` экспортирует поле `title` рядом с `name`, а `src/pipeline/tools.js` автоматически собирает
из них `toolMeta` и предоставляет `toolTitle(name)`. На момент написания плана зарегистрированы такие инструменты с
именами и заголовками:

```text
memory_search             → Поиск в памяти
memory_list               → Список личной памяти
memory_forget_entity      → Удаление факта из памяти
memory_forget_all         → Полное удаление личной памяти
scheduler_create_task     → Создание напоминания
secure_record_get         → Доступ к защищённой записи
search_flights            → Поиск авиабилетов
set_reply_mode            → Форма ответа: голос или текст
global_fact_add           → Добавление глобального факта
global_fact_delete        → Удаление глобального факта
global_fact_list          → Список глобальных фактов
global_knowledge_search   → Поиск в базе знаний
global_knowledge_add      → Добавление в базу знаний
global_knowledge_delete   → Удаление из базы знаний
```

Все события UI используют только `toolTitle(name)`. Не выводи `description` из function definition пользователю: оно
написано для модели, может быть длинным и может содержать служебные оговорки вроде «только администратор». Заголовки
живут вместе с самими инструментами в ядре и не зависят от Telegram — любой канал берёт человеческое имя из той же
`toolTitle(name)`.

Что остаётся сделать в рамках этой задачи: при добавлении новых инструментов следить, чтобы у каждого было непустое
поле `title`, и закрепить это тестом. Добавь тест, который собирает все доступные definition names через
`buildToolDefs(ctx)` при разных комбинациях флагов и прав (`ctx.isAdmin`, `config.globalMemory.*`,
`config.voiceOutput.enabled`) и падает, если для какого-то имени `toolTitle(name)` возвращает само имя (то есть `title`
отсутствует или пуст).

## 5. Дизайн: потоковый LLM-клиент

В `src/llm.js` оставь существующий `chat` как совместимый путь и добавь новый `chatStream`. Контракт должен возвращать
такой же финальный объект message, как `chat`, чтобы `src/agent.js` мог заменить только способ получения сообщения:

```js
export async function chatStream({
  model = config.llm.mainModel,
  messages,
  tools,
  toolChoice,
  onDelta,
}) {
  const body = { model, messages, stream: true };
  if (tools && tools.length) body.tools = tools;
  if (toolChoice) body.tool_choice = toolChoice;

  const stream = await client.chat.completions.create(body);
  const acc = { role: 'assistant', content: '', tool_calls: [] };

  for await (const chunk of stream) {
    const choice = chunk.choices?.[0];
    const delta = choice?.delta || {};
    if (delta.content) {
      acc.content += delta.content;
      if (onDelta) await onDelta(delta.content);
    }
    // Здесь аккумулируются delta.tool_calls по index: id, type, function.name, function.arguments.
  }

  if (!acc.tool_calls.length) delete acc.tool_calls;
  return acc;
}
```

Реализация аккумуляции `tool_calls` должна учитывать, что в стриме части одного вызова приходят по `index`: сначала
может прийти `id`, потом `function.name`, потом много фрагментов `function.arguments`. В финальном объекте нужно
получить тот же вид, который сейчас возвращает непотоковый `chat`:
`[{ id, type: 'function', function: { name, arguments } }]`.

Не стримь промежуточный текст пользователю в итерации, которая закончилась `tool_calls`, если модель вдруг прислала
частичный `content` перед вызовом инструмента. Для простоты и чистоты UX в `src/agent.js` передавай `onDelta` в
`chatStream` только на той итерации, где модель не вызывает инструмент, либо буферизуй дельты локально и публикуй их
только если финальный `finish_reason` не `tool_calls`. Это предотвращает ситуацию, когда пользователь видит половину
ответа, затем «Вызываю инструмент...», а потом ответ переписывается.

Сначала сделай совместимый фолбэк: если `stream = false` или потоковый вызов падает на уровне транспорта до первой
дельты, агент может вызвать старый `chat`. Если ошибка возникла после начала видимого стрима, лучше завершить обработку
ошибкой и отправить пользователю обычный текст сбоя, чем пытаться продолжить вторым ответом.

## 6. Дизайн: цикл инструментов со стримингом этапов

В `src/agent.js` в цикле до пяти шагов замени точку вызова модели на функцию `runModelTurn({ allowTextStream })`.
Логика:

1. На каждом шаге испускается `stage.started` с `stage: 'llm'`.
2. Модель вызывается через `chatStream`, если `stream === true`, иначе через старый `chat`.
3. Если модель вернула `tool_calls`, `messages.push(msg)` остаётся как сейчас.
4. Перед каждым `executeTool` испускается `tool.started` с `toolName` и `toolTitle`.
5. После `executeTool` испускается `tool.completed` с `ok: !result?.error`; технический результат не отправляется в UI.
6. В `toolsUsed` сохраняется текущая структура `{ name, args, result }`, чтобы существующие тесты не сломались.
7. Когда модель вернула финальный текст, он становится `answer`, испускается `assistant.completed`, затем сохраняется
   в БД.

Если один assistant turn содержит несколько tool calls, показывай их последовательно в порядке массива. Не отправляй
параллельные видимые статусы: Telegram-клиент всё равно не умеет показать несколько активных операций в одном чате
аккуратно, а последовательность соответствует текущему выполнению `executeTool`.

Если достигнут лимит пяти шагов и финальный ответ так и не получен, текущий код возвращает пустую строку. Улучши это:
сформируй безопасный fallback `Не получилось завершить цепочку инструментов. Попробуйте уточнить запрос.` и испусти
`agent.failed` или отдельный `agent.completed` с `degraded: true`. Это поведение нужно покрыть тестом, если его легко
смоделировать без реального LLM; иначе зафиксируй как ручной сценарий проверки.

## 7. Telegram UX: как красиво обыграть стриминг

Базовая UX-модель:

1. Сразу после принятия сообщения запускается `typing` loop через `sendChatAction`.
2. Пока нет видимого текста, Telegram показывает «печатает...».
3. Когда приходит первый `assistant.delta`, адаптер создаёт одно черновое сообщение, например с первой накопленной
   порцией текста, и останавливает `typing` loop, потому что видимое сообщение уже показывает прогресс.
4. Дальше адаптер редактирует это же сообщение методом `editMessageText` не чаще одного раза в 700-1200 мс и только если
   накопленный текст заметно изменился. На финале выполняется обязательный последний edit с полным ответом.
5. Если ответ длиннее Telegram-лимита, до лимита редактируется первое сообщение, а остаток отправляется обычными
   `sendMessage` после завершения. В потоковом режиме не нужно пытаться редактировать много частей одновременно.

Служебные статусы инструментов:

1. При `tool.started` пользователь должен увидеть текст `Вызываю инструмент: <toolTitle>`.
2. Лучший вариант: если черновой ответ ещё не создан, отправить отдельное короткое status-сообщение и редактировать его
   на следующий статус инструмента. Если черновой ответ уже создан, не смешивать статус инструмента с текстом ответа.
3. После `tool.completed` можно редактировать то же status-сообщение в `Инструмент выполнен: <toolTitle>` и затем
   удалить его через 1-2 секунды, либо оставить до первого фрагмента финального ответа и удалить тогда. В группах лучше
   не удалять слишком агрессивно; в личном чате это выглядит чище.
4. Если инструмент завершился ошибкой, статус можно заменить на `Инструмент не выполнился: <toolTitle>`, но финальный
   ответ модели всё равно должен объяснить пользователю результат естественным языком. Не показывай stack trace.

Рекомендуемый прагматичный вариант для первой реализации: держать одно `statusMessage` на чат для текущей обработки.
`tool.started` создаёт или редактирует его, `tool.completed` редактирует его, первый `assistant.delta` удаляет его, а
финал также удаляет его, если текстовых дельт не было. Это даёт видимость работы инструментов без засорения чата.

Продолжай использовать `typing` как страховочный индикатор на этапах классификации, памяти, ожидания первого LLM chunk и
между инструментом и следующим LLM-вызовом. Не показывай видимые `stage.started` для классификации и памяти: они слишком
быстрые и будут выглядеть как шум. Видимым должен быть только статус инструмента, потому что пользователь явно попросил
наблюдаемость инструментов.

## 8. Telegram-адаптер: рекомендуемая структура кода

Весь код этого раздела живёт только в Telegram-слое и является одной из реализаций потребителя событий ядра. Ядро его
не импортирует и о нём не знает; при запуске без Telegram этот файл просто не подключается, а `handleMessage`
вызывается с `onEvent = null`.

В `src/telegram.js` добавь helper `createTelegramProgress(chatId)`, который возвращает объект:

```js
{
  onEvent(event) { ... },
  finish() { ... },
  fail(error) { ... },
  getSentMessages() { ... },
}
```

Внутри helper держит состояние:

```js
{
  typingStop: null,
  draftMessage: null,
  statusMessage: null,
  buffer: '',
  lastEditAt: 0,
  editTimer: null,
  closed: false,
}
```

Основные методы:

- `startTyping()` и `stopTyping()` переиспользуют существующую идею `startTypingLoop(chatId)`.
- `appendDelta(text)` накапливает `buffer`, создаёт `draftMessage` при первой полезной порции и планирует throttled
  edit.
- `flushDraft({ force })` делает `editMessageText` для первого сообщения, если текст укладывается в лимит.
- `showToolStatus(title, state)` создаёт или редактирует `statusMessage`.
- `clearToolStatus()` удаляет или финально редактирует status-сообщение; выбери один стиль и зафиксируй в документации.
- `complete(finalText)` гарантирует, что пользователь получил полный финальный текст, включая длинный ответ.

Добавь Telegram helper methods `editMessageText` и, если выбран сценарий удаления статусов, `deleteMessage`. Ошибки edit
часто бывают нормальными: например, текст не изменился или сообщение уже недоступно для редактирования. Такие ошибки не
должны ронять агентский ответ; при невозможности редактировать нужно отправить финальный `sendMessage`.

Интеграция в `handleUpdate`:

```js
const progress = createTelegramProgress(chatId);
try {
  progress.startTyping();
  const res = await handleMessage({
    externalId,
    userMessage: text,
    domainKey,
    stream: true,
    onEvent: progress.onEvent,
  });
  chatDomains.set(chatId, res.domainKey);
  const sent = await progress.complete(res.answer || '(пустой ответ)');
  await saveSentRefs(chatId, sent, res.assistantMessageId, 'text');
  await refreshChatMenu(chatId, externalId);
} catch (err) {
  await progress.fail(err);
  ...
} finally {
  await progress.finish();
  releaseSlot();
}
```

Не запускай потоковый режим для реакции `delivery.kind = 'reaction'`: там нет полного ответа LLM, и текущий путь через
`recordReactionTurn` плюс `deliverAgentResult` остаётся правильным. Для голосового входа после распознавания можно
использовать тот же потоковый прогресс: пока идёт STT, показывается `typing`, после распознавания и входа в агентский
контур начинается обычный стриминг.

## 9. Ограничения Telegram, которые надо учесть

`sendChatAction` не заменяет видимого прогресса: он короткоживущий и исчезает при сообщении от бота. Поэтому он нужен до
первого видимого текста и между долгими этапами, а не как единственный механизм стриминга.

`editMessageText` имеет лимиты и rate limits. Не редактируй сообщение на каждый токен. Делай троттлинг по времени и
изменению длины, например не чаще одного раза в 900 мс и не менее чем на 20 новых символов, кроме финального flush.

Telegram имеет лимит около 4096 символов на сообщение, в проекте уже используется запас `TG_MAX_LEN = 4000`. В потоковом
режиме редактируй только первый chunk до `TG_MAX_LEN`; если ответ растёт дальше, покажи в первом сообщении начало, а
после финала отправь оставшиеся части через существующий `sendMessage` и сохрани все external refs.

Не используй HTML/Markdown parse mode в первой реализации стриминга, если текущий бот отправляет plain text. Форматтеры
усложняют инкрементальное редактирование: незакрытый markdown во время стрима часто ломает отображение.

## 10. Конфигурация и флаги

Добавь флаги в `src/config.js`:

```js
streaming: {
  enabled: flag(env.LLM_STREAMING_ENABLED, true),
  telegramEnabled: flag(env.TELEGRAM_STREAMING_ENABLED, true),
  editIntervalMs: Number(env.TELEGRAM_STREAM_EDIT_INTERVAL_MS || 900),
  minEditChars: Number(env.TELEGRAM_STREAM_MIN_EDIT_CHARS || 20),
  toolStatuses: flag(env.TELEGRAM_TOOL_STATUS_ENABLED, true),
}
```

Если проект предпочитает меньше флагов, допустимо оставить только `LLM_STREAMING_ENABLED` и
`TELEGRAM_STREAMING_ENABLED`, а интервалы задать константами в `src/telegram.js`. Но для тестирования rate-limit
поведения параметры удобнее вынести в
конфиг. Значения по умолчанию должны давать новый UX без обязательной настройки `.env`.

При выключенном `LLM_STREAMING_ENABLED` `handleMessage` должен работать по старому пути. При выключенном
`TELEGRAM_STREAMING_ENABLED` Telegram должен продолжать слать `typing` и финальный `sendMessage`, но события
инструментов можно всё равно логировать в debug или игнорировать. При выключенном `TELEGRAM_TOOL_STATUS_ENABLED`
текстовые статусы
инструментов не отправляются, однако `typing` сохраняется.

## 11. Тесты

Добавь unit-level тест для аккумулятора streaming chunks в `src/llm.js` или вынеси чистую функцию
`accumulateChatDelta` в экспортируемый для тестов helper. Тест должен покрывать:

- накопление `delta.content` в финальный `content`;
- накопление одного `tool_call` из нескольких фрагментов `function.arguments`;
- накопление двух `tool_calls` по разным `index`;
- отсутствие `tool_calls` в финальном объекте, когда их не было.

Добавь тест событий `handleMessage` на реальном или минимально подменяемом контуре так, чтобы он проверял порядок:
`tool.started` перед `executeTool`, `tool.completed` после него, финальный `assistant.completed` после последнего
инструмента. Если текущая структура не позволяет легко мокать `chat`, не делай тяжёлый DI-рефактор ради одного теста:
добавь более простой тест на `toolMeta` и ручной сценарий в `tests/check-llm.js` или отдельный smoke script.

Добавь тест на покрытие заголовков: каждое имя, реально попадающее в `buildToolDefs(ctx)` при разных комбинациях флагов
и `ctx.isAdmin`, имеет непустой `title` (то есть `toolTitle(name) !== name`). Перебирать инструменты можно прямо через
экспортируемый массив `allTools` или через `buildToolDefs(ctx)`; отдельный helper-список заводить не обязательно.

Добавь тест Telegram progress helper без реального Telegram API: передай fake `tg(method, body)`, собери вызовы и
проверь, что серия `assistant.delta` создаёт одно `sendMessage` и несколько throttled `editMessageText`, а
`tool.started` создаёт или редактирует статус `Вызываю инструмент: ...`. Если helper останется внутри
`src/telegram.js`, вынеси чистую фабрику в `src/telegram/progress.js`, чтобы тестировать без запуска polling.

После реализации запусти:

```bash
npm test
npm run test:reactions
npm run test:voice
npm run check:llm
```

Если `npm run check:llm` зависит от внешнего прокси и окружения, зафиксируй фактический результат. Нельзя объявлять
стриминг готовым, пока не проверено, что выбранный прокси действительно отдаёт Chat Completions streaming и streaming
tool call deltas в формате, который аккумулирует `chatStream`.

## 12. Наблюдаемость и ошибки

Расширь debug-категорию `llm`: при `DEBUG=llm` логируй старт потокового вызова, количество chunks, итоговый finish
reason и факт tool calls, но не печатай полные аргументы инструментов и приватные данные. Для Telegram добавь категорию
`telegram` или используй существующий `console.error` только для ошибок доставки.

Ошибки `onEvent` и Telegram-редактирования не должны ломать бизнес-ответ. Ошибки LLM, tools и БД — должны сохранять
текущее поведение: пользователь получает понятный общий сбой, а ошибка пишется в stderr. Если инструмент вернул
`{ error }`, это не transport failure: событие `tool.completed` получает `ok: false`, а модель видит результат
инструмента и формирует финальный ответ.

## 13. Что не надо делать в первой итерации

Не переходи с Chat Completions на Responses API: текущий проект явно использует OpenAI-совместимый прокси, и Chat
Completions уже встроен в `src/llm.js`. Миграция API увеличит риск и не нужна для стриминга.

Не стримь JSON-этапы (`chatJSON`) и эмбеддинги. Классификация, извлечение фактов, реакции и история остаются готовыми
структурированными ответами.

Не показывай пользователю внутренние stage-события классификации, память, RAG, историю и аргументы инструментов. Это
будет шумом и может раскрыть внутренние данные. Видимые статусы нужны только для долгого ожидания (`typing`) и вызовов
инструментов.

Не создавай отдельное Telegram-сообщение на каждую дельту текста. Это быстро упрётся в rate limits и испортит историю
чата. Один редактируемый черновик плюс финальное разбиение длинного ответа — достаточная модель.

## 14. Критерии готовности

Реализация считается готовой, когда выполняются все пункты:

- ядро (`src/agent.js`, `src/llm.js`, `src/pipeline/**`) не импортирует `src/telegram.js` и работает с `onEvent = null`,
  то есть полностью функционально без Telegram;
- `handleMessage` совместим со старым вызовом без `stream` и `onEvent`;
- при `stream: true` финальный `answer`, сохранение истории и `toolsUsed` совпадают по смыслу со старым режимом;
- `chatStream` корректно собирает `content` и `tool_calls` из streaming chunks;
- Telegram показывает `typing` до первого видимого текста или долгого инструмента;
- Telegram редактирует один черновик ответа с троттлингом, а не шлёт поток сообщений;
- при вызове инструмента пользователь видит `Вызываю инструмент: <человеческое имя инструмента>`;
- статусы инструментов не содержат аргументов, результатов, stack trace и внутренних идентификаторов;
- длинные ответы доставляются полностью и сохраняют связи `mem.message_external_refs`;
- реакции, голосовой вход, проактивная outbox-доставка и команды продолжают работать;
- тесты и `check:llm` пройдены или честно зафиксирована внешняя причина невозможности проверки.
