# План-промпт: полноценные cron- и RRULE-расписания в планировщике

## Роль и цель

Ты работаешь в проекте `D:\DEV\SAND\inter-2-prep`. Нужно реализовать полноценную поддержку `schedule_kind = 'cron'` и
`schedule_kind = 'rrule'` в планировщике напоминаний, не ломая существующие сценарии `one_time` и `interval`.

Сейчас схема базы уже готова: `mem.scheduled_tasks` содержит `timezone`, `cron_expr`, `rrule`, `next_run_at`, а ENUM
`mem.task_schedule_kind` включает `one_time`, `interval`, `cron`, `rrule`. Но код вычисляет cron/RRULE как фиксированный
суточный шаг и поэтому не выполняет обещанное поведение из данных и интерфейсов.

Итог реализации: регулярные задачи по cron и RRULE должны получать корректный ближайший будущий `next_run_at` с учётом
часового пояса пользователя, проходить тесты и быть описаны в документации `docs/ai-bot-with-memory`.

## Что уже известно по текущему коду

- Планировщик находится в `src/pipeline/scheduler.js`.
- `computeFirstRun(task)` сейчас возвращает `new Date()` для cron/RRULE без `run_at`, то есть задача запускается сразу.
- `calculateNextRun(task)` сейчас для cron/RRULE возвращает `Date.now() + 86400000`.
- `createTask` уже пишет `timezone`, `cron_expr` и `rrule` в `mem.scheduled_tasks`.
- `extractTask` в `src/pipeline/scheduler.js` уже имеет схему с `timezone`, `cron_expr`, `rrule`, но системный промпт
  пока просит только вычислить `run_at`, без правил выбора cron/RRULE.
- Инструмент `scheduler_create_task` в `src/pipeline/tools.js` сейчас НЕ принимает `cron_expr`, `rrule` и `timezone` в
  JSON Schema инструмента, а исполнитель `schedulerCreateTask` не прокидывает `cron_expr` и `rrule` в `createTask`.
- В `agent.js` пользовательский часовой пояс доступен как `ctx.timezone`.
- Текущая документация `docs/ai-bot-with-memory/10-operations.md` прямо говорит, что полноценный разбор cron/RRULE ещё
  является точкой расширения.
- Проверки живут в `tests/run.js` и запускаются через `npm test`; отдельного unit-тестового раннера для чистых функций
  сейчас нет.

## Требования к поведению

1. `one_time` остаётся прежним: `computeFirstRun` берёт `run_at`, `calculateNextRun` после успешного запуска возвращает
   `null`, задача переводится в `completed`.
2. `interval` остаётся прежним по смыслу: первый запуск считается от текущего времени, следующий запуск после выполнения
   считается от момента выполнения/опорной точки, а не от старого `next_run_at`.
3. `cron`:
   - использует `task.cron_expr`;
   - считает первый запуск и следующий запуск как первый момент строго после опорной даты;
   - учитывает `task.timezone`;
   - выражение `0 9 * * 1-5` означает 09:00 по локальному времени пользователя в будние дни.
4. `rrule`:
   - использует `task.rrule`;
   - считает первый запуск и следующий запуск как первый момент строго после опорной даты;
   - должен поддерживать реальные iCalendar RRULE-строки;
   - для правил без `DTSTART` нужно явно определить опорный старт, чтобы результат не зависел от неочевидного дефолта
     библиотеки.
5. Невалидное cron/RRULE-выражение не должно зациклить воркер и не должно молча превращаться в ежедневное расписание.
   Минимально приемлемое поведение: зафиксировать ошибку в данных задачи или запуске и перевести задачу в `failed`, либо
   безопасно отложить её с явной диагностикой. Выбери вариант, который лучше ложится на существующую модель `runTask`.
6. Некорректный или пустой `timezone` должен иметь контролируемый fallback на `config.timezone` или `Europe/Moscow`, с
   тестом. Не допускай падения воркера из-за плохого IANA timezone.
7. Не добавляй миграцию без необходимости: схема уже содержит нужные поля. Если для диагностики нужна запись ошибки,
   сначала используй существующие `metadata`, `scheduled_task_runs.error_text`, `status`, `attempts`.

## Зависимости

Добавь зависимости в `package.json` через npm, чтобы обновился `package-lock.json`:

- `cron-parser` для cron-выражений;
- `rrule` для RRULE.

Перед использованием проверь актуальный ESM API установленных версий в `node_modules` или документации пакетов. Не
угадывай названия методов: у `cron-parser` API менялся между версиями.

## План реализации

### 1. Подготовить чистый слой расчёта дат

Вынеси вычисление расписаний в тестируемые функции в `src/pipeline/scheduler.js` или небольшой соседний модуль, если это
уменьшит связность. Минимальный набор:

- нормализация timezone с проверкой через `Intl.DateTimeFormat`;
- функция получения следующего запуска для cron;
- функция получения следующего запуска для RRULE;
- единая функция `computeNextRun(task, afterDate = new Date())` или аналог, которую используют и `computeFirstRun`, и
  `calculateNextRun`.

Сделай функции экспортируемыми, если это нужно для прямых тестов в `tests/run.js`. Экспорт не должен менять публичное
поведение воркера.

### 2. Переписать `computeFirstRun` и `calculateNextRun`

`computeFirstRun(task)`:

- для `one_time` требует валидный `run_at`;
- для `interval` использует `interval_seconds`;
- для `cron` и `rrule` считает ближайший запуск строго после `new Date()`, а не запускает немедленно.

`calculateNextRun(task)`:

- для `one_time` возвращает `null`;
- для `interval` считает следующий запуск от текущего времени или явной опорной даты;
- для `cron` и `rrule` считает следующий запуск строго после текущего выполнения.

Избегай поведения, при котором регулярная задача после запуска снова получает `next_run_at <= now()` и выполняется в
плотном цикле.

### 3. Обновить `scheduler_create_task`

В `src/pipeline/tools.js` расширь JSON Schema инструмента:

- добавить `timezone`, `cron_expr`, `rrule` как поля с типом `string | null`;
- сделать их обязательными в `required`, как принято в проекте для структурированных nullable-полей, либо оставить
  необязательными только если это согласовано с текущим стилем схем инструментов;
- описать, когда использовать `one_time`, `interval`, `cron`, `rrule`;
- добавить короткие примеры: «каждый будний день в 9:00» → `schedule_kind = "cron"`, `cron_expr = "0 9 * * 1-5"`;
  «каждый месяц 1 числа в 10:00» → cron или RRULE по выбранной политике.

В `schedulerCreateTask` прокинь `args.cron_expr`, `args.rrule` и `args.timezone || ctx.timezone` в `createTask`.

### 4. Обновить извлечение задач

В `extractTask` промпт и схему уже допускают cron/RRULE, но инструкция неполная. Обнови системный промпт так, чтобы
модель:

- для разовых задач заполняла `run_at`;
- для простого «каждые N минут/часов/дней» использовала `interval`;
- для календарных регулярностей с конкретным локальным временем использовала `cron`;
- для сложных календарных правил использовала `rrule`;
- всегда возвращала `timezone` из входного часового пояса пользователя, если пользователь явно не указал другой.

Не заставляй модель вычислять `run_at` для cron/RRULE как суррогат первого запуска: первый запуск должен считать код.

### 5. Обработать ошибки расписаний

Сейчас ошибки выполнения задач обрабатываются внутри `runTask`, но ошибка расчёта следующего запуска может возникнуть
после успешной доставки уведомления. Определи и реализуй явную политику:

- если задача создана с невалидным расписанием, `createTask` должен либо отклонить её понятной ошибкой, либо создать в
  состоянии `failed` без немедленного выполнения;
- если ошибка обнаружена после выполнения регулярной задачи, запуск должен получить `failed` или задача должна перейти в
  `failed` с `error_text`, чтобы пользователь/оператор видел причину.

Важно: не должно быть тихого fallback на «завтра», потому что это скрывает дефект расписания и нарушает намерение
пользователя.

### 6. Добавить тесты

Расширь `tests/run.js` отдельным блоком рядом с проверками планировщика. Нужны минимум такие проверки:

- cron `0 9 * * 1-5` из опорной даты перед будним утром даёт ближайший будний день 09:00 в указанном timezone;
- та же cron-строка для двух разных timezone даёт разные UTC-моменты;
- cron после пятницы после 09:00 переносится на понедельник 09:00, а не на субботу;
- RRULE для ежедневного/еженедельного правила возвращает следующий момент после опорной даты;
- `computeFirstRun` для cron/RRULE не создаёт немедленный запуск при отсутствии `run_at`;
- невалидный cron/RRULE не приводит к плотному циклу и получает явную ошибку/failed-состояние;
- через `executeTool` или `handleMessage` можно создать cron-задачу так, чтобы в БД сохранились `cron_expr`, `timezone`
  и будущий `next_run_at`.

Если прямые проверки чистых функций требуют контролировать `Date.now()`, не делай глобальный monkey patch без отката.
Лучше передай опорную дату параметром в экспортируемую функцию.

### 7. Обновить документацию

Обнови документы после кода:

- `docs/ai-bot-with-memory/10-operations.md`: заменить текст о «точке расширения» на фактическое поведение cron/RRULE,
  описать timezone и ошибочные выражения.
- `docs/ai-bot-with-memory/08-prompts-and-models.md`: обновить `[PROMPT-4]` для выбора `one_time`/`interval`/`cron`/
  `rrule`.
- `docs/ai-bot-with-memory/09-proactivity.md` обновлять только если текст затрагивает общий воркер или пересекается с
  расписаниями; основной документ по планировщику здесь `10-operations.md`, не `09-proactivity.md`.
- `docs/ai-bot-implementation-status.md` или `claudedocs/ai-bot-implementation-status.md`: если файл существует и

- Комментарий-заглушку в `src/pipeline/scheduler.js` удалить или заменить точным описанием.

Соблюдай правило проекта: строки Markdown до 120 символов, цель 100-120 символов, без искусственно коротких строк.

## Критерии готовности

- `npm install` добавил зависимости и обновил lockfile.
- `npm test` проходит или, если он падает по внешней причине, в финальном ответе указана конкретная причина и какой
  более узкий тест/скрипт был пройден.
- Существующие проверки `one_time`, `interval`, ошибок задач и tool calling не ухудшились.
- Cron/RRULE больше не используют фиксированный суточный шаг.
- Часовой пояс пользователя влияет на UTC-значение `next_run_at`.
- Ошибочные выражения диагностируются явно и не создают бесконечный цикл выполнения.
- Документация в `docs/ai-bot-with-memory` соответствует фактическому коду.
