# План-промпт: пользовательский выбор TTS-голоса

## Задача

Реализовать выбор голоса для голосовых ответов обычной фразой пользователя. Пользователь может назвать конкретный
голос (`nova`, `onyx`, `ash` и т. п.) или попросить мужской/женский/нейтральный голос. Бот должен сохранить выбор как
устойчивую пользовательскую настройку и использовать её при каждом последующем вызове TTS.

Голоса и пол:

| voice | gender |
|-------|--------|
| `alloy` | `mf` |
| `ash` | `m` |
| `ballad` | `mf` |
| `cedar` | `m` |
| `coral` | `f` |
| `marin` | `f` |
| `nova` | `f` |
| `fable` | `mf` |
| `onyx` | `m` |
| `sage` | `f` |
| `verse` | `m` |

## Что уже есть в коде

В `src/config.js` есть глобальный `voiceOutput.voice`, сейчас читаемый из `VOICE_OUTPUT_VOICE` с fallback `alloy`.
`src/voice/tts.js` вызывает `audio/speech` и всегда передаёт именно `config.voiceOutput.voice`. Telegram-доставка в
`src/telegram/bot.js` идёт через `deliverAgentResult -> deliverVoice -> synthesizeSpeech`, а `deliverVoice` сейчас не
получает пользовательских настроек голоса из результата агента.

Форма ответа уже сделана правильно как управляющая пользовательская настройка: миграция `009_reply_mode.sql` добавляет
`mem.users.reply_mode`, `src/repo.js` содержит `setReplyMode/getUserReplyMode`, агент кладёт `replyMode` в `ctx`, а
инструмент `src/pipeline/agent-tools/set-reply-mode.js` меняет настройку по обычной фразе пользователя. Новый выбор
голоса должен повторить этот паттерн, а не полагаться только на асинхронное извлечение фактов в `memory_items`.

Обычная долговременная память (`mem.memory_items`) извлекается после ответа в `src/pipeline/extract.js` и пишется через
`src/pipeline/merge.js`. Это хорошо для профильных фактов, но плохо как единственный источник TTS-голоса: каналу нужен
точный `voice` в момент синтеза, а extractor может сработать позже, пропустить формулировку или записать
неструктурированный текст. Поэтому source of truth для голоса должен быть в `mem.users`, как `reply_mode`.

## Требуемое поведение

1. Если пользователь говорит «отвечай голосом nova», «поставь голос onyx», «используй ash» или похожую фразу, модель
   вызывает новый инструмент и сохраняет точный `voice`.
2. Если пользователь просит «мужской голос», «женский голос» или «нейтральный голос», бот выбирает детерминированный
   голос по умолчанию для категории и сохраняет его. Предлагаемые defaults: `ash` для `m`, `nova` для `f`, `alloy` для
   `mf`.
3. Если пользователь просит неизвестный голос, бот не должен записывать произвольную строку. Инструмент возвращает
   ошибку или список допустимых голосов, а ассистент коротко объясняет, что можно выбрать.
4. Смена голоса должна действовать уже на текущий голосовой ответ, как сейчас действует `set_reply_mode`: handler меняет
   и базу, и `ctx`.
5. Если пользователь находится в текстовом режиме, выбор голоса всё равно сохраняется. Он начнёт использоваться, когда
   пользователь включит голосовые ответы.
6. Если пользователь не выбирал голос, используется глобальный fallback из `VOICE_OUTPUT_VOICE`, после валидации по тому
   же каталогу голосов.

## План реализации

1. Добавить каталог голосов в отдельный модуль, например `src/voice/voices.js`.
   В модуле должны быть `VOICE_CATALOG`, `VOICE_IDS`, `DEFAULT_VOICE_BY_GENDER`, `normalizeVoiceId`,
   `resolveVoicePreference` и `isValidVoice`. Логика должна быть чистой и тестируемой без сети и БД.

2. Поддержать русские и английские алиасы в `resolveVoicePreference`.
   Минимум: точные имена голосов без учёта регистра; `мужской`, `male`, `man` -> default `ash`; `женский`, `female`,
   `woman` -> default `nova`; `нейтральный`, `универсальный`, `neutral` -> default `alloy`.

3. Добавить миграцию `015_voice_preference.sql`.
   В `mem.users` нужна nullable-колонка `voice_output_voice text` с `CHECK`, допускающим только значения каталога или
   `NULL`. `NULL` означает «использовать глобальный fallback из `config.voiceOutput.voice`». Миграция должна быть
   идемпотентной и не должна менять существующих пользователей на конкретный голос без явного выбора.

4. Обновить `src/config.js`.
   Валидировать `VOICE_OUTPUT_VOICE` через каталог голосов. Если env содержит неизвестное значение, безопасно падать на
   `alloy` или логировать предупреждение и использовать `alloy`; выбрать один стиль, который уже принят в проекте.

5. Обновить `src/repo.js`.
   Добавить `setVoicePreference(userId, voice)`, `getUserVoicePreference(externalId)` при необходимости каналу и helper,
   который возвращает эффективный голос: `user.voice_output_voice || config.voiceOutput.voice`. `ensureUser` уже
   возвращает всю строку пользователя, поэтому `handleMessage` сможет прочитать новую колонку без отдельного запроса.

6. Добавить инструмент `src/pipeline/agent-tools/set-voice-preference.js`.
   Инструмент должен быть включён только при `config.voiceOutput.enabled`, иметь английские descriptions, русский
   `title`, строгую JSON-схему и handler, который валидирует `voice`, пишет `mem.users.voice_output_voice`, кладёт
   выбранный голос в `ctx.voiceOutputVoice` и возвращает `{ voice_output_voice, gender }`.

7. Подключить инструмент в `src/pipeline/agent-tools/index.js`.
   Проверить, что он входит в базовые инструменты, не блокируется active skill и автоматически попадает в
   `buildToolDefs`. Название лучше сделать `set_voice_preference`, чтобы оно не смешивалось с `set_reply_mode`.

8. Усилить главный системный промпт в `src/agent.js`.
   Добавить правило: если пользователь просит выбрать голос, тембр или мужской/женский голос для голосовых ответов,
   нужно вызвать `set_voice_preference`; если просит перейти на голосовой/текстовый формат, нужно вызвать
   `set_reply_mode`.

9. Протянуть выбранный голос через результат агента.
   При создании `ctx` в `handleMessage` установить `voiceOutputVoice` из
   `user.voice_output_voice || config.voiceOutput.voice`. В return добавить `voiceOutputVoice`, чтобы канальный адаптер
   не ходил в БД после ответа.

10. Обновить TTS API в `src/voice/tts.js`.
    `synthesizeSpeech(text, opts = {})` должен передавать `voice: opts.voice || config.voiceOutput.voice`. Сохранить
    обратную совместимость тестов, где `opts.fetch` уже используется для подмены сети.

11. Обновить Telegram-доставку в `src/telegram/bot.js`.
    В `deliverVoice` брать `result.voiceOutputVoice || config.voiceOutput.voice` и вызывать
    `synthesizeSpeech(text, { voice })`. При логировании ошибки не раскрывать ключи и не печатать лишние данные запроса.

12. Обновить тесты.
    Добавить чистые тесты каталога голосов; расширить `tests/voice-output.test.mjs`, чтобы fake fetch проверял body с
    пользовательским `voice`; расширить слой голосового вывода в `tests/run.js`, чтобы проверить миграцию, tool handler,
    запись в `mem.users.voice_output_voice`, обновление `ctx.voiceOutputVoice` и возврат `voiceOutputVoice` из
    `handleMessage`.

13. Проверить сценарии вручную или через e2e.
    Минимум: «отвечай голосом», затем «выбери голос onyx», затем содержательный короткий вопрос. В логике должно быть
    видно, что `set_voice_preference` вызван, в БД сохранён `onyx`, а запрос `audio/speech` уходит с `voice: "onyx"`.
    Для Telegram Web использовать процедуру из `/test-telegram-bot`, если нужно подтвердить реальную доставку
    `sendVoice`.

14. Обновить документацию канала Telegram.
    В `docs/telegram/telegram-bot.md` описать выбор конкретного голоса обычной фразой, список поддерживаемых голосов,
    fallback при неизвестном голосе и то, что выбранный голос применяется только к голосовой доставке.

15. Финальным пунктом привести документацию `docs/ai-bot-with-memory` в актуальное состояние в соответствии с
    `docs/ai-bot-with-memory/00-documentation-principles.md`.
    Описывать функционал как единственное действующее состояние, без исторических формулировок. Не упоминать Telegram
    как обязательную цель. Обновить как минимум `05-data-schema.md` (новая колонка пользователя), `06-memory.md`
    (предпочтения формы ответа и голоса как управляющие пользовательские настройки), `08-prompts-and-models.md` (новый
    инструмент и его tool definition), `10-operations.md` (покрытие инструмента и тесты) и при необходимости
    `02-criteria.md`/`README.md`.
