# Предложение: навыки агента (Agent Skills) поверх механизма доменов

## 1. Цель и краткая суть

Сейчас у агента есть понятие **домена** (специализации): входящее сообщение классифицируется дешёвой моделью,
ей назначается `domain_key`, и этот ключ определяет раздел памяти, схему предметных данных и подсказку
классификатору. Задача — дать возможность **добавлять агенту навыки (Skills)**: бот «берёт навык на баланс»
(узнаёт о его существовании и назначении) и **маршрутизирует запрос в нужный навык по контексту**, подключая
к ответу инструкции навыка, его инструменты и, при необходимости, справочные материалы.

Навык — это самодостаточный пакет способности в формате стандарта Anthropic «Agent Skills»: каталог с файлом
`SKILL.md` (инструкция плюс метаданные во фронтматтере YAML), необязательными справочными файлами и привязкой
к инструментам. Ключевая идея навыка — **прогрессивное раскрытие** (progressive disclosure): в контекст модели
постоянно загружается только короткое описание навыка (чтобы решить, нужен ли он), а полная инструкция и
тяжёлые справочные материалы подгружаются лишь тогда, когда навык действительно выбран. Это снимает главную
проблему «раздувания контекста» при росте числа специализаций.

Предлагается рассматривать навык как **расширение и обобщение** существующего домена, а не как полностью
отдельный параллельный механизм. Домен в нынешнем виде — это «тонкий» навык без тела-инструкции и без привязки
инструментов; навык добавляет ровно то, чего домену не хватает.

## 2. Что есть сейчас (точка отсчёта)

Чтобы предложение опиралось на факты, ниже зафиксировано текущее устройство по коду.

- **Реестр доменов** — таблица `mem.agent_domains` (`migrations/001_init.sql`). Колонки `domain_key`, `title`,
  `description`, а также `default_tools jsonb` и `memory_policy jsonb`. Поиск по коду показывает, что
  `default_tools` и `memory_policy` **нигде не читаются** — это «заготовки на будущее». Базовые домены:
  `general`, `flight_search`, `joke_teller`, `math_tutor`.
- **Маршрутизация** — `src/pipeline/classify.js`. Системная подсказка собирается динамически: перечень доменов
  берётся из `listDomains()` (`src/repo.js`), модель возвращает JSON с полем `domain_key` (плюс `intent`,
  `entities`, `needs_memory`, `needed_memory_scopes`, `needs_tools`, `candidate_tools`). Поля `needs_tools` и
  `candidate_tools` сейчас классификатор заполняет, но `src/agent.js` их **не использует**.
- **Применение домена** — `src/agent.js`. После классификации `effectiveDomain` влияет на: выборку памяти
  (`retrieveMemory` по `domain_id`), блок памяти в промпте, темы диалога, привязку диалога (`ensureConversation`).
  **Тела-инструкции у домена нет**: единственный устойчивый системный текст — общий `MAIN_SYSTEM`, одинаковый для
  всех доменов.
- **Инструменты** — `src/pipeline/tools.js` и `src/pipeline/agent-tools/**`. Каждый инструмент — модуль с полями
  `name`, `title`, `definition`, опциональными `requiresAdmin` и `isEnabled(ctx, config)`, и `handler`. Набор
  инструментов для запроса собирает `buildToolDefs(ctx)`, фильтруя **только** по глобальным флагам и правам
  (`isEnabled`). **Привязки инструментов к домену нет**: например, `search_flights` не имеет `isEnabled` и
  поэтому подключается всегда, независимо от того, выбран ли домен `flight_search`.
- **Схемы предметных данных** — `src/schema/registry.js` и таблица `mem.domain_schemas` (`migrations/006`).
  Источник истины в рантайме — БД, файлы `schemas/*.json` служат черновиками для ревью. Схема описывает сущности
  домена и закрытую схему `data` для валидации фактов.

Вывод: **навык ≈ домен + тело-инструкция + прогрессивное раскрытие справочников + привязка инструментов +
выбор модели**. Большинство «крючков» для этого в коде уже есть (классификатор уже умеет возвращать
кандидатов-инструменты; у доменов уже зарезервированы `default_tools`/`memory_policy`).

## 3. Как стандарт Skills ложится на пайплайн

Стандарт Anthropic описывает три уровня прогрессивного раскрытия. Их естественное отображение на этот проект:

| Уровень | Что это в стандарте Skills | Куда ложится в проекте |
|---------|-----------------------------|------------------------|
| Уровень 1 | Фронтматтер: `name` + `description` + `when_to_use` (всегда в контексте) | Список навыков в подсказке роутера (`classify.js`) — «навык на балансе» |
| Уровень 2 | Тело `SKILL.md` (загружается при выборе навыка) | Системный блок инструкции, подмешиваемый в `src/agent.js` только для выбранного навыка |
| Уровень 3 | Файлы `references/**` (подгружаются по необходимости) | Новый инструмент `skill_read_reference`, который агент вызывает сам при нужде |

Таким образом «взять навык на баланс» = зарегистрировать навык так, чтобы его описание (уровень 1) попало в
подсказку роутера. «Сроутить в навык по контексту» = роутер выбрал `skill_key`, и тело навыка (уровень 2) плюс
его инструменты подключились к основному ответу. Тяжёлые справочники (уровень 3) не грузятся, пока агент сам их
не запросит, — это и есть экономия контекста.

## 4. Предлагаемая модель навыка

### 4.1. Структура каталога навыка

Источник истины для содержательной части навыка — файлы на диске (это соответствует стандарту, переносимо
между средами и даёт «живую» перезагрузку при правке). Пример:

```
skills/
└── flight-search/
    ├── SKILL.md            # обязательный: фронтматтер + инструкция (уровень 1 и 2)
    ├── references/         # необязательный: тяжёлые материалы (уровень 3)
    │   └── airlines.md
    └── data-schema.json    # необязательный: схема предметных данных (как нынешние schemas/*.json)
```

### 4.2. Фронтматтер `SKILL.md` (метаданные навыка)

К стандартным полям стандарта добавляются поля, отражающие специфику этого проекта (привязка к разделу памяти,
к инструментам и к модели). Все добавочные поля идут в блоке `metadata`, чтобы файл оставался валидным навыком
стандарта и переносимым.

```yaml
---
name: flight-search
description: >
  Поиск авиабилетов: рейсы, аэропорты, даты вылета и пересадки, сравнение цен.
  Использовать, когда пользователь спрашивает про билеты, перелёты, маршруты или даты вылета.
metadata:
  domain_key: flight_search        # раздел памяти и схема (связь с mem.agent_domains)
  allowed_tools: [search_flights]  # какие инструменты подключать, когда навык активен
  model: gpt-5.4-mini              # необязательное переопределение основной модели
  memory_scopes: [profile, domain, dialog]  # подсказка роутеру по нужным разделам памяти
  enabled: true                    # навык «на балансе» (можно гасить без удаления файла)
---

# Поиск авиабилетов

## Как действовать
1. Уточни маршрут и дату только если их нельзя вывести из запроса и памяти.
2. Для поиска вызови инструмент `search_flights`.
3. Покажи 2–3 варианта, отметь ночные рейсы и разницу в цене.

## Когда смотреть справочник
Список особенностей авиакомпаний и правил багажа — в `references/airlines.md`. Читай его только если вопрос
касается конкретного перевозчика или провоза багажа.
```

Поле `description` (плюс необязательное `when_to_use`) — это «уровень 1». Именно его видит роутер. Тело после
фронтматтера — «уровень 2». Файлы `references/**` — «уровень 3».

### 4.3. Связь навыка и домена

Чтобы не ломать существующую модель памяти (внешние ключи `domain_id` в `memory_items`, `conversations`,
`secure_records`, `scheduled_tasks`), навык **не заменяет** `domain_key`, а ссылается на него через
`metadata.domain_key`. Раздел памяти и схема данных по-прежнему адресуются доменом. Навык добавляет сверху
тело-инструкцию, привязку инструментов и выбор модели.

- Соответствие «один навык → один домен» — самый простой и рекомендуемый вариант на старте.
- Несколько навыков могут ссылаться на один и тот же `domain_key`, если делят общий раздел памяти (например,
  «репетитор по алгебре» и «репетитор по геометрии» внутри домена `math_tutor`).
- Если у навыка нет своего домена, он наследует `general` — то есть пишет в общий раздел памяти.

## 5. Изменения в коде (по компонентам)

Ниже — перечень точечных изменений. Все они аддитивные и закрываются единым флагом конфигурации, поэтому при
выключенном флаге поведение остаётся ровно прежним (доменным).

### 5.1. Реестр навыков — новый модуль `src/pipeline/skills/registry.js`

Отвечает за чтение каталога `skills/`, разбор фронтматтера, кэширование и три уровня доступа:

- `listSkills()` — уровень 1: массив `{ skill_key, description, when_to_use, domain_key, enabled }` для роутера.
  Кэшируется в памяти процесса по аналогии с `domainCache`/`schemaCache`; сбрасывается при изменении файлов.
- `getSkillBody(skillKey)` — уровень 2: тело инструкции навыка (без фронтматтера).
- `getSkillReference(skillKey, relPath)` — уровень 3: содержимое файла из `references/**` с защитой от выхода
  за пределы каталога навыка (никаких `..` и абсолютных путей).
- `getSkillMeta(skillKey)` — метаданные (домен, инструменты, модель) для `agent.js` и `tools.js`.

Разбор фронтматтера — небольшой парсер YAML (в проекте уже есть зависимости; если нет — добавить минимальную).
«Живая» перезагрузка при правке файла — необязательная опция; в простом варианте кэш сбрасывается по сигналу
администратора или по времени жизни.

### 5.2. Роутер — расширение `src/pipeline/classify.js`

- В подсказку роутера вместо/вместе со списком доменов подставляется **список навыков** (уровень 1):
  `skill_key`, `description`, `when_to_use`.
- В JSON-схему ответа добавляется поле `skill_key` (и, при желании, `skill_confidence`). Поле `domain_key`
  остаётся для обратной совместимости и выводится из навыка, если роутер вернул `skill_key`.
- **Залипающая маршрутизация** (sticky routing): чтобы навык не «прыгал» от реплики к реплике, переключение
  на другой навык происходит только если уверенность выше порога; иначе сохраняется текущий навык диалога.
- Если ни один навык не подходит — `general` (поведение «по умолчанию» как сейчас).

### 5.3. Основной пайплайн — `src/agent.js`

- После классификации определяется `ctx.skillKey`, из него выводится `effectiveDomain` (раздел памяти).
- Тело навыка (уровень 2) подмешивается **отдельным системным блоком** в массив `messages`, ниже стабильного
  `MAIN_SYSTEM` (чтобы не ломать кэширование общего префикса), но как справочная инструкция, а не как команда —
  в том же духе, что и существующие блоки `MEMORY_CONTEXT`/`CONVERSATION_CONTEXT`.
- Если у навыка задано `metadata.model`, основной вызов модели использует его вместо `config.llm.mainModel`.
- Набор инструментов собирается с учётом активного навыка (см. 5.4).

### 5.4. Привязка инструментов — `src/pipeline/tools.js`

- `buildToolDefs(ctx)` дополняется фильтром по навыку: инструмент попадает в набор, если он глобально включён
  (нынешний `isEnabled`) **и** (входит в `allowed_tools` активного навыка **или** относится к базовому набору).
- **Базовый набор** (доступен всегда, вне зависимости от навыка): управление личной памятью
  (`memory_list`, `memory_search`, `memory_forget_entity`, `memory_forget_all`), `set_reply_mode`, инструменты
  глобальной памяти. Это инструменты «про самого пользователя/бота», а не про предметную область.
- Это закрывает текущую дыру, когда предметные инструменты (например, `search_flights`) подключались всегда.
  Если у навыка `allowed_tools` не задан — поведение прежнее (все глобально включённые инструменты доступны),
  чтобы миграция была плавной.

### 5.5. Новый инструмент `skill_read_reference` (уровень 3)

Модуль в `src/pipeline/agent-tools/skills/`. Принимает имя справочного файла активного навыка, возвращает его
содержимое. Включается только когда активен навык, у которого есть каталог `references/`. Так тяжёлые материалы
не грузятся в каждый запрос, а подтягиваются строго по необходимости — это и есть выигрыш прогрессивного
раскрытия.

### 5.6. Конфигурация — `src/config.js`

Новый раздел `skills` с флагом включения и параметрами, по образцу существующих необязательных контуров
(`companion`, `proactive`, `globalMemory`):

```js
skills: {
  enabled: flag(env.SKILLS_ENABLED, false), // главный флаг: при выключенном — прежнее доменное поведение
  dir: env.SKILLS_DIR || 'skills',          // каталог навыков
  topK: Number(env.SKILLS_TOP_K || 1),      // сколько навыков подключать максимум (1 = строго один)
  switchThreshold: Number(env.SKILLS_SWITCH_THRESHOLD || 0.6), // порог переключения для залипающей маршрутизации
},
```

### 5.7. Администрирование (необязательно, второй этап)

Тонкая таблица `mem.agent_skills` (`skill_key`, `domain_key`, `enabled`, `version`, `updated_at`) — только для
рантайм-управления (включить/выключить навык «на баланс» без правки файлов и для аудита). Содержательная часть
(`SKILL.md`, справочники) остаётся в файлах. Это компромисс между переносимостью файлов и привычной для проекта
управляемостью через БД.

## 6. Поток выполнения (как это работает на одном запросе)

1. Пользователь пишет сообщение. `src/agent.js` вызывает роутер.
2. Роутер видит список навыков (уровень 1) и возвращает `skill_key` (или `general`) с учётом залипания.
3. `agent.js` подмешивает тело навыка (уровень 2) системным блоком и выводит `effectiveDomain` для памяти.
4. `buildToolDefs` отдаёт базовые инструменты плюс инструменты навыка (`allowed_tools`).
5. Модель отвечает; при необходимости вызывает инструмент навыка или `skill_read_reference` (уровень 3).
6. Память пишется в раздел домена навыка; схема данных берётся из домена, как и сейчас.

## 7. Соотношение с доменами: расширение, а не замена

Рекомендуемая трактовка — **навык обобщает домен**:

- Домен остаётся «адресом раздела памяти и схемы данных» (внешние ключи, валидация). Ничего не ломаем.
- Навык становится «единицей способности»: маршрутизация, инструкция, инструменты, модель, справочники.
- Существующие домены конвертируются в навыки один к одному: для каждого создаётся `skills/<key>/SKILL.md`
  с `metadata.domain_key`, описание берётся из `description` домена, тело пишется заново (короткая инструкция),
  `allowed_tools` проставляются по смыслу (`flight_search` → `search_flights`, и так далее).

Полная замена доменов на навыки возможна как отдалённая цель, но не нужна для запуска и потребовала бы трогать
внешние ключи памяти — это лишний риск без выгоды на старте.

## 8. План внедрения по этапам

1. **Этап 1 — каркас за флагом.** Реестр навыков (5.1), флаг конфигурации (5.6), роутер с `skill_key` (5.2),
   подмешивание тела и выбор модели (5.3), фильтр инструментов (5.4). Один пилотный навык (`flight-search`)
   как образец. При выключенном флаге — поведение прежнее. Тесты роутинга (срабатывает/не срабатывает) и
   фильтрации инструментов.
2. **Этап 2 — прогрессивное раскрытие и админ.** Инструмент `skill_read_reference` (5.5), таблица
   `mem.agent_skills` и переключение навыков «на баланс» (5.7), залипающая маршрутизация.
3. **Этап 3 — миграция доменов.** Перевод `general`, `joke_teller`, `math_tutor` в навыки; перенос схем
   данных рядом с навыками; постепенный отказ от прямого доменного роутинга в пользу навыкового.

## 9. Риски и компромиссы

- **Контекст на каждый запрос.** Тело навыка добавляется в каждый ответ выбранного навыка. Снижение риска —
  держать тело коротким (инструкция, а не справочник), а объём выносить в `references/**` (уровень 3).
- **Точность роутинга при росте числа навыков.** Чем больше навыков, тем длиннее список описаний у роутера и
  выше шанс ошибки. Снижение — лаконичные `description` с явными триггер-фразами, залипающая маршрутизация,
  ограничение `topK` (по умолчанию 1).
- **Композиция нескольких навыков** (`topK > 1`) раздувает контекст и усложняет привязку памяти. На старте —
  строго один навык; множественный выбор — позже и осознанно.
- **Файлы против БД.** Источник истины для `SKILL.md` — файлы (переносимость, «живая» правка), но проект
  привык управлять состоянием через БД. Компромисс — тонкая таблица только для флагов/версий (5.7).
- **«Живая» перезагрузка.** В долгоживущем процессе удобна, в бессерверной среде неактуальна. Делается
  необязательной опцией, по умолчанию — кэш со сбросом по сигналу администратора.
- **Совместимость.** Главный флаг `SKILLS_ENABLED=false` гарантирует прежнее доменное поведение до полной
  готовности и проверки навыкового контура.

## 10. Открытые вопросы для согласования

1. **Хранение содержания навыков** — файлы на диске (рекомендуется, по стандарту) или, как у схем доменов,
