# Известная проблема: тест разовой задачи планировщика падает на этой машине

## Краткая суть

В общем наборе тестов (`npm test`, файл `tests/run.js`) стабильно падает один тест —
«9. Планировщик выполняет разовую задачу ровно один раз». Остальные тесты проходят: на момент записи итог
составляет 110 пройденных и 1 проваленный. Проблема **не связана** с переносом телеграм-адаптера в каталог
`src/telegram/` и выносом раскладки эмодзи: тест планировщика не импортирует ни один из изменённых при том
переезде модулей, а его падение воспроизводится одинаково и до, и после изменений.

## Где именно

- Тест: `tests/run.js`, блок «9. Планировщик выполняет задачу один раз», строки примерно 337–347.
- Проверяемое условие: `runs === 1 && outbox === 1 && status === 'completed'`.
- Наблюдаемый результат (фактическая диагностика теста): **`успехов 0, outbox 0, статус active`**.

То есть разовая задача с уже наступившим временем запуска не была выполнена **ни разу**. Это важно: проблема
не в повторном выполнении (как можно было бы предположить по названию теста «ровно один раз»), а в том, что
задача вообще не попала в обработку. Её статус так и остался `active`, запись об успешном запуске в
`mem.scheduled_task_runs` не появилась, и сообщение в очередь доставки `mem.notification_outbox` не легло.

## Как устроена выборка задач

Тест создаёт задачу так (упрощённо):

```js
run_at: new Date(Date.now() - 1000).toISOString()   // время запуска — на 1 секунду в прошлом по часам Node
```

Затем дважды вызывает `tick()`. Внутри `tick()` просроченные задачи забирает функция `claimDueTasks`
(`src/pipeline/scheduler.js`, строки 289–306) запросом с условием:

```sql
WHERE status = 'active' AND next_run_at <= now()
  AND (locked_until IS NULL OR locked_until < now())
ORDER BY priority ASC, next_run_at ASC
LIMIT $1
FOR UPDATE SKIP LOCKED
```

Ключевой момент: `next_run_at` для разовой задачи вычисляется из `run_at` (то есть из значения, посчитанного
по часам **процесса Node**), а функция `now()` в условии отбора — это текущее время на стороне **сервера
PostgreSQL**. Запас между ними в тесте составляет всего одну секунду.

## Ведущая гипотеза причины: рассинхронизация часов Node и PostgreSQL

Если часы сервера PostgreSQL отстают от часов процесса Node больше чем на одну секунду, то вычисленное по
Node значение `next_run_at` (примерно «текущее время Node минус одна секунда») оказывается всё ещё в будущем
относительно `now()` на стороне PostgreSQL. Тогда условие `next_run_at <= now()` ложно, задача не попадает в
выборку `claimDueTasks`, и оба вызова `tick()` её не видят. Это в точности даёт наблюдаемую картину: ноль
выполнений, статус остаётся `active`.

Такая рассинхронизация типична, когда PostgreSQL работает в контейнере или в WSL2 на Windows-хосте: дрейф
часов виртуальной среды относительно хоста на одну-две секунды — известное явление. Запаса в одну секунду,
заложенного в тесте, для надёжности недостаточно.

## Запасная гипотеза: накопление задач в общей тестовой базе

Функция `claimDueTasks` берёт за один проход не более 20 задач (`LIMIT 20`). Тесты работают на общей базе
данных и создают много задач планировщика. Если от предыдущих прогонов набора в базе остались незавершённые
задачи в статусе `active` с уже наступившим временем, они конкурируют за место в выборке. При большом числе
просроченных задач задача данного теста может не попасть в первые две пачки по 20 штук, и тогда два вызова
`tick()` до неё не доходят. Эта гипотеза менее вероятна, чем рассинхронизация часов, но её стоит проверить,
если первая не подтвердится.

## Как воспроизвести

1. Запустить полный набор: `npm test`.
2. В выводе найти строку с пометкой провала: `❌ 9. Планировщик выполняет разовую задачу ровно один раз —
   успехов 0, outbox 0, статус active`.
3. Падение стабильно повторяется от запуска к запуску на этой машине.

## Как проверить и устранить

- **Подтвердить рассинхронизацию часов.** Сравнить время на стороне базы и на стороне Node: выполнить
  `SELECT now()` через то же соединение и сопоставить с `new Date()` в Node в тот же момент. Если расхождение
  больше секунды — подтверждается ведущая гипотеза.
- **Сделать тест устойчивым к расхождению часов.** Увеличить запас прошлого времени в тесте (например, задавать
  `run_at` на минуту в прошлом вместо одной секунды) либо задавать момент запуска от времени самой базы
  (`now() - interval '1 minute'`), а не от часов Node. Это убирает зависимость теста от синхронности часов.
- **Проверить запасную гипотезу.** Перед блоком теста посмотреть, сколько в базе просроченных активных задач
  (`SELECT count(*) FROM mem.scheduled_tasks WHERE status='active' AND next_run_at <= now()`). Если их заметно
  больше двадцати — проблема в накоплении задач от прошлых прогонов, и нужна очистка тестовой базы между
  запусками либо изоляция данных теста.

## Статус

Это предшествующее состояние тестовой среды на данной машине, не вызванное текущими изменениями кода и не
блокирующее перенос телеграм-адаптера. Профильные тесты переезда (`test:reactions`, `test:voice`,
`test:voice-output`) проходят полностью. Здесь зафиксирована именно диагностика и направления проверки, чтобы
к ней можно было вернуться отдельной задачей.
