# Улучшенная дедупликация памяти: план-промпт для Codex

## Задача

Реализуй более качественную дедупликацию долговременной памяти в проекте `inter-2-prep`. Текущий механизм в
`src/pipeline/merge.js` дедуплицирует в основном точные совпадения одной сущности внутри одного `scope`, поэтому в
активной памяти накапливаются смысловые дубли: стиль общения, имя ассистента, feature requests, напоминания и контекст
поиска билетов повторяются в разных `scope`, `memory_kind`, `domain_id` и с разными `entity_key`.

Цель изменения: сохранить быстрый и надёжный дедуп при записи нового факта, добавить безопасный maintenance-механизм для
ретроактивной очистки уже накопленных дублей, покрыть это тестами и привести спецификацию памяти к новому поведению.

## Контекст текущей реализации

Перед началом прочитай `src/pipeline/merge.js`, `src/pipeline/extract.js`, `src/schema/validate.js`,
`src/pipeline/retrieve.js`, `src/pipeline/scheduler.js`, `migrations/001_init.sql`, `tests/run.js`,
`tests/schema.test.js`, `docs/ai-bot-with-memory/06-memory.md` и `docs/ai-bot-with-memory/11-per-domain-schema.md`.
Сейчас `findSimilar()` фильтрует кандидатов по `user_id`, `status='active'` и тому же `scope`; затем ищет совпадение по
`entity_type/entity_key` или `search_tsv`. `decideMerge()` обновляет/заменяет только ту же сущность или нормализованно
одинаковый текст. Это нужно сохранить как быстрый строгий путь, но расширить его смысловым слоем.

## Инварианты

Не ломай существующий контракт приватности: `sensitivity='high'` и `sensitivity='secret'` не сохраняются как обычные
факты без подтверждения, а maintenance-дедуп не должен раскрывать секретные значения и не должен трогать защищённые
записи вне `mem.memory_items`. Удаление дублей должно быть мягким: лишние строки переводятся в `status='archived'`, в
`metadata` пишется аудит, а физическое удаление не вводится.

Не делай LLM-вызов обязательным для каждого кандидата памяти. Быстрые и очевидные случаи должны решаться локальными
правилами и SQL. Модель можно использовать только для спорных групп, где есть высокий риск смысловой близости, но нет
достаточно строгого совпадения по ключам. Если LLM недоступна, система должна деградировать до rule-based поведения.

Не смешивай домены без правила совместимости. Дедуп может пересекать `scope`, но должен учитывать смысловую группу:
профильные предпочтения можно сравнивать с системными инструкциями о стиле пользователя, feature requests можно
сравнивать между `profile`, `domain` и `dialog`, а предметные факты разных доменов нельзя сливать без общего ключа.

## План реализации

1. Добавь устойчивую смысловую идентичность факта: `dedupe_key` и `canonical_group_id`. Минимальный вариант — новые
   колонки в `mem.memory_items`; если миграция получается слишком тяжёлой, допустим промежуточный вариант с хранением
   `dedupe_key` и `canonical_group_id` в `metadata`, но предпочтительнее явные колонки и индексы. Добавь миграцию с
   `dedupe_key text`, `canonical_group_id uuid`, `dedupe_status text` и индексом по `(user_id, dedupe_key)` для активных
   строк. Значения `dedupe_status` используй как простые строки: `canonical`, `duplicate`, `superseded`, `candidate`.

2. Реализуй генератор `dedupe_key` в отдельном модуле, например `src/pipeline/memory-dedupe.js`, чтобы логика не
   разрасталась внутри `merge.js`. Генератор должен принимать нормализованный candidate и возвращать стабильный ключ
   вида `profile:communication_style:short_direct_answers`, `profile:assistant_name:bobik`,
   `feature_request:streaming_responses`, `feature_request:global_memory`,
   `flight_search:trip:sgn_mow_2026_06_16_2_adults_baggage`. Для неизвестных случаев используй аккуратный fallback из
   `domainKey`, `scopeGroup`, `entity_type`, `entity_key` и нормализованного короткого slug текста.

3. Введи понятие совместимых scope-групп для дедупа. `profile`, `system` и некоторые `domain`-инструкции о стиле могут
   попадать в группу `profile`; `domain goal`, `dialog reminder`, `profile goal` и `constraint` про одну функцию могут
   попадать в группу `feature_request`; `dialog open_loop`, `domain progress` и `domain goal` про одну поездку могут
   попадать в группу `trip_context`. Эти группы используются только для поиска дублей и не заменяют исходный `scope`.

4. Расширь `findSimilar()` так, чтобы он искал кандидатов шире текущего `scope`. Приоритеты поиска: тот же `dedupe_key`;
   та же `entity_type/entity_key` в совместимых scope-группах; близкое полнотекстовое совпадение; embedding-близость,
   если embedding доступен. Ограничь выборку по `user_id`, `status='active'`, приватности и разумному `LIMIT`, чтобы
   запись памяти не превращалась в дорогую операцию.

5. Замени бинарный `decideMerge()` на скоринговое решение. Очевидные случаи должны решаться локально: `score >= 0.92`
   автоматически обновляет или заменяет существующую запись, `score < 0.78` создаёт новую запись. Диапазон `0.78..0.92`
   считается спорным; для него добавь опциональное LLM-решение `MergeDecision`, но только если доступен кандидат на
   слияние и нет privacy-блокера. Схема решения: `merge`, `replace`, `keep_separate`, `archive_old`, `ask_confirmation`,
   `canonical_memory_text`, `canonical_scope`, `canonical_memory_kind`, `reason`.

6. Реализуй локальный scoring без модели. Учитывай совпадение `dedupe_key`, совпадение `entity_type/entity_key`,
   совместимость scope-группы, близость нормализованного текста, полнотекстовый матч, embedding-релевантность,
   совпадение ключевых полей `data`, близость `memory_kind`. Не делай один признак абсолютным, кроме точного
   `dedupe_key` и точной сущности в совместимой группе.

7. Улучши выбор canonical-записи при слиянии. Каноническая запись выбирается не просто по свежести, а по score:
   `importance`, `confidence`, `recency`, `specificity`, `usage_count` и заполненность `data`. Более конкретный текст
   должен побеждать общий: «короткие, прямые ответы без лишней церемонии» лучше, чем «короткие ответы». При обновлении
   сохраняй историю в `metadata.last_update`, а при архивировании дубля пиши `metadata.replaced_by` и блок
   `metadata.dedupe` с причиной, score, источником решения (`rule` или `llm`) и датой.

8. Добавь CLI/скрипт ретроактивной очистки, например `scripts/memory-dedupe.js`, и npm-команду `memory:dedupe`.
   Скрипт должен поддерживать `--user <uuid>`, `--all-users`, `--dry-run`, `--apply`, `--limit <n>` и `--json`.
   `--dry-run` является режимом по умолчанию и показывает группы дублей, canonical-запись, кандидатов на архивирование и
   причины. `--apply` реально архивирует дубли и должен требовать явного флага, чтобы случайный запуск не менял БД.

9. Добавь maintenance-задачу `memory_dedupe_cleanup`. Она должна вызываться из планировщика или отдельного
   экспортируемого раннера и обрабатывать память пачками. Не смешивай её с текущим `memory_cleanup`, который архивирует
   только записи с
   истёкшим `expires_at`. Новая задача должна уметь работать по пользователю, сохранять аудит и быть безопасной при
   повторном запуске: уже архивированные/заменённые строки повторно не трогать.

10. Уточни prompt извлечения в `src/pipeline/extract.js`, чтобы модель чаще выдавала стабильные сущности. Для
    повторяющихся feature requests используй стабильные `entity_type="feature_request"` и `entity_key`:
    `streaming_responses`, `global_memory`, `reminder_view`, `bot_menu`, `status_command`, `audio_recognition`,
    `emoji_reactions`, `log_analysis`, `self_development`. Для стиля общения используй стабильные ключи вроде
    `communication_style/short_direct`, `communication_style/streamed_answers`, `communication_style/text_not_voice`,
    `notification_style/first_person`.

11. Расширь тесты. В `tests/run.js` добавь проверки: один и тот же стиль общения в разных формулировках не создаёт
    активные дубли; один feature request, пришедший как goal/reminder/constraint, сводится к одной canonical-группе;
    поиск билетов с одинаковыми параметрами не размножается между `dialog open_loop`, `domain goal` и `domain progress`;
    `memory:dedupe --dry-run` ничего не меняет; `--apply` архивирует дубли и пишет `metadata.dedupe`. В
    `tests/schema.test.js` сохрани проверку дедупа по каноническому ключу и добавь кейс, где синонимы дают один
    `dedupe_key`. Прогони релевантные тесты и затем полный `npm test`, если окружение позволяет.

12. Добавь наблюдаемость. Для merge-решений возвращай из `processCandidate()` действие, `dedupe_key`, target id,
    canonical group id и источник решения. Не засоряй обычный ответ пользователю, но сделай это доступным тестам и
    debug-логам. Для CLI dry-run вывод должен быть понятным человеку и машинно-читаемым через `--json`.

13. Проверь реальный проблемный сценарий на пользователе `d2db2764-b24c-439b-b7d8-e6b410799dce` только в dry-run режиме.
    Не применяй изменения к его памяти без отдельного явного запроса. Dry-run должен показать группы вроде «Бобик»,
    «Шарик», «короткие прямые ответы», «стриминг», «глобальная память», «уведомления от первого лица», «Хошимин →
    Москва 16 июня» и задачи про bot-menu/status/audio/emoji/log-analysis/self-development.

14. Приведи документацию `docs/ai-bot-with-memory` в актуальное состояние строго в соответствии с
    `docs/ai-bot-with-memory/00-documentation-principles.md`: описывай новый дедуп как единственное действующее
    состояние, без исторических формулировок вроде «раньше», «теперь», «добавлено», «упрощение». Обнови согласованно
    обзор, критерии, схему данных, память, промпты/модели, операции, per-domain-схемы и историю сжатия, если затронуты.
