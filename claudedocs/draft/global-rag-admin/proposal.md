# Предложение: CRUD-интерфейс глобального RAG в админке

Дата: 2026-06-11. Статус: черновик на согласование. Прототип интерфейса: `prototype.html` рядом с этим файлом.

## 1. Задача

В админке нужен полноценный интерфейс управления глобальной базой знаний (таблица `mem.global_knowledge`),
которая сейчас редактируется только через инструменты агента (`global_knowledge_add/delete/search`) и команды
CLI (`/kb-add`, `/kb-find`, `/kb-del`). Требования из постановки:

1. CRUD над глобальным RAG прямо в админке.
2. После любого изменения или сохранения записи эмбеддинг формируется автоматически.
3. Отсутствие эмбеддинга видно в таблице.
4. При любом изменении текста триггер в БД сбрасывает эмбеддинг, чтобы исключить рассинхрон вектора и текста —
   даже если запись поменяли в обход приложения (psql, скрипт миграции, сторонний код).
5. Таблица поддерживает сортировки и фильтры по полям.
6. Форма редактирования — всплывающий диалог, ресайзится мышью со всех сторон.

## 2. Текущее состояние (что уже есть)

- Таблица `mem.global_knowledge` (migrations/001_init.sql:418–440): `id`, `domain_id`, `title`, `content`,
  `tags text[]`, `importance numeric(3,2)`, `status` (enum: active / archived / deleted / pending_confirmation /
  rejected), `source`, `created_by`, `embedding vector(1536)`, генерируемая `search_tsv`, `metadata jsonb`,
  `created_at`, `updated_at`. Индексы: HNSW по `embedding` (частичный, `WHERE embedding IS NOT NULL`), GIN по
  `search_tsv`, btree по `(domain_id, status)`.
- Эмбеддинг считается один раз при вставке в `addGlobalKnowledge()` (src/pipeline/global-memory.js:128–146)
  моделью `text-embedding-3-small` (1536 измерений). Если сервис эмбеддингов недоступен, запись создаётся с
  `embedding = NULL` и находится только полнотекстовым поиском. Обновления записей в коде нет вовсе.
- Триггеров на таблице нет; при изменении `content` через SQL вектор остаётся старым — это и есть тот рассинхрон,
  который требуется закрыть.
- Админка: Vue 3 + PrimeVue 4 (тема Aura, светлая). Вкладки объявлены в `web/src/components/App.vue` (массив
  `TABS`), таблицы — PrimeVue `DataTable` с `removable-sort` и фильтрами `filter-display="row"` (образец — вкладка
  «Память»). Готовый ресайзабельный диалог с восемью хэндлами и сохранением размера в localStorage уже реализован
  в `web/src/components/llm-log/PayloadView.vue:112–300` — паттерн переиспользуем.
- REST-эндпоинтов для глобального знания в `src/server/admin-api.js` нет; авторизация админки — HMAC-сессия в
  cookie `mb_admin_session`, ответ 401 автоматически отправляет фронт на экран логина.

## 3. Предлагаемое решение

### 3.1. Миграция 005: триггер сброса эмбеддинга

Новый файл `migrations/005_global_knowledge_embedding_trigger.sql`:

```sql
CREATE OR REPLACE FUNCTION mem.global_knowledge_reset_embedding()
RETURNS trigger AS $$
BEGIN
    -- Текст изменился, а новый вектор в этом же UPDATE не передан — старый вектор больше не описывает
    -- содержимое, сбрасываем его. Если приложение пишет текст и вектор одним UPDATE, сброса не происходит.
    IF (NEW.title IS DISTINCT FROM OLD.title OR NEW.content IS DISTINCT FROM OLD.content)
       AND NEW.embedding IS NOT DISTINCT FROM OLD.embedding THEN
        NEW.embedding := NULL;
    END IF;
    NEW.updated_at := now();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

DROP TRIGGER IF EXISTS trg_global_knowledge_reset_embedding ON mem.global_knowledge;
CREATE TRIGGER trg_global_knowledge_reset_embedding
    BEFORE UPDATE ON mem.global_knowledge
    FOR EACH ROW
    EXECUTE FUNCTION mem.global_knowledge_reset_embedding();
```

Свойства такого триггера:

- Любое изменение `title`/`content` в обход приложения (psql, скрипт) гарантированно обнуляет вектор — устаревший
  эмбеддинг не может участвовать в семантическом поиске, запись остаётся доступной через полнотекстовый фолбэк.
- Условие `NEW.embedding IS NOT DISTINCT FROM OLD.embedding` позволяет приложению записать текст и свежий вектор
  одним UPDATE без ложного сброса.
- Заодно триггер поддерживает `updated_at`, который сейчас никто не обновляет.

### 3.2. Жизненный цикл эмбеддинга при сохранении

Сценарий «сохранить запись» на сервере выполняется в два шага:

1. `UPDATE`/`INSERT` текста и атрибутов. Если текст менялся, триггер сбрасывает `embedding` в NULL — с этого
   момента рассинхрон невозможен по построению.
2. Сразу после записи сервер вызывает `embed(title + '. ' + content)` и вторым UPDATE кладёт вектор (текст в этом
   UPDATE не меняется, триггер его не трогает). Если сервис эмбеддингов недоступен, шаг тихо завершается, запись
   остаётся с `embedding = NULL`, а клиент получает в ответе `hasEmbedding: false` и показывает индикатор.

Дополнительно — эндпоинт ручного пересчёта (кнопка «⟳ пересчитать» у записей без вектора) и, опционально,
фоновая задача планировщика `embedding_repair`, которая раз в N минут добивает записи с `embedding IS NULL AND
status = 'active'` (по образцу `memory_cleanup` в src/pipeline/scheduler.js). Фоновую задачу предлагаю включить
сразу: она же закрывает случай, когда запись поменяли через psql и никто не нажал «пересчитать».

### 3.3. Серверная часть: API и слой данных

В `src/pipeline/global-memory.js` добавляются функции `listGlobalKnowledge()`, `updateGlobalKnowledge()`,
`reembedGlobalKnowledge()`; `addGlobalKnowledge()` переиспользуется. В `src/server/admin-api.js` — пять
эндпоинтов под существующей админ-сессией:

| Метод и путь                          | Назначение                                                              |
|---------------------------------------|-------------------------------------------------------------------------|
| `GET  /api/knowledge`                 | Список записей: все поля + `hasEmbedding` (сам вектор наружу не отдаём) |
| `POST /api/knowledge`                 | Создание; сервер считает эмбеддинг по схеме из 3.2                      |
| `PUT  /api/knowledge/:id`             | Обновление; та же схема с триггером и пересчётом                        |
| `DELETE /api/knowledge/:id`           | Мягкое удаление (`status = 'deleted'`), как в остальной системе         |
| `POST /api/knowledge/:id/embed`       | Ручной пересчёт эмбеддинга                                              |

`GET /api/knowledge` по умолчанию возвращает записи со статусами `active` и `archived`; удалённые доступны через
параметр `?status=deleted` (даёт «корзину» с возможностью восстановления — статус правится через PUT). Объём
глобальной базы знаний мал (десятки–сотни записей), поэтому список отдаётся целиком, а сортировка, фильтрация и
пагинация выполняются на клиенте средствами DataTable — ровно так уже работает вкладка «Память». Серверная
пагинация не нужна, пока база не вырастет до тысяч записей.

Форма ответа элемента списка:

```json
{
  "id": "uuid",
  "title": "string|null",
  "content": "string",
  "domainKey": "string|null",
  "tags": ["string"],
  "importance": 0.5,
  "status": "active",
  "source": "string|null",
  "hasEmbedding": true,
  "createdAt": "ISO",
  "updatedAt": "ISO"
}
```

### 3.4. Клиентская часть: вкладка «База знаний»

- В `App.vue` в массив `TABS` добавляется третья вкладка «База знаний» (видна только админам — как и вся админка,
  доступ уже ограничен сессией). Страница — новый компонент `web/src/components/knowledge/KnowledgePage.vue`,
  полноширинный, без сайдбара (как вкладка «Логи LLM»).
- Тулбар над таблицей: кнопка «Добавить запись», счётчик записей, индикатор «без эмбеддинга: N» (кликабельный —
  включает фильтр по отсутствию вектора).
- Таблица — PrimeVue `DataTable` (`size="small"`, `removable-sort`, `striped-rows`, `filter-display="row"`,
  `paginator` при росте объёма). Колонки:

| Колонка     | Содержимое                                            | Сортировка | Фильтр                          |
|-------------|--------------------------------------------------------|------------|----------------------------------|
| Эмбеддинг   | бейдж «✓» / «⚠ нет» (+ кнопка пересчёта при «нет»)    | да         | селект: все / есть / нет         |
| Заголовок   | `title`                                                | да         | текстовый contains               |
| Содержимое  | первые ~160 символов `content`, полностью — в диалоге  | да         | текстовый contains               |
| Домен       | ключ домена или «все домены»                           | да         | мультиселект из данных           |
| Теги        | чипы                                                   | —          | мультиселект из данных           |
| Важность    | число 0–1                                              | да         | —                                |
| Статус      | active / archived / deleted                            | да         | мультиселект                     |
| Источник    | `source`                                               | да         | текстовый contains               |
| Обновлено   | `updated_at`, локальный формат                         | да (деф.)  | —                                |
| Действия    | ✎ редактировать, × удалить                             | —          | — (класс `.col-actions`)         |

  Сортировка по умолчанию — «Обновлено», по убыванию. Мультиселект-фильтры строятся из фактических значений в
  данных, как на вкладке «Память» (`FilterMatchMode.IN`).
- Диалог создания/редактирования — компонент `KnowledgeDialog.vue` по паттерну ресайзабельного диалога из
  `PayloadView.vue`: телепорт в body, оверлей, восемь хэндлов ресайза (4 стороны + 4 угла), минимальный размер
  520×360, размер сохраняется в `localStorage` (ключ `knowledge.dialog.size`), закрытие по ESC. Поля формы:
  заголовок (input), содержимое (textarea, растягивается вместе с диалогом — главный мотив ресайза), домен
  (select со списком доменов + «все домены»), теги (chips-ввод), важность (число 0–1), источник (input), статус
  (select). Внизу — строка состояния эмбеддинга («будет пересчитан после сохранения» / «актуален») и кнопки
  «Сохранить» / «Отмена». После сохранения строка в таблице сразу показывает «⚠ нет», затем по ответу сервера
  (или после ручного обновления) — «✓».
- Удаление — нативный `confirm()`, как на вкладке «Память»; запись получает `status = 'deleted'` и пропадает из
  представления по умолчанию.

### 3.6. Порядок реализации

1. Миграция 005 (триггер) + юнит-проверка: UPDATE текста без вектора обнуляет `embedding`, UPDATE с вектором — нет.
2. Слой данных и REST-эндпоинты + тесты в `tests/run.js` (CRUD, сброс и пересчёт эмбеддинга, мягкое удаление).
3. Вкладка с таблицей (чтение, сортировки, фильтры, индикатор эмбеддинга).
4. Диалог создания/редактирования с ресайзом, удаление, ручной пересчёт.
5. Фоновая задача `embedding_repair` и актуализация документации.
