# Очистка памяти пользователем: анализ состояния и задание на инструменты агента

## Часть 1. Анализ: реализована ли очистка памяти и как пользователь может ею воспользоваться

### Что реализовано

Модуль `src/pipeline/admin.js` существует и содержит ровно три функции, как и требует критерий из
`ai-bot-with-memory/02-criteria.md` (строки 34–35):

- **`listMemory(userId, { includeArchived })`** — показывает активные записи памяти пользователя; с флагом
  `includeArchived` можно вытащить и архив, и помеченные на удаление записи.
- **`deleteMemory(userId, memoryId)`** — мягкое удаление одной записи: ставит `status='deleted'`, сама строка в
  таблице `mem.memory_items` остаётся, но выпадает из всех обычных выборок (они фильтруют по `status='active'`).
- **`forgetAll(userId)`** — полное забывание: переводит все активные записи пользователя в `status='deleted'` и
  возвращает количество затронутых записей.

Удаление везде именно мягкое — физически из базы ничего не стирается, остаётся след. Это совпадает с тем, что написано
в критерии и в `ai-bot-with-memory/06-memory.md` (строки 161–162).

Функции **покрыты обязательным тестом** удаления: в `tests/run.js` (строки 273–278) проверяется и удаление одной
записи через `deleteMemory`, и полное забывание через `forgetAll`.

### Чего нет — как пользователь реально чистит память

**Прямого способа у пользователя сейчас нет.** Функции реализованы на уровне модуля, но **ни к одному интерфейсу не
подключены**:

- в Telegram-боте (`src/telegram.js`) нет команды вида `/forget` или обработки фразы «забудь обо мне»;
- в командной строке (`src/cli.js`) нет команды управления памятью;
- агенту (`src/pipeline/tools.js`, `src/agent.js`) не выдан инструмент для вызова удаления.

Поиск по словам `forget`, `удали`, `очист` во всех точках входа не дал ни одного вызова. Единственные, кто вызывает эти
функции, — это тесты.

Таким образом, формально критерий «опорный модуль должен давать мягкое удаление и полное забывание» выполнен (модуль
даёт), но **функциональность не выведена пользователю**. Сейчас удалить память можно только программно — вызвав
`deleteMemory`/`forgetAll` из кода или теста, либо вручную через SQL.

## Часть 2. Задание: инструменты агента для работы с памятью пользователя

### Цель

Подключить готовую логику из `src/pipeline/admin.js` к агенту, добавив в `src/pipeline/tools.js` три новых инструмента
function calling, чтобы пользователь мог управлять своей памятью обычными фразами в диалоге («покажи, что ты обо мне
помнишь», «забудь про мой паспорт», «удали всё обо мне»).

### Инструмент 1 — просмотр памяти

- **Имя:** `memory_list`
- **Назначение:** показать пользователю список того, что бот о нём помнит. Вызывать, когда пользователь спрашивает, какие
  факты о нём сохранены.
- **Параметры:** `scope` (необязательный фильтр области: `profile` / `domain` / `dialog`); `include_archived` (булево,
  по умолчанию `false` — показывать ли удалённые записи).
- **Исполнитель:** опирается на существующую `listMemory(userId, { includeArchived })`. Возвращает массив записей с
  полями `id`, `entity_type`, `entity_key`, `memory_text`, `importance`, `status`.
- **Приватность:** не возвращать полные защищённые значения (поле `sensitivity` высокого уровня — паспорт, телефон и
  т. п.); для таких записей показывать только обобщённое название (`entity_type`/`entity_key`), как это уже делает
  `memory_search`.

### Инструмент 2 — удаление сущности по названию

- **Имя:** `memory_forget_entity`
- **Назначение:** мягко удалить из памяти конкретную сущность по её названию. Вызывать, когда пользователь просит забыть
  что-то конкретное («забудь мой адрес», «удали данные о машине»).
- **Параметры:** `entity_name` (строка — название сущности или ключ, например «адрес», «паспорт»); необязательный
  `entity_type` для уточнения, если названий несколько.
- **Исполнитель — требует доработки модуля.** Текущая `deleteMemory(userId, memoryId)` удаляет строго по идентификатору
  записи, а не по названию. Нужно добавить в `src/pipeline/admin.js` функцию `deleteByEntity(userId, entityName,
  entityType)`, которая находит активные записи по совпадению `entity_key`/`entity_type` (с нечётким сопоставлением имени)
  и мягко переводит их в `status='deleted'`. Возвращать количество удалённых записей и их перечень — чтобы агент мог
  подтвердить пользователю, что именно забыто. Если совпадений несколько и они разнородны — вернуть список кандидатов и
  попросить агента уточнить, прежде чем удалять.

### Инструмент 3 — очистка всей памяти

- **Имя:** `memory_forget_all`
- **Назначение:** полностью забыть всё об активной памяти пользователя. Вызывать только по явной и недвусмысленной
  просьбе («удали всё, что ты обо мне знаешь»).
- **Параметры:** `confirm` (булево) — защита от случайного срабатывания; исполнять, только если `confirm=true`.
- **Исполнитель:** опирается на существующую `forgetAll(userId)`. Возвращает количество удалённых записей.
- **Подтверждение:** в системном промпте агента предписать обязательное переспрашивание пользователя перед вызовом
  (необратимость с точки зрения пользователя, хотя физически удаление мягкое).

### Технические требования (общие)

- Описания инструментов добавить в массив `toolDefs`, исполнители — в объект `EXECUTORS` в `src/pipeline/tools.js`,
  по образцу уже имеющихся (`memory_search`, `scheduler_create_task`, `secure_record_get`).
- Каждый вызов проходит через `executeTool`, поэтому автоматически попадает в журнал вызовов инструментов
  (`logToolCall`) — отдельного логирования писать не нужно.
- Все операции выполняются строго в рамках `ctx.userId` — пользователь может видеть и удалять только свою память.
- Удаление остаётся **мягким** (`status='deleted'`), физическое стирание не вводить.

### Тесты

- Расширить `tests/run.js`: проверить новый инструмент `memory_forget_entity` (удаление по названию находит и помечает
  нужную запись, не трогает остальные) и прохождение всех трёх инструментов через `executeTool` с записью в журнал.
- Существующий тест удаления (строки 273–278) сохранить как проверку базовых функций модуля.
