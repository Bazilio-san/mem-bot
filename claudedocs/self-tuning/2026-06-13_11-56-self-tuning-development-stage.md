# Этап развития инфраструктуры тюнинга: реплей, регулярный прогон, панель качества

Детализация раздела 10 основного предложения
`claudedocs/self-tuning/2026-06-13_10-00-self-tuning-infrastructure.md`. Этап стартует только
после обкатки основного цикла (шаги 1–10 порядка реализации): все компоненты ниже опираются на уже работающие
прогонщик сценариев, `eval.js`, файл критериев `tests/eval/criteria.yaml` и дисциплину экономии контекста (§2a
основного документа).

Дата: 2026-06-13. Статус: предложение, не реализовано.

## 1. Реплей реальных диалогов

### 1.1. Зачем

Самые ценные регрессионные кейсы приходят не из придуманных сценариев, а из боевых разговоров — особенно
неудачных: бот не вспомнил факт, классификатор выбрал не тот скилл, ответ ушёл не в том формате. Реплей
превращает такие случаи в постоянные тесты: однажды разобранная ошибка больше не возвращается незамеченной.

### 1.2. Скрипт отбора `scripts/replay-capture.js`

Источник — `log.llm_request` и `log.agent_event`: по `request_id` восстанавливается полный цикл обработки,
по `conversation_id` и времени — последовательность реплик диалога. Работа по принципу послойного доступа:

```
# Слой 1: кандидаты — только метаданные (id цикла, время, kind'ы, цена, статус, признак отбора)
node scripts/replay-capture.js --candidates --errors --since 2026-06-01
node scripts/replay-capture.js --candidates --kind main_agent_answer --min-cost 0.05
node scripts/replay-capture.js --candidates --user <id> --around <request_id> --turns 5

# Слой 2: сборка сценария из выбранных циклов
node scripts/replay-capture.js --build --request-id <uuid> --turns-before 3 --out tests/scenarios/replays/
```

Критерии отбора кандидатов (комбинируются):

- циклы со статусом ошибки или с `request_kind: untyped`;
- циклы, на которые пользователь отреагировал негативно (переспросил, поправил бота — выявляется
  LLM-проходом по следующей реплике, отдельный `request_kind: 'eval_replay_select'`);
- аномально дорогие или медленные циклы (пороги — из секции мониторинга `criteria.yaml`);
- явное указание: админ в просмотрщике логов помечает запись «в регрессию» (см. §3.4).

Скрипт собирает из выбранного фрагмента диалога сценарий в формате §4.1 основного документа: реплики
пользователя как `turns`, исходный ответ бота — как `reference_answer` для судьи (что считалось плохим или
хорошим — указывается в метаданных кейса).

### 1.3. Анонимизация и приватность

Боевые диалоги содержат персональные данные, а сценарии попадают в git. Поэтому:

- обязательный проход анонимизации перед записью файла: имена, телефоны, адреса, документы заменяются
  правдоподобными вымышленными (LLM-проход + детерминированные регулярные выражения для телефонов/email;
  `request_kind: 'eval_anonymize'`);
- кейсы из циклов с `sensitivity: high` (секретница, медицина, финансы) в реплей не отбираются вообще —
  фильтр на слое кандидатов;
- собранный сценарий не коммитится автоматически: скрипт кладёт его в каталог реплеев, человек просматривает
  и коммитит сам (правило §7.2 основного документа о раздельной правке эталонных наборов).

### 1.4. Хранение

`tests/scenarios/replays/<дата>-<краткое-имя>.json` плюс строка в индексном файле
`tests/scenarios/replays/INDEX.md` (id, дата, одна строка сути, исходный `request_id`). Claude Code при
работе читает индекс, а не каталог целиком — экономия контекста по §2a.

## 2. Регулярный прогон (nightly)

### 2.1. Зачем

Качество деградирует и без правок кода: провайдер обновляет модель за тем же именем, меняются данные
в памяти, накапливаются побочные эффекты. Регулярный прогон ловит дрейф и случайные регрессии до того,
как их заметят пользователи.

### 2.2. Механика

- запуск ночью по расписанию: отдельная задача в существующем планировщике (`src/scheduler-run.js`)
  либо системный cron, вызывающий `node scripts/eval.js --suite all --label nightly`;
- прогон выполняется на текущем master с `NODE_ENV=test`; тестовые пользователи удаляются по завершении
  (`node scripts/delete-user.js --test-users --yes`);
- результаты сохраняются в `claudedocs/experiments/nightly/<дата>/` (тот же формат
  `summary.json` + `cases/`) и дополнительно в БД для панели (см. §3.2);
- сравнение — не с вчерашним днём, а со **скользящей базой**: медиана агрегатов за последние N успешных
  ночных прогонов. Это гасит шум недетерминизма LLM: одиночный плохой прогон не считается деградацией,
  устойчивое отклонение — считается;
- бюджет: ночной прогон подчиняется тому же стоп-лимиту `budget.eval_run_max_usd` из `criteria.yaml`;
  при превышении прогон останавливается и помечается неполным, а не молча урезается.

### 2.3. Оповещение о деградации

Канал доставки уже есть — сам бот. При выходе агрегатов за пороги `acceptance` относительно скользящей базы
планировщик отправляет администраторам (флаг `is_admin`) короткое сообщение: какой набор просел, дельта,
ссылка на отчёт. Никакой автоматической реакции на деградацию (откатов, правок) не предусматривается —
только сигнал человеку; разбор деградации затем выполняется обычной итерацией тюнинга через skill `tune-bot`.

### 2.4. Хранение истории и ретенция

Ночные результаты накапливаются месяцами — нужна ретенция: детальные `cases/` старше M дней удаляются,
`summary.json` хранится долго (он маленький и нужен для трендов панели). Механизм чистки — по образцу
существующего `src/pipeline/log-retention.js`.

## 3. Панель качества в админке

### 3.1. Зачем

Файлы `summary.json` отвечают на вопрос «что в этом прогоне», но не «как меняется качество со временем».
Панель — это взгляд на тренды: видно, когда началась деградация, что её вызвало (коммит, смена модели,
дрейф провайдера) и сколько стоит качество в деньгах.

### 3.2. Данные: таблицы результатов в БД логов

Панели нужен SQL-доступ к результатам, файлов в `claudedocs/` недостаточно. Две таблицы в базе логов
(рядом с `log.llm_request`):

- `log.eval_run` — один прогон: метка, время, git-коммит, ветка, модели, хеши промптов, агрегаты по наборам
  (доля пройденных, средние баллы по осям), суммарные токены/цена, перцентили задержки, признак nightly;
- `log.eval_case` — один кейс прогона: id кейса, набор, вердикт, баллы по осям, токены, цена, длительность,
  `request_id` (связь с `log.llm_request` для drill-down).

`eval.js` пишет в обе таблицы тем же батч-механизмом `src/pipeline/log-writer.js`, что и остальные логи.
Файлы `summary.json`/`cases/` при этом остаются — это рабочий формат Claude Code, таблицы — формат панели.

### 3.3. Интерфейс

Новая страница в существующей веб-админке (`web/`, API в `src/server/`):

- **Тренды**: графики по времени — средний балл судьи по осям, доля пройденных по наборам, цена прогона,
  p95 задержки. Вертикальные отметки на графике — коммиты и смены моделей (данные есть в `log.eval_run`);
- **Список прогонов**: таблица с агрегатами, фильтр nightly/эксперимент, сравнение двух любых прогонов
  (тот же отчёт, что строит `eval-compare.js`, но в вебе);
- **Drill-down кейса**: из провалившегося кейса — переход в существующий просмотрщик `log.llm_request`
  по `request_id`, где уже есть полный payload/response и AI-анализ записи;
- API: `GET /api/eval/runs`, `GET /api/eval/runs/:id/cases`, `GET /api/eval/trends` — по образцу
  существующих маршрутов `llm-log` (`src/server/llm-log-data.js`).

### 3.4. Обратная связь из панели в реплей

В просмотрщике записи лога — кнопка «в регрессию»: помечает цикл как кандидата для `replay-capture.js`
(§1.2). Так замыкается контур: админ заметил плохой ответ в логах → одним действием превратил его
в будущий регрессионный кейс → Claude Code на следующей итерации собрал сценарий и разобрал причину.

## 4. Вынос промптов из кода (опционально, в конце этапа)

Сейчас промпты — инлайн-константы в исходниках, их координаты ведёт `docs/prompt-inventory.md`. После
обкатки всего предыдущего можно рассмотреть вынос в файлы (`prompts/<request_kind>/<имя>.md`) с загрузчиком:

- `prompt_hash` в логе считается автоматически при загрузке файла — исчезает ручная синхронизация;
- диффы промптов в git становятся читаемыми (сейчас правка промпта тонет в диффе кода);
- `docs/prompt-inventory.md` генерируется из каталога, а не ведётся руками;
- эксперимент «только промпт» перестаёт трогать код вообще — чище сравнение по §7.2.

Это рефакторинг с риском (промпты собираются динамически, с интерполяцией — см. классификатор), поэтому
выполняется последним, по одному `request_kind` за итерацию, с прогоном оценок после каждого переноса.

## 5. Порядок реализации этапа

| Шаг | Что | Зависимости |
|-----|-----|-------------|
| 1 | Таблицы `log.eval_run` / `log.eval_case` + запись из `eval.js` | основной цикл обкатан |
| 2 | Ночной прогон в планировщике + скользящая база + оповещение через бота | шаг 1 |
| 3 | Ретенция результатов прогонов | шаги 1–2 |
| 4 | Панель качества: тренды, список прогонов, drill-down | шаг 1 |
| 5 | `scripts/replay-capture.js`: кандидаты, сборка, анонимизация | основной цикл обкатан |
| 6 | Кнопка «в регрессию» в просмотрщике логов | шаги 4–5 |
| 7 | Вынос промптов из кода (по одному request_kind) | опционально, последним |

Шаги 1–4 и 5 независимы друг от друга и могут идти параллельно; контур замыкается на шаге 6.
