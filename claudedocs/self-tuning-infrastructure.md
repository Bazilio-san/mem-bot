# Инфраструктура интеллектуального тюнинга бота

Предложение: что нужно построить, чтобы Claude Code мог самостоятельно улучшать код, промпты и архитектуру
бота — с измеримой оценкой качества, стоимости и производительности каждого изменения.

Дата: 2026-06-13. Статус: предложение, не реализовано.

## 1. Идея замкнутого цикла тюнинга

Самостоятельная модификация безопасна и полезна только внутри замкнутого цикла, где каждое изменение измеряется:

1. **Наблюдение** — Claude Code видит, что реально уходит в LLM и что возвращается (полные payload/response).
2. **Гипотеза** — на основе наблюдений формулируется конкретное изменение (промпт, код, параметр модели).
3. **Изменение** — правка в отдельной git-ветке.
4. **Прогон** — эталонные сценарии прогоняются через полный пайплайн без Telegram.
5. **Оценка** — автоматические проверки плюс LLM-судья выставляют оценки; считаются токены, цена, задержка.
6. **Сравнение** — отчёт «было/стало» по тем же сценариям; решение принять или откатить.
7. **Фиксация** — изменение вливается человеком после просмотра отчёта; результаты архивируются как новая база
   сравнения.

Шаги 1 и частично 4 уже обеспечены существующей инфраструктурой. Основная работа — шаги 4–6 и регламент.

## 2. Что уже есть (фундамент)

| Компонент | Где | Что даёт для тюнинга |
|-----------|-----|----------------------|
| Полный лог LLM-запросов | `log.llm_request` (отдельная БД логов), запись через `src/pipeline/llm-log.js` | payload и response целиком (до 100 КБ), вид запроса (`request_kind`, 16 видов), модель, токены, цена, длительность, корреляция `request_id`, флаг `is_test` |
| Журнал событий агента | `log.agent_event`, `src/pipeline/agent-event-log.js` | трассировка шагов пайплайна внутри одного запроса |
| CLI-чат через весь пайплайн | `src/cli.js` (`npm run chat`) | вызывает `handleMessage` из `src/agent.js` — тот же путь, что и Telegram, но без мессенджера |
| Реестр промптов | `docs/prompt-inventory.md` | координаты всех промптов и tool-описаний в коде (промпты лежат инлайн в исходниках) |
| Зачаток эталонного набора | `tests/memory_cases.json` | кейсы «сохранять/не сохранять факт» с ожидаемыми scope и sensitivity |
| Тестовый каркас | `tests/run.js`, ~25 файлов тестов | интеграционные тесты подсистем; `NODE_ENV=test` помечает пользователей и записи лога флагом `is_test` |
| Очистка тестовых данных | `node scripts/delete-user.js --test-users --yes` | удаление тестовых пользователей с каскадом |
| Конфигурация моделей | `config/default.yaml` (node-config) | модели и параметры меняются без правки кода |
| Просмотрщик логов в админке | `src/server/llm-log-data.js`, веб-интерфейс, `POST /api/llm-log/analyze` | ручной анализ запросов, включая AI-анализ записи |

Вывод: наблюдаемость (что уходит в LLM и что возвращается) уже решена на уровне хранения. Не хватает удобного
доступа к этим данным из рабочего процесса Claude Code и всей надстройки для прогонов, оценок и сравнений.

## 3. Этап 1 — доступ Claude Code к данным наблюдений

### 3.1. Скрипт экспорта лога LLM

Новый скрипт `scripts/llm-log-export.js`: выборка из `log.llm_request` в человекочитаемый Markdown или JSON,
чтобы Claude Code не писал SQL вручную и не таскал лишние данные в контекст.

```
node scripts/llm-log-export.js --last 20 --kind main_agent_answer
node scripts/llm-log-export.js --request-id <uuid>          # все вызовы одного цикла обработки
node scripts/llm-log-export.js --user <id> --since 2026-06-12 --format json
node scripts/llm-log-export.js --kind intent_classify --errors-only
```

Вывод по каждой записи: вид запроса, модель, длительность, токены, цена, статус, затем payload (сообщения,
ушедшие в LLM) и response. Флаги `--no-payload` / `--meta-only` для обзорных выборок без тяжёлого содержимого.

### 3.2. Привязка записей лога к версии кода и промпта

Добавить в записи лога (новые колонки или поле в `payload`/`binary_meta`):

- `git_commit` — короткий хеш HEAD на момент запуска процесса (берётся один раз при старте);
- `prompt_hash` — хеш текста системного промпта данного вида запроса (считается в месте вызова).

Без этого невозможно сказать, какой версией промпта порождена конкретная запись лога, и сравнение «до/после»
по историческим данным становится ненадёжным.

### 3.3. Документация для Claude Code

Раздел в `AGENTS.md` (или отдельный файл, подключаемый из него): как читать лог (скрипт экспорта, имена таблиц,
смысл `request_kind`), как запускать прогоны и оценки, где лежат эталонные наборы. Это «карта местности»,
с которой начинается каждая сессия тюнинга.

## 4. Этап 2 — неинтерактивный прогон сценариев

CLI-чат интерактивен (readline) — Claude Code не может им пользоваться в автоматическом цикле. Нужен
пакетный прогонщик.

### 4.1. Скрипт `scripts/run-scenario.js`

Вход — JSON-файл сценария: последовательность реплик пользователя, опционально домен, паузы (виртуальные),
команды планировщика (`tick`). Скрипт:

1. создаёт тестового пользователя (`NODE_ENV=test`, флаг `is_test` проставится автоматически);
2. последовательно прогоняет реплики через `handleMessage` — полный пайплайн: классификация, память,
   компрессия истории, инструменты, извлечение фактов и тем;
3. собирает транскрипт: реплика → ответ бота → `request_id` цикла → список LLM-вызовов цикла с токенами и ценой;
4. сохраняет результат в JSON (для машинной обработки) и Markdown (для чтения);
5. по флагу `--keep-user` оставляет пользователя для ручного дослеживания, иначе удаляет после прогона.

Формат сценария (эскиз):

```json
{
  "name": "facts-and-recall",
  "domain": "general",
  "turns": [
    { "user": "Меня зовут Олег, я готовлюсь к ЕГЭ по математике." },
    { "user": "Я плохо понимаю логарифмы." },
    { "scheduler_tick": true },
    { "user": "Что мне стоит подтянуть в первую очередь?" }
  ]
}
```

Сценарии хранить в `tests/scenarios/*.json` — рядом с существующими фикстурами.

### 4.2. Недетерминизм LLM

Ответы LLM недетерминированы, поэтому прогон не даёт побитовой воспроизводимости. Меры:

- для служебных видов запросов (классификатор, извлечение фактов/тем, компрессия) — `temperature: 0`
  в режиме оценки;
- для оценки основного ответа — повторные прогоны (`--repeat N`) и агрегирование оценок (медиана);
- сравнивать не тексты, а свойства ответа (см. этап 3).

## 5. Этап 3 — эталонные наборы и автоматическая оценка (evals)

### 5.1. Эталонные наборы по видам запросов

Расширить подход `tests/memory_cases.json` на остальные виды запросов. Для каждого `request_kind` со
структурированным выходом — свой набор кейсов «вход → ожидаемые свойства выхода»:

| Набор | Что проверяет | Тип проверки |
|-------|---------------|--------------|
| `classify_cases` | классификатор намерений: правильный intent, выбор скилла | детерминированная (точное сравнение полей JSON) |
| `fact_cases` (есть) | извлечение фактов: сохранять ли, scope, sensitivity | детерминированная |
| `topic_cases` | извлечение тем и вовлечённости | детерминированная с допуском |
| `compress_cases` | компрессия истории: сохранение ключевых фактов в сводке | проверка «обязательных упоминаний» + LLM-судья |
| `dialog_scenarios` | основной ответ: качество end-to-end диалога | LLM-судья по рубрике |

### 5.2. LLM-судья

Модуль `src/pipeline/eval/judge.js` (или `scripts/eval/judge.js`): оценка ответа бота по рубрике через уже
существующий `chatJSON` из `src/llm.js`. Рубрика — оценки 1–5 с обоснованием по осям:

- релевантность ответа реплике и контексту;
- использование памяти (упомянул ли бот факты, которые обязан был знать из сценария);
- соответствие персоне и тону бота;
- формат (длина, разметка канала, отсутствие служебного мусора);
- отсутствие галлюцинаций о пользователе.

Вызовы судьи логировать с отдельным `request_kind: 'eval_judge'`, чтобы они не смешивались с боевыми
запросами и их стоимость была видна отдельно.

### 5.3. Прогонщик оценок `scripts/eval.js`

```
node scripts/eval.js --suite classify            # один набор
node scripts/eval.js --suite all --repeat 3      # всё, с повторами
node scripts/eval.js --suite dialog --scenario facts-and-recall
```

Результат прогона — JSON-файл в `claudedocs/experiments/<дата>-<метка>/result.json`:

- по каждому кейсу: вход, фактический выход, вердикт (pass/fail или баллы судьи), токены, цена, длительность;
- агрегаты: доля пройденных, средние баллы по осям, суммарная цена прогона, перцентили задержки;
- метаданные: git-коммит, ветка, модели из конфига, хеши промптов.

## 6. Этап 4 — сравнение вариантов

Скрипт `scripts/eval-compare.js <runA> <runB>` — Markdown-отчёт «было/стало»:

- агрегатные дельты: качество, цена, токены, задержка;
- таблица кейсов, где вердикт изменился (улучшился/ухудшился), с диффом фактических выходов;
- итоговая рекомендация по порогам приёмки.

Пороги приёмки (первая версия, уточняются по опыту):

- доля пройденных детерминированных кейсов — не ниже базовой;
- средний балл судьи — не ниже базового минус 0.2 (шумовой допуск);
- цена цикла обработки и p95 задержки — не выше базовых более чем на оговорённый процент, если целью
  изменения не была именно экономия.

## 7. Этап 5 — регламент работы Claude Code

### 7.1. Рабочий цикл (закрепить в `AGENTS.md` или отдельном skill)

1. Зафиксировать базу: прогнать `eval.js --suite all` на текущем коде, сохранить как baseline.
2. Сформулировать гипотезу со ссылкой на наблюдения (конкретные записи `log.llm_request`).
3. Создать ветку `tuning/<тема>`; внести изменение. Если меняется промпт — обновить
   `docs/prompt-inventory.md` (он канонический).
4. Прогнать lint и существующие тесты (`npm run lint`, `npm test`) — это ворота до всяких оценок.
5. Прогнать оценки, построить сравнение с базой.
6. Сохранить отчёт эксперимента в `claudedocs/experiments/`, удалить тестовых пользователей
   (`node scripts/delete-user.js --test-users --yes`).
7. Показать человеку отчёт; слияние в master — только по явному решению человека.

### 7.2. Ограничители (guardrails)

- **Никакого автодеплоя.** Цикл заканчивается отчётом и веткой, не выкладкой.
- **Только тестовые пользователи.** Прогоны — строго `NODE_ENV=test`; запрещено писать в данные реальных
  пользователей и слать им сообщения.
- **Бюджет на прогон.** Лимит стоимости одного eval-прогона (например, через подсчёт цены в самом прогонщике
  с остановкой при превышении порога из конфига).
- **Один эксперимент — одно изменение.** Не смешивать правку промпта классификатора с правкой архитектуры
  памяти: иначе сравнение неинтерпретируемо.
- **Эталонные наборы меняются отдельно** от кода и промптов: правка кейсов — отдельный коммит с обоснованием,
  иначе можно «подогнать экзамен под ученика».

## 8. Этап 6 — развитие (после обкатки основного цикла)

- **Реплей реальных диалогов.** Скрипт, собирающий из `log.llm_request` реальные циклы (анонимизированные)
  в регрессионные сценарии — самые ценные кейсы приходят из боевых разговоров, особенно неудачных.
- **Регулярный прогон.** Ночной запуск `eval.js --suite all` (планировщик уже есть — `src/scheduler-run.js`,
  либо cron) с уведомлением при деградации: ловит дрейф моделей у провайдера и случайные регрессии.
- **Панель качества в админке.** Поверх результатов прогонов — график баллов/цены по версиям, рядом с уже
  существующей статистикой `llm-usage-stats`.
- **Вынос промптов из кода.** Сейчас промпты инлайн; после обкатки цикла можно рассмотреть вынос в файлы
  с версионированием — это упростит диффы промптов и `prompt_hash`. Отдельное решение, не условие старта.

## 9. Порядок реализации

| Шаг | Что | Зависимости |
|-----|-----|-------------|
| 1 | `scripts/llm-log-export.js` + раздел в `AGENTS.md` | нет |
| 2 | `git_commit`/`prompt_hash` в лог | нет |
| 3 | `scripts/run-scenario.js` + формат сценариев | нет |
| 4 | Эталонные наборы для классификатора и фактов (детерминированные) | шаг 3 |
| 5 | LLM-судья + диалоговые сценарии | шаги 3–4 |
| 6 | `scripts/eval.js` с сохранением результатов | шаги 4–5 |
| 7 | `scripts/eval-compare.js` + пороги приёмки | шаг 6 |
| 8 | Регламент и guardrails в `AGENTS.md`/skill | шаги 1–7 |
| 9 | Этап развития (реплей, регулярный прогон, панель) | после обкатки |

Шаги 1–4 дают уже работающий минимальный цикл: наблюдать, менять, прогонять, сравнивать детерминированные
проверки. Судья и отчёты сравнения превращают его в полноценную систему тюнинга.
