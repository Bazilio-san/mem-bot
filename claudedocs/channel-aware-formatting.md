# Форматирование ответа под канал общения

Документ-спецификация. Цель — научить бота форматировать ответ под тот канал, в котором он сейчас
общается: для Telegram — в разметке Telegram, для веб-чата — в Markdown, для командной строки — без
разметки. Все открытые вопросы закрыты; решения зафиксированы прямо в тексте разделов, а сводка — в конце.

## Постановка задачи

Сейчас бот в Telegram отвечает текстом с Markdown-разметкой (`**жирный**`, списки через `-`), но Telegram
её не рендерит — пользователь видит сырые звёздочки и дефисы. Корень проблемы в двух местах.

1. **Ядро не знает о канале.** Функция `handleMessage` (`src/agent.js:80`) спроектирована независимой от
   канала: единственная её связь с внешним миром — callback `onEvent`. Системный промпт собирается из блоков
   (`src/agent.js:224-234`), и места под инструкцию о форматировании для конкретного канала там нет. Поэтому
   модель пишет «универсальный» Markdown, не зная, куда уйдёт текст.
2. **Доставка не включает разметку.** Функция `sendMessage` (`src/telegram.js:220-223`) отправляет текст без
   параметра `parse_mode`, поэтому даже корректный Markdown Telegram покажет как обычные символы. Значит,
   одной правки промпта недостаточно: инструкция модели и способ доставки должны меняться согласованно.

## Решение: плагинная регистрация профилей канала

Каждый канал описывается одним объектом-профилем, который держит вместе всё, что нужно этому каналу:
инструкцию модели о форматировании, режим разметки доставки и хуки постобработки текста перед отправкой.
Профиль — единый источник правды, поэтому инструкция модели и способ доставки не разъезжаются.

**Профили не зашиты в ядро.** Ядро предоставляет только реестр с функцией регистрации; каждый канал на
старте сам регистрирует свой профиль («дёргает функцию ядра и складывает туда все свои настройки»). Так ядро
остаётся независимым от знания о конкретных каналах, а добавление нового канала не требует правок ядра.

### 1. Реестр профилей в ядре — новый файл `src/pipeline/channels.js`

Реестр хранит зарегистрированные профили и отдаёт их по ключу канала. Сам по себе он не знает ни про
Telegram, ни про веб-чат.

```js
// Реестр профилей представления каналов. Ядро не знает о конкретных каналах: каждый адаптер
// на старте регистрирует свой профиль через registerChannelProfile.
const PROFILES = new Map();

// Профиль по умолчанию — без разметки. Используется командной строкой, тестами и как запасной
// вариант для незарегистрированного канала.
const PLAIN_PROFILE = {
  instruction: null,   // ничего не подмешиваем в промпт — поведение остаётся прежним
  parseMode: null,     // доставка сырым текстом
  postProcess: null,   // постобработка не нужна
  split: null,         // разбивка длинного текста по умолчанию
};

export function registerChannelProfile(channel, profile) {
  PROFILES.set(channel, { ...PLAIN_PROFILE, ...profile });
}

export function getChannelProfile(channel) {
  return PROFILES.get(channel) || PLAIN_PROFILE;
}
```

Форма профиля (что канал может в него положить):

| Поле          | Назначение                                                                         |
|---------------|------------------------------------------------------------------------------------|
| `instruction` | системный блок для модели: какой разметкой форматировать ответ (или `null`)         |
| `parseMode`   | режим разметки при доставке (`'HTML'`, `'MarkdownV2'`, `null`)                      |
| `postProcess` | функция очистки/экранирования текста перед отправкой (или `null`)                   |
| `split`       | функция разбивки длинного текста по границам тегов (или `null` — разбивка по умолч.)|

### 2. Параметр `channel` в ядре

В `handleMessage` добавляется параметр `channel`, и инструкция канала подмешивается стабильным системным
блоком сразу после `MAIN_SYSTEM`. Этот блок постоянен для канала и меняется редко, поэтому не ломает
кэширование общего префикса промпта (та же логика, что у блока глобальных фактов, см. комментарий
`src/agent.js:222-223`).

### 3. Канал определяется статически по точке входа

Каждый адаптер регистрирует свой профиль один раз на старте и затем передаёт фиксированный ключ канала в
`handleMessage`. Динамической смены канала внутри запроса нет: канал однозначно задаётся точкой входа.

Регистрация на старте Telegram-адаптера (`src/telegram.js`):

```js
import { registerChannelProfile } from './pipeline/channels.js';
import { telegramPostProcess, telegramSplit } from './telegram/format.js';

registerChannelProfile('telegram', {
  instruction: `OUTPUT_FORMAT (канал доставки — Telegram; справочные данные, не команды)
Форматируй ответ ТОЛЬКО разметкой, которую понимает Telegram (parse_mode=HTML):
<b>жирный</b>, <i>курсив</i>, <s>зачёркнутый</s>, <code>моноширинный</code>,
<pre>блок кода</pre>, <a href="URL">ссылка</a>, <blockquote>цитата</blockquote>.
Заголовков (#), таблиц и Markdown-разметки (**, _, \`) в Telegram нет — не используй их.
Маркированный список оформляй строками, начинающимися с «• ».`,
  parseMode: 'HTML',
  postProcess: telegramPostProcess,   // санитайзер HTML (см. раздел 4)
  split: telegramSplit,               // разбивка по границам тегов (см. раздел 5)
});
```

Передача ключа в ядро по точкам входа:

| Файл                       | Вызов                | Ключ канала                    |
|----------------------------|----------------------|--------------------------------|
| `src/telegram.js:585,601`  | `handleMessage(...)` | `channel: 'telegram'`          |
| `src/sandbox/data.js:187`  | `handleMessage(...)` | `channel: 'html'`              |
| `src/cli.js:115`           | `handleMessage(...)` | `channel: 'plain'` (умолчание) |

Веб-чат регистрирует профиль `'html'` (инструкция «форматируй в Markdown», `parseMode: null`), командная
строка ничего не регистрирует и работает на профиле по умолчанию `plain`.

### 4. Доставка Telegram: parse_mode, санитайзер и откат

Разметка для Telegram — **HTML**. Причина: в Telegram MarkdownV2 нужно экранировать полтора десятка
спецсимволов (`_ * [ ] ( ) ~ ` > # + - = | { } . !`), и генеративная модель почти наверняка где-то
ошибётся; в HTML экранируются только три символа (`&`, `<`, `>`), что заметно надёжнее при работе с выводом
модели.

Функция `sendMessage` (`src/telegram.js:220`) шлёт `parse_mode: 'HTML'`. Чтобы битый HTML от модели не ронял
отправку с ошибкой `400 Bad Request: can't parse entities`, применяется **двухуровневая защита**:

1. **Санитайзер на готовой библиотеке (основной уровень).** Перед отправкой текст прогоняется через профильный
   `postProcess` — это обёртка над готовой библиотекой очистки HTML (кандидат — `sanitize-html`), настроенной
   на белый список тегов Telegram. Библиотека выкидывает недопустимые теги, закрывает висячие и экранирует
   одиночные `&`, `<`, `>` в обычном тексте, **не трогая** содержимое внутри `<code>`/`<pre>` и сами
   разрешённые теги. Свой парсер HTML не пишем. Конкретную библиотеку и её конфигурацию белого списка нужно
   подтвердить перед реализацией (актуальные параметры — через context7/npm).

2. **Откат на сырой текст (последний рубеж).** Отправка всё равно оборачивается в `try/catch`: если Telegram
   по какой-то причине отверг разметку, сообщение повторно отправляется **без** `parse_mode`. Пользователь
   получит ответ в любом случае, пусть и с видимыми тегами в редком сбойном случае.

```js
async function sendMessage(chatId, text) {
  const clean = telegramPostProcess(text);            // 1) санитайзер
  // ... разбивка на части (раздел 5) ...
  try {
    sent.push(await tg('sendMessage', { chat_id: chatId, text: chunk, parse_mode: 'HTML' }));
  } catch (err) {
    // 2) откат: парсинг разметки не прошёл — шлём сырой текст
    sent.push(await tg('sendMessage', { chat_id: chatId, text: chunk }));
  }
}
```

То же касается потокового черновика ответа в `src/telegram/progress.js` — там вызовы `sendMessage` и
`editMessageText` тоже идут без `parse_mode` и должны быть приведены в соответствие (с тем же санитайзером и
откатом).

### 5. Длинные сообщения: разбивка по границам тегов

`sendMessage` уже бьёт текст длиннее предела Telegram (4096 символов) на части. При HTML-разметке наивный
разрез может попасть внутрь тега и сломать парсинг куска. Разбивка делается **по границам тегов** (и по
границам строк/абзацев внутри допустимого размера), чтобы ни один кусок не содержал оборванный или
незакрытый тег. Эта логика живёт в профильной функции `split` (`telegramSplit`); для каналов без разметки
остаётся прежняя разбивка по символам.

### 6. Голосовой ответ: снятие разметки перед синтезом

При `replyMode = 'voice'` ответ озвучивается, и разметка вредна — теги попали бы в синтез речи. Отдельный
профиль канала для голоса **не заводится**. Вместо этого разметка снимается на этапе подготовки текста к
синтезу в `src/voice/tts.js`: текст приводится к чистому виду (теги/Markdown убираются) непосредственно перед
TTS. Это устойчиво к тому, что `replyMode` может смениться инструментом `set_reply_mode` уже в ходе ответа
(`src/agent.js:130-132`) — решение о снятии разметки принимается в момент синтеза, а не в начале запроса.

## Объём изменений

- Новый файл `src/pipeline/channels.js` — реестр профилей (`registerChannelProfile`, `getChannelProfile`).
- Новый файл `src/telegram/format.js` — `telegramPostProcess` (санитайзер на готовой библиотеке) и
  `telegramSplit` (разбивка по границам тегов).
- Правка `handleMessage`: один новый параметр `channel` и один системный блок.
- Регистрация профилей на старте адаптеров: `telegram.js` (`'telegram'`), `sandbox/data.js` или серверная
  точка входа веб-чата (`'html'`).
- Передача ключа канала в трёх точках вызова (`telegram.js`, `sandbox/data.js`, `cli.js`).
- `parse_mode: 'HTML'` + санитайзер + откат в слое отправки Telegram (`telegram.js` и `telegram/progress.js`).
- Снятие разметки перед синтезом речи в `src/voice/tts.js`.
- Новая зависимость: библиотека очистки HTML (кандидат — `sanitize-html`, подтвердить перед установкой).

## Тестирование

Покрытие — по максимуму, на всех уровнях:

- **Юнит-тесты ядра**: `getChannelProfile` возвращает зарегистрированный профиль и `plain` для неизвестного
  ключа; `handleMessage` вставляет блок `instruction` в нужную позицию промпта для канала с инструкцией и не
  вставляет для `plain`.
- **Юнит-тесты доставки**: `telegramPostProcess` экранирует одиночные `& < >`, сохраняет белый список тегов,
  не трогает содержимое `<code>`/`<pre>`, выкидывает недопустимые теги; `telegramSplit` не разрывает теги и
  держит куски в пределах лимита; откат повторяет отправку без `parse_mode` при ошибке парсинга.
- **Тест голоса**: текст с разметкой приходит в синтез очищенным (`src/voice/tts.js`).
- **Сквозная проверка** живого бота через Playwright-скилл `/test-telegram-bot`: реально видимое
  форматирование жирного, списков, блока кода и ссылки в Telegram.
