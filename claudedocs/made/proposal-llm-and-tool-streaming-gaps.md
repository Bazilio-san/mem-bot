# Зазоры реализации: стриминг ответа LLM и статусов инструментов

Исходный proposal: `claudedocs/draft/proposal-llm-and-tool-streaming.md`.

Статус на момент проверки: фича в основном реализована. Есть потоковый LLM-клиент, событийный контракт ядра,
Telegram progress helper, флаги конфигурации, документация и unit/smoke-тесты. Ниже перечислены обнаруженные
зазоры относительно требований proposal.

## 1. Дельты публикуются до понимания, что ход завершится tool calls

В `src/agent.js` потоковый вызов модели передаёт `onDelta` напрямую:

```js
onDelta: (chunk) => emit({ type: 'assistant.delta', text: chunk }),
```

Это значит, что если модель в одном assistant-turn сначала отдаст `delta.content`, а затем завершит ход
`tool_calls`, пользователь может увидеть промежуточный текст до статуса инструмента. Proposal прямо требовал
буферизовать такие дельты или не публиковать их в ходе, который закончился вызовом инструмента.

Текущий комментарий в коде опирается на вероятностное поведение модели: `content` при tool calls "почти всегда"
пустой. Это практичный компромисс, но не полное выполнение требования.

Ожидаемое закрытие: буферизовать дельты внутри model turn и публиковать их только если финальный `msg` не содержит
`tool_calls`; либо добавить в `chatStream` режим накопления без немедленного `onDelta`.

## 2. Нет fallback с chatStream на обычный chat при ранней транспортной ошибке

Proposal просил совместимый fallback: если `stream = false` или потоковый вызов падает на уровне транспорта до первой
дельты, агент может вызвать старый непотоковый `chat`.

В текущем `src/agent.js` при `streamingOn` вызывается только `chatStream`; если он бросит исключение, выполнение уходит
в общий `agent.failed`, а непотоковый `chat` не пробуется.

Ожидаемое закрытие: обернуть потоковый model turn так, чтобы ошибка до первой опубликованной дельты переключала ход на
`chat`. Если видимый стрим уже начался, текущее поведение с ошибкой допустимо.

## 3. Не прогнан полный набор проверок из proposal

Во время ревизии были прогнаны релевантные проверки:

```bash
npm run test:streaming
npm run test:progress
npm run test:progress-format
npm run test:channels
npm run check:streaming
```

Все они прошли. `npm run check:streaming` подтвердил, что прокси отдаёт и `delta.content`, и потоковые
`delta.tool_calls`.

Полный набор из proposal в этой ревизии не запускался:

```bash
npm test
npm run test:reactions
npm run test:voice
npm run check:llm
```

Ожидаемое закрытие: прогнать полный набор и зафиксировать результат либо внешнюю причину невозможности проверки.

## 4. Документация местами расходится с фактическими значениями по умолчанию

В `src/config.js` значение `TELEGRAM_STREAM_EDIT_INTERVAL_MS` по умолчанию равно `500`, а в
`docs/telegram/telegram-bot.md` описано значение `900 мс`.

Ожидаемое закрытие: привести документацию и конфиг к одному значению или явно описать выбранное значение как
фактическое.
