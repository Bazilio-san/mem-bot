# Инфраструктура интеллектуального тюнинга бота

Предложение: что нужно построить, чтобы Claude Code мог самостоятельно улучшать код, промпты и архитектуру
бота — с измеримой оценкой качества, стоимости и производительности каждого изменения.

Дата: 2026-06-13. Статус: предложение, не реализовано.

## 1. Идея замкнутого цикла тюнинга

Самостоятельная модификация безопасна и полезна только внутри замкнутого цикла, где каждое изменение измеряется:

1. **Наблюдение** — Claude Code видит, что реально уходит в LLM и что возвращается (полные payload/response).
2. **Гипотеза** — на основе наблюдений формулируется конкретное изменение (промпт, код, параметр модели).
3. **Изменение** — правка в отдельной git-ветке.
4. **Прогон** — эталонные сценарии прогоняются через полный пайплайн без Telegram.
5. **Оценка** — автоматические проверки плюс LLM-судья выставляют оценки; считаются токены, цена, задержка.
6. **Сравнение** — отчёт «было/стало» по тем же сценариям; решение принять или откатить.
7. **Фиксация** — изменение вливается человеком после просмотра отчёта; результаты архивируются как новая база
   сравнения.

Шаги 1 и частично 4 уже обеспечены существующей инфраструктурой. Основная работа — шаги 4–6 и регламент.

## 2. Что уже есть (фундамент)

| Компонент | Где | Что даёт для тюнинга |
|-----------|-----|----------------------|
| Полный лог LLM-запросов | `log.llm_request` (отдельная БД логов), запись через `src/pipeline/llm-log.js` | payload и response целиком (до 100 КБ), вид запроса (`request_kind`, 16 видов), модель, токены, цена, длительность, корреляция `request_id`, флаг `is_test` |
| Журнал событий агента | `log.agent_event`, `src/pipeline/agent-event-log.js` | трассировка шагов пайплайна внутри одного запроса |
| CLI-чат через весь пайплайн | `src/cli.js` (`npm run chat`) | вызывает `handleMessage` из `src/agent.js` — тот же путь, что и Telegram, но без мессенджера |
| Реестр промптов | `docs/prompt-inventory.md` | координаты всех промптов и tool-описаний в коде (промпты лежат инлайн в исходниках) |
| Зачаток эталонного набора | `tests/memory_cases.json` | кейсы «сохранять/не сохранять факт» с ожидаемыми scope и sensitivity |
| Тестовый каркас | `tests/run.js`, ~25 файлов тестов | интеграционные тесты подсистем; `NODE_ENV=test` помечает пользователей и записи лога флагом `is_test` |
| Очистка тестовых данных | `node scripts/delete-user.js --test-users --yes` | удаление тестовых пользователей с каскадом |
| Конфигурация моделей | `config/default.yaml` (node-config) | модели и параметры меняются без правки кода |
| Просмотрщик логов в админке | `src/server/llm-log-data.js`, веб-интерфейс, `POST /api/llm-log/analyze` | ручной анализ запросов, включая AI-анализ записи |

Вывод: наблюдаемость (что уходит в LLM и что возвращается) уже решена на уровне хранения. Не хватает удобного
доступа к этим данным из рабочего процесса Claude Code и всей надстройки для прогонов, оценок и сравнений.

## 2a. Сквозной принцип: экономия контекста Claude Code

Контекстное окно Claude Code — главный дефицитный ресурс цикла тюнинга: одна запись `log.llm_request` может
весить до 100 КБ, а прогон оценок порождает десятки таких записей. Поэтому вся инфраструктура строится по
правилу **послойного доступа**: сначала компактная сводка, затем — детали только явно выбранных записей.

Обязательные требования ко всем скриптам и форматам данных этого документа:

1. **Сводка по умолчанию.** Любой скрипт без флагов выводит только метаданные и агрегаты (таблица: id, вид,
   модель, токены, цена, статус). Полные payload/response никогда не выводятся «оптом» — только адресно.
2. **Адресный drill-down.** Полное содержимое — только по явному указанию конкретных записей (`--id`,
   `--case`), а не по широкому фильтру. Широкий фильтр + `--full` должен требовать подтверждения лимитом
   (`--limit N`, по умолчанию маленький).
3. **Выбор полей и срезов.** Флаги `--fields` (какие колонки/части записи выводить), `--max-chars N`
   (обрезка длинных значений с пометкой), `--grep <строка>` (фильтр по содержимому payload/response —
   сервером/скриптом, а не чтением всего в контекст).
4. **Раздельные файлы результатов.** Тяжёлые артефакты (полные транскрипты, payload кейсов) пишутся в
   отдельные файлы; в контекст Claude Code попадает только маленький сводный файл со ссылками на детали.
   Claude Code читает детальный файл лишь для кейсов, которые требуют разбора.
5. **Стабильные короткие идентификаторы.** Каждая запись/кейс имеет короткий id, по которому деталь можно
   запросить позже — не нужно держать содержимое в контексте «на всякий случай».
6. **Делегирование тяжёлого чтения.** Массовый разбор логов (десятки записей) выполняется сабагентом
   (Task/Explore), который возвращает только вывод; основной контекст получает заключение, а не сырые данные.
   Это закрепляется в регламенте (§7).

Эти требования ниже не повторяются в каждом разделе — они подразумеваются для всех скриптов этапов 1–4.

## 3. Этап 1 — доступ Claude Code к данным наблюдений

### 3.1. Скрипт экспорта лога LLM

Новый скрипт `scripts/llm-log-export.js`: выборка из `log.llm_request` в человекочитаемый Markdown или JSON,
чтобы Claude Code не писал SQL вручную и не таскал лишние данные в контекст. Работает по принципу §2a:
по умолчанию — только таблица метаданных, полное содержимое — адресно.

```
# Слой 1: обзор — компактная таблица (id, время, kind, модель, токены, цена, статус), без payload
node scripts/llm-log-export.js --last 20 --kind main_agent_answer
node scripts/llm-log-export.js --request-id <uuid>            # все вызовы одного цикла, тоже только сводка
node scripts/llm-log-export.js --kind intent_classify --errors-only --since 2026-06-12

# Слой 2: детали выбранных записей
node scripts/llm-log-export.js --id 18234 --fields payload.messages,response   # одна запись, нужные части
node scripts/llm-log-export.js --id 18234,18260 --max-chars 2000               # обрезка длинных значений

# Поиск по содержимому без вытягивания содержимого в контекст: вернёт только id и совпавший фрагмент
node scripts/llm-log-export.js --kind fact_extract --grep "паспорт" --last 200
```

`--fields` понимает пути внутри jsonb (`payload.messages`, `payload.tools`, `response.choices`), что позволяет,
например, посмотреть только системный промпт запроса, не загружая историю диалога и список инструментов.
Полный вывод (`--full`) разрешён только вместе с `--id` либо с явным `--limit`.

### 3.2. Привязка записей лога к версии кода и промпта

Добавить в записи лога (новые колонки или поле в `payload`/`binary_meta`):

- `git_commit` — короткий хеш HEAD на момент запуска процесса (берётся один раз при старте);
- `prompt_hash` — хеш текста системного промпта данного вида запроса (считается в месте вызова).

Без этого невозможно сказать, какой версией промпта порождена конкретная запись лога, и сравнение «до/после»
по историческим данным становится ненадёжным.

### 3.3. Документация для Claude Code

Раздел в `AGENTS.md` (или отдельный файл, подключаемый из него): как читать лог (скрипт экспорта, имена таблиц,
смысл `request_kind`), как запускать прогоны и оценки, где лежат эталонные наборы. Это «карта местности»,
с которой начинается каждая сессия тюнинга. Раздел намеренно короткий — шпаргалка команд плюс ссылка на этот
документ; он загружается в контекст каждую сессию, поэтому подробности здесь, а не в `AGENTS.md` (см. §7.3).

## 4. Этап 2 — неинтерактивный прогон сценариев

CLI-чат интерактивен (readline) — Claude Code не может им пользоваться в автоматическом цикле. Нужен
пакетный прогонщик.

### 4.1. Скрипт `scripts/run-scenario.js`

Вход — JSON-файл сценария: последовательность реплик пользователя, опционально домен, паузы (виртуальные),
команды планировщика (`tick`). Скрипт:

1. создаёт тестового пользователя (`NODE_ENV=test`, флаг `is_test` проставится автоматически);
2. последовательно прогоняет реплики через `handleMessage` — полный пайплайн: классификация, память,
   компрессия истории, инструменты, извлечение фактов и тем;
3. собирает транскрипт: реплика → ответ бота → `request_id` цикла → список LLM-вызовов цикла с токенами и ценой;
4. сохраняет результат двумя слоями (§2a): `transcript.summary.md` — короткая сводка (реплики, первые строки
   ответов, агрегаты токенов/цены, список `request_id`) и `transcript.full.json` — полный транскрипт.
   Claude Code читает сводку; полные ответы и LLM-вызовы достаёт адресно — из full-файла по номеру хода
   или через `llm-log-export.js --request-id`;
5. по флагу `--keep-user` оставляет пользователя для ручного дослеживания, иначе удаляет после прогона.

Формат сценария (эскиз):

```json
{
  "name": "facts-and-recall",
  "domain": "general",
  "turns": [
    { "user": "Меня зовут Олег, я готовлюсь к ЕГЭ по математике." },
    { "user": "Я плохо понимаю логарифмы." },
    { "scheduler_tick": true },
    { "user": "Что мне стоит подтянуть в первую очередь?" }
  ]
}
```

Сценарии хранить в `tests/scenarios/*.json` — рядом с существующими фикстурами.

### 4.2. Недетерминизм LLM

Ответы LLM недетерминированы, поэтому прогон не даёт побитовой воспроизводимости. Меры:

- для служебных видов запросов (классификатор, извлечение фактов/тем, компрессия) — `temperature: 0`
  в режиме оценки;
- для оценки основного ответа — повторные прогоны (`--repeat N`) и агрегирование оценок (медиана);
- сравнивать не тексты, а свойства ответа (см. этап 3).

## 5. Этап 3 — эталонные наборы и автоматическая оценка (evals)

### 5.1. Эталонные наборы по видам запросов

Расширить подход `tests/memory_cases.json` на остальные виды запросов. Для каждого `request_kind` со
структурированным выходом — свой набор кейсов «вход → ожидаемые свойства выхода»:

| Набор | Что проверяет | Тип проверки |
|-------|---------------|--------------|
| `classify_cases` | классификатор намерений: правильный intent, выбор скилла | детерминированная (точное сравнение полей JSON) |
| `fact_cases` (есть) | извлечение фактов: сохранять ли, scope, sensitivity | детерминированная |
| `topic_cases` | извлечение тем и вовлечённости | детерминированная с допуском |
| `compress_cases` | компрессия истории: сохранение ключевых фактов в сводке | проверка «обязательных упоминаний» + LLM-судья |
| `dialog_scenarios` | основной ответ: качество end-to-end диалога | LLM-судья по рубрике |

### 5.2. LLM-судья

Модуль `src/pipeline/eval/judge.js` (или `scripts/eval/judge.js`): оценка ответа бота по рубрике через уже
существующий `chatJSON` из `src/llm.js`. Рубрика — оценки 1–5 с обоснованием по осям:

- релевантность ответа реплике и контексту;
- использование памяти (упомянул ли бот факты, которые обязан был знать из сценария);
- соответствие персоне и тону бота;
- формат (длина, разметка канала, отсутствие служебного мусора);
- отсутствие галлюцинаций о пользователе.

Оси, шкалы и веса рубрики не зашиваются в код судьи — они берутся из файла критериев, а формулировки осей —
из файлов рубрик (см. §8). Вызовы судьи логировать с отдельным `request_kind: 'eval_judge'`, чтобы они
не смешивались с боевыми запросами и их стоимость была видна отдельно.

### 5.3. Прогонщик оценок `scripts/eval.js`

```
node scripts/eval.js --suite classify            # один набор
node scripts/eval.js --suite all --repeat 3      # всё, с повторами
node scripts/eval.js --suite dialog --scenario facts-and-recall
```

Результат прогона — каталог `claudedocs/experiments/<дата>-<метка>/`, разделённый по слоям (§2a):

```
summary.json      # маленький: агрегаты + список кейсов (id, вердикт, баллы, токены, цена) БЕЗ содержимого
cases/<id>.json   # детали одного кейса: вход, фактический выход, обоснование судьи, request_id
```

- `summary.json`: доля пройденных, средние баллы по осям, суммарная цена прогона, перцентили задержки,
  метаданные (git-коммит, ветка, модели из конфига, хеши промптов) и по строке на кейс;
- в stdout `eval.js` печатает только агрегаты и список провалившихся кейсов — это всё, что попадает
  в контекст Claude Code автоматически;
- детальные файлы `cases/<id>.json` Claude Code открывает только для кейсов, требующих разбора
  (провалы и ухудшения), а не подряд.

## 6. Этап 4 — сравнение вариантов

Скрипт `scripts/eval-compare.js <runA> <runB>` — Markdown-отчёт «было/стало». Отчёт строится от компактного
к детальному, чтобы Claude Code (и человек) останавливались на нужной глубине:

- сначала агрегатные дельты: качество, цена, токены, задержка — несколько строк;
- затем таблица только тех кейсов, где вердикт изменился (улучшился/ухудшился), по строке на кейс
  со ссылками на `cases/<id>.json` обоих прогонов; неизменившиеся кейсы в отчёт не попадают вовсе;
- диффы фактических выходов — не в основном отчёте, а в отдельном файле `diffs.md` (читается адресно);
- итоговая рекомендация по порогам приёмки.

Пороги приёмки не задаются в самом скрипте — он читает их из файла критериев `tests/eval/criteria.yaml`
(секция `acceptance`, см. §8): доля пройденных детерминированных кейсов не ниже базовой; средний балл судьи
не ниже базового минус шумовой допуск; цена цикла обработки и p95 задержки не выше базовых более чем на
оговорённый процент, если целью изменения не была именно экономия.

## 7. Этап 5 — регламент работы Claude Code

### 7.1. Рабочий цикл (краткая версия — в `AGENTS.md`, полная методика — в skill, см. §9)

1. Зафиксировать базу: прогнать `eval.js --suite all` на текущем коде, сохранить как baseline.
2. Сформулировать гипотезу со ссылкой на наблюдения (конкретные записи `log.llm_request`).
3. Создать ветку `tuning/<тема>`; внести изменение. Если меняется промпт — обновить
   `docs/prompt-inventory.md` (он канонический).
4. Прогнать lint и существующие тесты (`npm run lint`, `npm test`) — это ворота до всяких оценок.
5. Прогнать оценки, построить сравнение с базой.
6. Сохранить отчёт эксперимента в `claudedocs/experiments/`, удалить тестовых пользователей
   (`node scripts/delete-user.js --test-users --yes`).
7. Показать человеку отчёт; слияние в master — только по явному решению человека.

### 7.2. Ограничители (guardrails)

- **Никакого автодеплоя.** Цикл заканчивается отчётом и веткой, не выкладкой.
- **Только тестовые пользователи.** Прогоны — строго `NODE_ENV=test`; запрещено писать в данные реальных
  пользователей и слать им сообщения.
- **Бюджет на прогон.** Лимит стоимости одного eval-прогона (например, через подсчёт цены в самом прогонщике
  с остановкой при превышении порога из конфига).
- **Один эксперимент — одно изменение.** Не смешивать правку промпта классификатора с правкой архитектуры
  памяти: иначе сравнение неинтерпретируемо.
- **Эталонные наборы меняются отдельно** от кода и промптов: правка кейсов — отдельный коммит с обоснованием,
  иначе можно «подогнать экзамен под ученика».

### 7.3. Дисциплина экономии контекста (дополняет §2a)

- **Сначала сводка, потом адресные детали.** Запрещено вытягивать в контекст полные payload по широкому
  фильтру или читать `cases/*.json` подряд; сначала `summary.json` / таблица метаданных, затем 1–3 записи
  по id.
- **Массовый разбор — сабагенту.** Если нужно просмотреть больше ~5 полных записей лога или транскриптов,
  работа делегируется сабагенту (Explore/Task), который возвращает короткое заключение; сырые данные
  в основной контекст не попадают.
- **Поиск по содержимому — инструментом, не чтением.** Для вопросов вида «в каких запросах встречается X»
  использовать `--grep` скрипта экспорта или SQL, а не выгрузку записей с последующим просмотром.
- **Раздел в `AGENTS.md` — короткий.** Карта тюнинга в `AGENTS.md` (§3.3) — это несколько строк со ссылкой
  на настоящий документ и шпаргалка команд; подробности загружаются в контекст только при работе над тюнингом.

## 8. Критерии качества: в каком виде они существуют

Самоорганизующийся цикл работает только тогда, когда критерии качества отчуждены от исполнителя: Claude Code
не «решает сам», стало ли лучше, а сверяется с зафиксированными порогами. Поэтому критерии — это не текст
в документации, а **машиночитаемый файл, единый источник истины** для прогонщика оценок, судьи и отчёта
сравнения.

### 8.1. Файл критериев `tests/eval/criteria.yaml`

```yaml
version: 1                       # критерии версионируются; правка — отдельный коммит (§7.2)
suites:
  classify:                      # детерминированные наборы: порог доли пройденных кейсов
    type: deterministic
    pass_threshold: 0.95
  facts:
    type: deterministic
    pass_threshold: 0.90
  dialog:                        # наборы с LLM-судьёй: оси, шкалы, минимумы, веса
    type: judge
    axes:
      relevance:         { scale: [1, 5], min_avg: 4.0, weight: 0.30 }
      memory_usage:      { scale: [1, 5], min_avg: 4.0, weight: 0.25 }
      persona:           { scale: [1, 5], min_avg: 3.5, weight: 0.15 }
      format:            { scale: [1, 5], min_avg: 4.0, weight: 0.15 }
      no_hallucinations: { scale: [1, 5], min_avg: 4.5, weight: 0.15 }
acceptance:                      # пороги приёмки сравнения «было/стало» (§6)
  quality_drop_tolerance: 0.2    # шумовой допуск среднего балла судьи
  cost_increase_max_pct: 10      # рост цены цикла, если цель изменения — не экономия
  latency_p95_increase_max_pct: 15
budget:
  eval_run_max_usd: 2.0          # стоп-лимит стоимости одного eval-прогона (§7.2)
```

### 8.2. Требования к критериям

- **Один источник истины.** `eval.js`, судья и `eval-compare.js` читают пороги, оси и веса только отсюда;
  числа не дублируются ни в коде, ни в отчётах, ни в skill.
- **Три уровня критериев,** не смешивать: ожидания конкретного кейса (лежат в самом кейсе:
  `expect_*`-поля, обязательные упоминания), пороги набора (в `criteria.yaml`) и пороги приёмки
  эксперимента в целом (секция `acceptance`).
- **Текст рубрики судьи** (формулировки осей, что значит балл 1 и балл 5) хранится рядом —
  `tests/eval/rubrics/<ось>.md`; промпт судьи собирается из этих файлов, а не зашивается в код.
- **Изменение критериев — событие, а не побочный эффект.** Отдельный коммит с обоснованием; запрещено
  менять критерии в той же ветке, где меняется промпт или код (правило §7.2 о «подгонке экзамена»).
- **Эволюция через человека.** Claude Code может предлагать новые оси и пороги (по итогам разборов
  провалов), но правка `criteria.yaml` принимается только человеком.

## 9. Финальный шаг — skill цикла тюнинга

Последним шагом основного плана создаётся skill (например, `tune-bot`; файлы под `.claude/skills/` правятся
по протоколу `/edit-claude-files`). Skill — это «вся методика в одном месте»: Claude Code, получив задачу
«улучшить X», вызывает skill и дальше действует по его регламенту, не восстанавливая процесс по памяти
и не загружая в контекст этот документ целиком.

### 9.1. Что знает skill

- полный цикл итерации (§1 и §7.1): baseline → гипотеза → ветка → изменение → lint/тесты → eval →
  сравнение → отчёт → решение человека;
- шпаргалку команд: экспорт лога, прогон сценария, eval, сравнение, очистка тестовых пользователей;
- дисциплину экономии контекста (§2a, §7.3) как обязательные правила, а не рекомендации;
- где лежат критерии (§8) и как читать `summary.json` / отчёт сравнения;
- типовые сценарии итераций: «улучшить промпт одного request_kind», «снизить цену цикла»,
  «разобрать регрессию по жалобе пользователя».

### 9.2. Чекпоинты и сжатие контекста в процессе работы

Итерация тюнинга длинная (часы), и контекст Claude Code за это время забивается. Skill предписывает
**протокол чекпоинтов**, при котором всё нужное для продолжения живёт на диске, а не в контексте:

- файл состояния итерации `claudedocs/experiments/<метка>/state.md`: гипотеза, текущий шаг цикла,
  ссылка на baseline, краткие результаты прошедших шагов, следующий шаг. Обновляется на каждом чекпоинте;
- чекпоинты — границы шагов цикла: baseline зафиксирован; изменение внесено и закоммичено; eval прогнан;
  сравнение построено. После записи `state.md` весь сырой материал шага из контекста больше не нужен;
- после каждого чекпоинта контекст ужимается: компактификация сессии (/compact) или просто продолжение
  с опорой на файлы — `state.md` + `summary.json` достаточно, чтобы новая сессия продолжила работу
  с любого чекпоинта без потерь;
- skill начинается с инструкции «прочитай `state.md`, если он есть» — устойчивость к обрыву сессии
  получается автоматически;
- тяжёлые шаги (разбор десятков записей лога, чтение транскриптов) skill направляет в сабагентов,
  чтобы в основной сессии оставались только выводы (§7.3).

## 10. Развитие (после обкатки основного цикла)

Этап детально расписан в отдельном документе `claudedocs/self-tuning-development-stage.md` (отбор и
анонимизация реплеев, скользящая база ночных прогонов, таблицы `log.eval_run`/`log.eval_case`, панель,
порядок реализации). Здесь — краткое содержание:

- **Реплей реальных диалогов.** Скрипт, собирающий из `log.llm_request` реальные циклы (анонимизированные)
  в регрессионные сценарии — самые ценные кейсы приходят из боевых разговоров, особенно неудачных.
- **Регулярный прогон.** Ночной запуск `eval.js --suite all` (планировщик уже есть — `src/scheduler-run.js`,
  либо cron) с уведомлением при деградации: ловит дрейф моделей у провайдера и случайные регрессии.
- **Панель качества в админке.** Поверх результатов прогонов — график баллов/цены по версиям, рядом с уже
  существующей статистикой `llm-usage-stats`.
- **Вынос промптов из кода.** Сейчас промпты инлайн; после обкатки цикла можно рассмотреть вынос в файлы
  с версионированием — это упростит диффы промптов и `prompt_hash`. Отдельное решение, не условие старта.

## 11. Порядок реализации

| Шаг | Что | Зависимости |
|-----|-----|-------------|
| 1 | `scripts/llm-log-export.js` + раздел в `AGENTS.md` | нет |
| 2 | `git_commit`/`prompt_hash` в лог | нет |
| 3 | `scripts/run-scenario.js` + формат сценариев | нет |
| 4 | Эталонные наборы для классификатора и фактов (детерминированные) | шаг 3 |
| 5 | Файл критериев `tests/eval/criteria.yaml` + рубрики судьи (§8) | шаг 4 |
| 6 | LLM-судья (рубрики из §8) + диалоговые сценарии | шаги 3–5 |
| 7 | `scripts/eval.js` с сохранением результатов | шаги 5–6 |
| 8 | `scripts/eval-compare.js` (пороги приёмки из `criteria.yaml`) | шаг 7 |
| 9 | Регламент и guardrails в `AGENTS.md` | шаги 1–8 |
| 10 | **Финальный шаг:** skill `tune-bot` с протоколом чекпоинтов и сжатия контекста (§9) | шаги 1–9 |
| 11 | Этап развития (реплей, регулярный прогон, панель) | после обкатки |

Шаги 1–4 дают уже работающий минимальный цикл: наблюдать, менять, прогонять, сравнивать детерминированные
проверки. Критерии, судья и отчёты сравнения превращают его в измеримую систему, а skill (шаг 10) собирает
всё в самоорганизующийся механизм: методика, экономия контекста и устойчивость к обрыву сессии закреплены
в одном месте и не зависят от памяти конкретной сессии Claude Code.
