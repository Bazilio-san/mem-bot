# План создания Skill «разработка приложений на Telethon»

Документ описывает, как мы будем строить навык (Skill) для Claude Code, который помогает **писать
production-приложения на Python с использованием Telethon**. Это навык по разработке, а не инструкция
по работе с Telethon из командной строки и не обёртка над Claude Code CLI.

## 1. Цель и границы навыка

**Что навык должен давать.** Когда пользователь просит «сделай парсер каналов на Telethon», «напиши
авто-ответчик в Telegram», «собери userbot, который пересылает сообщения в LLM», Claude должен генерировать
не разрозненные фрагменты, а **цельное приложение** с правильной структурой каталогов, обработкой ошибок,
хранением сессий и конфигурации. Навык кодирует архитектуру и типовые сценарии, а не перечисляет методы.

**Главная идея — типовые приложения (app recipes).** Центр навыка — каталог готовых «рецептов» приложений
(см. раздел 6). Каждый рецепт описывает структуру, точки расширения и подводные камни конкретного класса
приложений. Это ключевая ставка плана: Claude должен сразу узнавать класс задачи и разворачивать под него
скелет, а детали Telethon-вызовов подтягивать из справочных файлов.

**Что строго в границах:**

- Только **Python**. Telethon — это Python-библиотека; TypeScript-аналог (GramJS) — отдельная история и в
  этот навык не входит (упомянём один раз в разделе ограничений как «не для этого навыка»).
- Только **разработка приложений**. Никаких инструкций «как через Claude Code дёргать Telethon в терминале».
- **MTProto-клиент**: и пользовательские аккаунты (userbot), и боты. Это принципиально отличает навык от уже
  имеющегося `telegram-bot-builder` (см. раздел 9 про разграничение).

**Что вне границ:** обход банов и анти-спам ограничений Telegram, накрутка, любые сценарии нарушения
условий использования Telegram. Массовые рассылки включаем только с явным предупреждением о рисках
блокировки аккаунта.

## 2. Версия Telethon: какую кодируем

Это первое и самое важное архитектурное решение, потому что API между версиями разный.

- **Стабильная ветка v1 — это `1.43.2` (апрель 2026), сейчас в режиме поддержки (maintenance).** На неё
  опирается подавляющее большинство существующего кода и документации. **Навык по умолчанию учит писать на
  v1.43.x.**
- **v2 — это `2.0.0a0` (альфа, октябрь 2025), нестабильна.** Полноценно рекомендовать её рано, но игнорировать
  нельзя: автор активно переписывает библиотеку. Поэтому в навык добавим **отдельный справочный файл с
  отличиями v1 → v2 и правилом выбора версии**, чтобы Claude не смешивал два несовместимых API.

Ключевые отличия, которые обязаны быть зафиксированы в навыке (иначе Claude будет путать версии):

| Аспект              | v1.43.x (по умолчанию)                              | v2.0.0a (только при явном запросе)                 |
|---------------------|-----------------------------------------------------|----------------------------------------------------|
| Класс клиента       | `TelegramClient(name, api_id, api_hash)`            | `Client(name, api_id, api_hash)`                   |
| Синхронный режим    | есть `telethon.sync`                                | удалён, только async                               |
| Вход                | `.start()`                                          | `await connect()` + `await interactive_login()`    |
| Итерация            | `client.iter_messages(...)`                         | единый метод (`await` и `async for`)               |
| Фильтры событий     | вложенные, могут быть «тяжёлыми»                     | независимые функции, только синхронные, без IO     |
| Свои сообщения      | по умолчанию не триггерят хендлеры                  | триггерят                                           |

**Источник истины по репозиторию:** канонический репозиторий теперь на Codeberg —
`https://codeberg.org/Lonami/Telethon`. GitHub-зеркало `LonamiWebs/Telethon` заархивировано в феврале 2026
(read-only, может быть удалено). В навыке все ссылки на исходники ведём на Codeberg, GitHub помечаем как
устаревший.

## 3. Источники и стратегия наполнения

Соотношение источников (как в исходном промпте, с уточнением):

1. **Официальная документация Telethon — ~70%.** База: `https://docs.telethon.dev/en/stable/`. Конкретные
   разделы, которые фетчим при сборке навыка:
   - `/basic/signing-in.html` — вход (user и bot).
   - `/basic/quick-start.html` — минимальный клиент.
   - `/quick-references/client-reference.html` — методы клиента.
   - `/modules/events.html` — события.
   - `/concepts/sessions.html` — сессии.
   - `/concepts/errors.html` — ошибки и FloodWait.
   - `/misc/compatibility-and-convenience.html` — асинхронность, нюансы.
   - Полный TL-справочник: `https://tl.telethon.dev/`.
2. **Исходники Telethon — ~20%.** Codeberg, каталоги `telethon/client/`, `telethon/events/`,
   `telethon/sessions/`, `telethon/network/`. Сверяем сигнатуры и поведение, которое плохо описано в доке.
3. **Реальные open-source проекты — ~10%.** Берём паттерны структуры (handlers/services/repositories),
   обработки FloodWait, хранения сессий. Используем как образцы архитектуры, а не копируем код.
4. **Наши правила архитектуры.** Раздел production-практик (раздел 5 ниже) — это наш слой поверх
   первоисточников.

**Правило точности при сборке:** перед написанием любого Telethon-кода в навыке фетчим соответствующую
страницу доки и сверяем имена методов и аргументов с конкретной версией (1.43.x). Не полагаемся на память —
API двигается.

## 4. Метаданные навыка (frontmatter)

```yaml
name: telethon-app-builder
description: >
  Build production Python applications with Telethon (MTProto) — userbots and bots: project structure,
  event handlers, sessions, config, FloodWait/error handling, and ready-made app recipes (auto-responder,
  channel scraper, group monitor, Telegram→LLM pipe, lead capture, voice processing, mass broadcast,
  client agents). Use when the user wants to write a Telegram app on Telethon, imports `telethon`, builds a
  userbot or MTProto client/bot in Python, scrapes channels, monitors groups, or automates a Telegram user
  account. Targets stable Telethon v1.43.x; notes v1→v2 differences.
license: MIT
```

Замечания по `description`:

- Перечисляем имена типовых приложений и триггер-фразы — так навык будет надёжно срабатывать.
- Явно указываем `telethon` (имя импорта) и «userbot / MTProto» — это отделяет его от `telegram-bot-builder`
  (который про Bot API).
- Имя `telethon-app-builder` подчёркивает «разработка приложений», а не «работа с CLI».

## 5. Структура навыка (progressive disclosure)

`SKILL.md` держим компактным (ориентир — до ~150–200 строк): что это, как выбрать версию, навигация по
рецептам и справочникам, критичные production-правила. Детали выносим в `references/`, рабочие скелеты — в
`templates/`.

```
telethon-app-builder/
├── SKILL.md                       # точка входа: выбор версии, навигация, критичные правила
├── references/
│   ├── architecture.md            # каноническая раскладка app/ (handlers/services/repositories/clients/models)
│   ├── client-and-auth.md         # TelegramClient, вход user vs bot, api_id/api_hash, my.telegram.org
│   ├── sessions.md                # SQLiteSession / StringSession / MemorySession, где хранить, безопасность
│   ├── events.md                  # NewMessage, CallbackQuery, ChatAction, Album, InlineQuery, ... + фильтры
│   ├── messages-and-media.md      # send_message/send_file, iter_messages, скачивание, альбомы, голосовые
│   ├── entities-and-chats.md      # get_entity, каналы/группы/пользователи, права, вступление, ID и кэш
│   ├── errors-and-flood.md        # FloodWaitError(.seconds), SessionPasswordNeeded, retry, throttling
│   ├── production-practices.md    # .env, секреты, логирование, graceful shutdown, разделение слоёв, деплой
│   └── v1-to-v2.md                # таблица отличий и правило выбора версии (раздел 2 этого плана в деталях)
├── templates/                     # типовые приложения — «рецепты» (раздел 6)
│   ├── _scaffold/                 # общий каркас приложения (config, client factory, main.py, .env.example)
│   ├── auto_responder/
│   ├── channel_scraper/
│   ├── group_monitor/
│   ├── telegram_llm_pipe/
│   ├── lead_capture/
│   ├── voice_processing/
│   ├── mass_broadcast/            # с предупреждением о рисках
│   └── client_agent/
└── examples/
    └── minimal_userbot.py         # самый короткий рабочий клиент для smoke-проверки
```

Принцип навигации: `SKILL.md` → выбираешь рецепт в `templates/` → по ходу подтягиваешь нужные `references/`.

## 6. Каталог типовых приложений (ядро навыка)

Для каждого рецепта в `templates/<name>/` кладём: краткий `README.md` (когда применять, схема, точки
расширения, риски), скелет кода поверх общего каркаса `_scaffold/`, и `.env.example`. Список рецептов:

1. **Авто-ответчик (`auto_responder`)** — реагирует на входящие по правилам/ключевым словам, шлёт ответы,
   умеет «не отвечать самому себе» и работать в рабочие часы. База: `events.NewMessage`.
2. **Парсер каналов (`channel_scraper`)** — выгружает историю и/или слушает новые посты каналов, нормализует
   и складывает в хранилище. База: `iter_messages`, `events.NewMessage(chats=...)`. Акцент на FloodWait и
   пагинацию.
3. **Мониторинг групп (`group_monitor`)** — следит за активностью групп (вступления/выходы, ключевые слова,
   упоминания), шлёт алёрты. База: `events.NewMessage`, `events.ChatAction`.
4. **Telegram → LLM → Telegram (`telegram_llm_pipe`)** — принимает сообщение, прогоняет через LLM, отвечает.
   **Перекрёстная ссылка на навык `building-pydantic-ai-agents`** для слоя агента. Telethon тут — транспорт.
5. **Сбор лидов (`lead_capture`)** — диалоговый сбор данных (`client.conversation` в v1), валидация,
   сохранение в repository (CRM/БД). Показывает паттерн «диалог как сервис».
6. **Обработка голосовых (`voice_processing`)** — скачивает голосовые/аудио, отправляет на распознавание.
   **Перекрёстная ссылка на навык `assemblyai`** для транскрипции. Telethon — приём и скачивание медиа.
7. **Массовые рассылки (`mass_broadcast`)** — рассылка по списку с обязательным троттлингом и обработкой
   FloodWait. **Явное предупреждение о рисках блокировки аккаунта и нарушении ToS**; по умолчанию
   консервативные задержки.
8. **Клиентские Telegram-агенты / userbot (`client_agent`)** — постоянно работающий userbot с набором
   команд и фоновыми задачами (`run_until_disconnected` + background worker). Базовый «комбайн», на котором
   видно разделение слоёв.

CRM-бот из исходного промпта покрывается комбинацией `lead_capture` + `client_agent` — отдельный рецепт не
плодим, а в README `lead_capture` дадим врезку «как дорастить до CRM».

## 7. Блок production-практик (наш слой поверх доки)

То, что Claude систематически забывает и что обязано быть в `references/production-practices.md` и продублировано
короткими правилами в `SKILL.md`:

- **Секреты не в коде.** `api_id`, `api_hash`, токен бота — только из `.env` (через `python-dotenv` или
  `pydantic-settings`). В репозиторий кладём `.env.example`.
- **Сессии отдельно и под защитой.** Каталог `sessions/`, файл `*.session` в `.gitignore`. Объяснить разницу
  SQLite-сессии (файл на диске) и `StringSession` (для контейнеров/CI, хранить в секрете). Сессия = доступ к
  аккаунту, утечка критична.
- **FloodWait — обязательный retry.** Ловим `FloodWaitError`, спим `e.seconds`, при необходимости настраиваем
  `client.flood_sleep_threshold`. Никаких «голых» циклов без троттлинга.
- **Разделение слоёв:** Telethon-клиент ↔ бизнес-логика ↔ хранилище. Хендлеры тонкие, логика в `services/`,
  доступ к данным в `repositories/`.
- **Асинхронность по-настоящему:** не блокировать event loop, не мешать `telethon.sync` с настоящим async,
  тяжёлые/блокирующие вызовы — в executor.
- **Graceful shutdown и логирование:** корректное закрытие клиента, структурные логи в `logs/`, перехват
  сигналов.
- **Идемпотентность и состояние:** где хранить «последний обработанный message id», чтобы парсер/монитор
  переживал перезапуск.

## 8. Блок архитектуры

Каноническая раскладка приложения (из промпта, оформляем в `references/architecture.md` и как стартовый
каркас `templates/_scaffold/`):

```
telegram_app/
├── app/
│   ├── handlers/        # тонкие обработчики событий Telethon
│   ├── services/        # бизнес-логика
│   ├── repositories/    # доступ к данным (БД, файлы, внешние API)
│   ├── clients/         # фабрика TelegramClient + внешние клиенты (LLM, STT)
│   └── models/          # модели данных (dataclass/pydantic)
├── sessions/            # *.session (в .gitignore)
├── logs/
├── config/              # загрузка настроек из .env
└── main.py              # сборка зависимостей, запуск клиента
```

В `references/architecture.md` объясняем, как каждый рецепт из раздела 6 ложится на эту структуру (что попадает
в handlers, что в services), чтобы Claude масштабировал один и тот же скелет.

## 9. Разграничение с уже установленными навыками

В проекте уже есть `telegram-bot-builder`, `assemblyai`, `building-pydantic-ai-agents`. Чтобы навыки не
конфликтовали и правильно срабатывали, в `SKILL.md` добавим раздел «Когда какой навык»:

- **`telegram-bot-builder`** — Telegram **Bot API** (HTTP, токен бота, вебхуки, inline-клавиатуры). Берём,
  когда нужен классический бот без аккаунта-пользователя.
- **`telethon-app-builder` (этот)** — **MTProto**: userbot и приложения от имени **пользовательского
  аккаунта** (парсинг каналов, мониторинг, автоматизация аккаунта), а также боты, где нужен MTProto.
- **`assemblyai`** — транскрипция; вызывается из рецепта `voice_processing`.
- **`building-pydantic-ai-agents`** — слой LLM-агента; вызывается из рецепта `telegram_llm_pipe`.

Эти перекрёстные ссылки прописываем в соответствующих рецептах.

## 10. Решения (подтверждены)

1. **Имя и расположение — ✅ решено.** Навык называется `telethon-app-builder`, устанавливается сразу в
   `.claude/skills/telethon-app-builder/` (запись через обход блокировки `.claude/**`, как с предыдущими
   навыками).
2. **Версия по умолчанию — ✅ решено.** Основная — v1.43.x; отличия v1→v2 выносим в отдельный справочник
   `references/v1-to-v2.md`. Сразу на v2 не таргетим.
3. **Глубина шаблонов — ✅ решено.** Рецепты делаем как **аннотированные образцы**: структура каталогов плюс
   ключевые куски кода с пояснениями (не полноценные запускаемые проекты). Дешевле в сборке; пользователь
   доводит под свою задачу.
4. **Хранилище в рецептах — предложение (требует финального согласования при сборке).** Абстрактный интерфейс
   `repositories/` + реализация по умолчанию на SQLite (stdlib). Поскольку шаблоны — аннотированные образцы,
   репозиторий показываем как интерфейс с короткой SQLite-вставкой, а не полную реализацию.
5. **Только Python — ✅ решено.** Строго Python; GramJS (TypeScript-аналог) — вне навыка, упоминается один
   раз в ограничениях.

## 11. Порядок сборки навыка

1. Зафиксировать решения из раздела 10 (короткий опрос пользователя).
2. Фетчнуть и законспектировать страницы доки v1.43.x (раздел 3) → наполнить `references/*`.
3. Сверить сигнатуры по исходникам на Codeberg для спорных мест (события, сессии, FloodWait).
4. Собрать общий каркас `templates/_scaffold/` и `references/architecture.md`.
5. Реализовать рецепты раздела 6 поверх каркаса (по одному, начиная с `auto_responder` и `channel_scraper`).
6. Написать `references/production-practices.md`, `errors-and-flood.md`, `v1-to-v2.md`.
7. Написать компактный `SKILL.md` с навигацией, выбором версии и разделом «Когда какой навык».
8. Smoke-проверка: `examples/minimal_userbot.py` компилируется; шаблоны проходят `py_compile`; описания
   рецептов согласованы со справочниками.
9. Установить навык в `.claude/skills/` и убедиться, что он виден в списке навыков.

## 12. Риски и открытые вопросы

- **Безопасность сессий.** Файл сессии = полный доступ к аккаунту. Это сквозная тема всех рецептов, не только
  отдельного справочника.
- **Пересечение с `telegram-bot-builder`.** Без чёткого раздела «когда какой навык» оба могут срабатывать на
  «сделай телеграм-бота». Раздел 9 обязателен.
