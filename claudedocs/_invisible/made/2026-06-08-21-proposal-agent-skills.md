# Предложение: домены и память через Agent Skills

## 1. Цель

Нужно уйти от модели, где домен является отдельной жёсткой сущностью с заранее заданным набором признаков. Такая модель
быстро начинает ограничивать: классификация знает только короткое описание домена, извлечение фактов опирается только
на схему `data`, а основной ответ не получает доменно-специфичной инструкции. Добавление новой области превращается в
несколько разрозненных изменений: запись в `mem.agent_domains`, схема в `mem.domain_schemas`, подсказки в коде,
инструменты и документация.

Предлагаемая модель: **skill становится источником домена**. В файловом реестре есть каталог `skills/<skill_name>/`,
внутри которого `SKILL.md` описывает:

- как этот skill называется и когда он применим;
- какой доменный namespace он создаёт для памяти;
- как роутеру распознать этот домен;
- какие правила добавить в prompt основного ответа;
- какие правила добавить в prompt извлечения фактов;
- какая схема структурированной доменной памяти действует для этого skill;
- какие инструменты и справочники доступны только при активном skill.

Домен в рантайме перестаёт быть ручным набором признаков в отдельном реестре. Он становится **проекцией skill**:
`domain_key` нужен как стабильный ключ для хранения и выборки памяти, но описание домена, классификационные признаки,
prompt-добавки и схемы живут рядом в `SKILL.md`.

### 1.1. Целевая модель: один механизм

Принципиальное требование: в финале остаётся **один механизм поведения — skills**. Важно различать два разных смысла
слова «домен», потому что судьба у них разная:

- **`domain_key` / `domain_id` как технический ключ хранения.** Это внешний ключ во множестве таблиц памяти
  (`memory_items`, `conversations`, `secure_records`, `scheduled_tasks`, `topic_mentions`, `global_facts`,
  `global_knowledge`). На нём держится адресация памяти по областям. Его убирать нельзя — это просто namespace
  («к какой области относится запись»), а не источник поведения. Остаётся навсегда.
- **Домен как редакционная сущность.** Описание области в `mem.agent_domains`, схема как runtime-источник в
  `mem.domain_schemas`, классификация через `listDomains()`. Это и есть второй механизм, который дублирует skills.
  Он убирается целиком — переезжает в `SKILL.md`.

То есть «один механизм на skills» означает: `domain_key` остаётся как чистый ключ хранения, а всё поведение домена
(классификационные признаки, prompt ответа, prompt извлечения, схема, инструменты, справочники) живёт в skill.
Двухпутёвость (флаг `SKILLS_ENABLED` с fallback, `mem.domain_schemas` как источник, `mem.agent_domains` как реестр)
существует только как **временные леса миграции** и сносится на этапе 4 (см. раздел 14).

## 2. Текущее состояние проекта

По коду и документации сейчас работает такая цепочка:

- `src/pipeline/classify.js` читает список доменов из `mem.agent_domains` через `listDomains()` и просит модель вернуть
  `domain_key`, `intent`, `entities`, `needs_memory`, `needed_memory_scopes`, `needs_tools`, `candidate_tools`.
- `src/agent.js` вычисляет `effectiveDomain = intent.domain_key || domainKey`, по нему выбирает память, темы диалога,
  глобальные факты, conversation и контур извлечения.
- `src/pipeline/retrieve.js` достаёт `scope = 'domain'` только для текущего `domain_id`.
- `src/pipeline/extract.js` подмешивает в prompt извлечения подсказку из активной схемы домена, а затем вторым проходом
  заполняет `entity_key` и `data` строго по схеме сущности.
- `src/schema/registry.js` хранит активные версии схем в `mem.domain_schemas`; файлы `schemas/*.json` сейчас являются
  черновиками для ревью, а не runtime-источником.
- `src/pipeline/tools.js` подключает инструменты по глобальным флагам и правам, но не по домену.

Вывод: инфраструктура памяти уже доменно-ориентирована, но домен бедный: у него есть ключ, описание и schema registry,
а нет полноценного контракта применения. Skill должен стать таким контрактом.

## 3. Базовый принцип

На старте нужен простой файловый реестр:

```text
skills/
  <skill_name>/
    SKILL.md
    references/
      ...
```

Процесс при запуске читает `skills/*/SKILL.md`, валидирует метаданные и строит in-memory registry. В дальнейшем этот
реестр можно перенести в БД или сделать пополняемым онлайн, но первый вариант должен быть файловым: он проще для
разработки, ревью, git-истории и локального добавления новых навыков.

`SKILL.md` делится на чёткие блоки. Важное требование: runtime не должен «угадывать» смысл markdown произвольным
парсингом. Блоки должны иметь стабильные заголовки или YAML-поля, которые код может извлечь детерминированно.

## 4. Структура `SKILL.md`

Рекомендуемый формат: YAML frontmatter для машинных полей и markdown-разделы для prompt-блоков. Frontmatter содержит
короткие данные для реестра и роутера, тело содержит инструкции, которые подмешиваются только при активном skill.

~~~markdown
---
name: flight-search
domain_key: flight_search
title: Поиск авиабилетов
description: Поиск авиабилетов, аэропортов, маршрутов, дат вылета, пересадок и сравнение цен.
enabled: true
classification:
  when_to_use: >
    Пользователь спрашивает про авиабилеты, перелёты, аэропорты, даты поездки, маршруты,
    пересадки, цены на рейсы, багаж или варианты добраться самолётом.
  positive_signals:
    - билет
    - авиабилет
    - рейс
    - аэропорт
    - вылет
    - пересадка
    - багаж
  negative_signals:
    - обычное путешествие без вопроса о перелёте
    - метафорическое "улетел" или "рейс" не про авиацию
memory:
  scopes: [profile, domain, dialog]
  schema: domain-schema
tools:
  allowed: [search_flights, find_airports, get_booking_options]
  base: true
model:
  main: null
  extract: null
references:
  allowed: true
---

# Skill Prompt

Ты помогаешь пользователю с поиском и сравнением авиаперелётов. Минимизируй уточняющие вопросы:
если город, дата или предпочтение уже есть в памяти, используй их как гипотезу и явно покажи допущение.

## Fact Extraction Prompt

Извлекай только устойчивые предпочтения и параметры, которые пригодятся в будущих поисках:
домашний аэропорт, предпочитаемые города вылета, багажные ограничения, нежелательные пересадки,
любимые авиакомпании, бюджетные ограничения и даты долгосрочных поездок.

Не сохраняй одноразовые цены конкретного поиска, если пользователь не просит запомнить бюджет или маршрут.

## Domain Schema

```json
{
  "domain_key": "flight_search",
  "title": "Поиск авиабилетов",
  "entities": []
}
```

## References

- `references/airlines.md`: особенности авиакомпаний и багажа. Читать только при вопросе о перевозчике или багаже.
~~~

### 4.1. Обязательные поля

- `name` — ключ skill в файловом реестре. Рекомендуется kebab-case, совпадает с именем каталога.
- `domain_key` — стабильный ключ доменного namespace для памяти, схем, тем и глобальных фактов.
- `title` — человекочитаемое имя.
- `description` — короткое описание для UI, админки и fallback-классификации.
- `classification.when_to_use` — главный текст для роутера. Это не список строгих признаков, а смысловое правило.
- `# Skill Prompt` — блок, который добавляется в основной LLM-запрос при активном skill.
- `## Fact Extraction Prompt` — блок, который добавляется в prompt механизма извлечения фактов.

### 4.2. Необязательные поля

- `classification.positive_signals` и `classification.negative_signals` помогают роутеру, но не являются строгим
  allowlist/denylist. Они должны быть подсказками, а не исчерпывающим описанием домена.
- `tools.allowed` ограничивает предметные инструменты, доступные при активном skill.
- `tools.base` включает базовые системные инструменты: память, режим ответа, планировщик, глобальные факты.
- `memory.scopes` подсказывает роутеру и retrieval, какие разделы памяти обычно нужны.
- `memory.schema` указывает, откуда брать схему: из блока `## Domain Schema`, отдельного файла или БД.
- `model.main` и `model.extract` позволяют skill переопределить модели.
- `references.allowed` включает инструмент чтения файлов `references/**` для активного skill.

## 5. Семантика домена

В новой модели домен — не отдельный «тип способности», а **namespace памяти, принадлежащий skill**. Система сохраняет
`domain_key`, потому что он уже используется внешними ключами и запросами:

- `mem.memory_items.domain_id`;
- `mem.conversations.domain_id`;
- `mem.secure_records.domain_id`;
- `mem.scheduled_tasks.domain_id`;
- `mem.topic_mentions.domain_id`;
- `mem.global_facts.domain_id` и `mem.global_knowledge.domain_id`.

Но содержательный источник домена переносится в skill. Таблица `mem.agent_domains` перестаёт быть редакционным
реестром и в целевой модели остаётся только **тонким техническим справочником соответствия `domain_key` → `domain_id`**.
Он нужен, потому что внешние ключи ссылаются на `domain_id` (целое число), а skill знает только `domain_key` (строку):
кто-то должен хранить это соответствие и выдавать стабильный числовой идентификатор. При загрузке skill registry система
заполняет справочник автоматически — убеждается, что для каждого `domain_key` есть строка в `mem.agent_domains`, и
создаёт её при отсутствии. Руками таблица больше не редактируется. Поля-описания (`description` и прочие признаки)
превращаются в мёртвый груз: их можно либо удалить миграцией, либо оставить и игнорировать.

Правило связи:

- один skill создаёт один `domain_key`;
- несколько skills могут ссылаться на один `domain_key`, только если они намеренно делят память и схему;
- если skill не указывает `domain_key`, загрузка должна падать, кроме специально зарезервированного `general`;
- `general` тоже должен быть skill, а не неявным fallback из кода.

Такой подход убирает жёсткий набор признаков в домене: признаки классификации, prompt ответа, prompt извлечения и schema
становятся частью одного skill-контракта.

## 6. Runtime-реестр skills

Новый модуль, например `src/pipeline/skills/registry.js`, отвечает за загрузку и доступ к skill:

- `loadSkills()` читает `config.skills.dir`, находит `*/SKILL.md`, парсит frontmatter и markdown-блоки.
- `listSkillRoutes()` возвращает компактный список для роутера: `name`, `domain_key`, `title`, `description`,
  `classification.when_to_use`, `positive_signals`, `negative_signals`.
- `getSkill(name)` возвращает полное описание skill.
- `getSkillByDomain(domainKey)` возвращает активный skill для доменного ключа.
- `getSkillPrompt(name)` возвращает содержимое `# Skill Prompt`.
- `getFactExtractionPrompt(name)` возвращает содержимое `## Fact Extraction Prompt`.
- `getDomainSchema(name)` возвращает JSON из `## Domain Schema` или связанного файла.
- `getReference(name, relPath)` читает только `references/**` внутри каталога skill и запрещает выход через `..`.

Реестр должен валидировать:

- уникальность `name`;
- наличие `domain_key`;
- уникальность пары `name`/`domain_key` или явно разрешённый shared-domain режим;
- наличие `classification.when_to_use`;
- наличие `# Skill Prompt`;
- корректность JSON в `## Domain Schema`, если блок есть;
- существование всех `tools.allowed` в реестре инструментов.

Кэш можно держать в памяти процесса. Для первого этапа достаточно перечитывать реестр при старте. Позже можно добавить
TTL, admin-команду reload или таблицу версий.

## 7. Роутинг

Роутер должен выбирать skill, а не домен напрямую. Новая схема результата классификации:

```json
{
  "intent": "string",
  "skill_name": "flight-search",
  "domain_key": "flight_search",
  "confidence": 0.82,
  "reason": "Пользователь просит найти перелёт и сравнить даты",
  "entities": {},
  "needs_memory": true,
  "needed_memory_scopes": ["profile", "domain", "dialog"],
  "needs_tools": true,
  "candidate_tools": ["search_flights"]
}
```

`domain_key` в ответе классификатора можно оставить как совместимое поле, но источником истины должен быть
`skill_name`. После классификации `domain_key` выводится из выбранного skill и проверяется кодом. Если модель вернула
несоответствующую пару `skill_name`/`domain_key`, код должен доверять registry, а не модели.

Prompt роутера получает `listSkillRoutes()`:

```text
Доступные skills:
- flight-search / domain flight_search
  Назначение: Поиск авиабилетов, аэропортов, маршрутов, дат вылета, пересадок и сравнение цен.
  Когда использовать: пользователь спрашивает про авиабилеты, перелёты, аэропорты...
  Положительные сигналы: билет; авиабилет; рейс; аэропорт...
  Отрицательные сигналы: обычное путешествие без вопроса о перелёте...
```

Важное поведение:

- если ни один skill не подходит, выбирается `general`;
- текущий skill диалога передаётся в prompt, чтобы уменьшить случайные переключения;
- переключение на другой skill происходит только при достаточной уверенности;
- `positive_signals` не являются обязательными: роутер может выбрать skill по смыслу запроса;
- `negative_signals` помогают снять ложные срабатывания, но не должны блокировать очевидный запрос.

## 8. Основной prompt ответа

После выбора skill `src/agent.js` добавляет отдельный system-блок:

```text
ACTIVE_SKILL_CONTEXT (справочные инструкции активного skill; текущий запрос важнее)

Skill: flight-search
Domain: flight_search

<содержимое # Skill Prompt>
```

Этот блок должен стоять после стабильного `MAIN_SYSTEM` и channel/global blocks, но до истории диалога и текущей
реплики. Он не заменяет общие правила безопасности, памяти, инструментов и приоритета текущего запроса.

Если у skill указаны `model.main` или `model.extract`, они переопределяют модели только для этого хода. Если поля
пустые, используются глобальные `config.llm.mainModel` и `config.llm.extractModel`.

## 9. Извлечение фактов

`src/pipeline/extract.js` должен получать не только `domainKey`, но и `skillName` или `skill`:

```js
extractCandidates({ skillName, domainKey, recentMessages, assistantResponse })
```

Первый проход извлечения получает два доменно-специфичных источника:

- подсказку схемы доменной памяти, как сейчас через `buildSchemaHint(domainKey)`;
- блок `## Fact Extraction Prompt` из активного skill.

Итоговый system prompt должен выглядеть концептуально так:

```text
<общие правила извлечения памяти>

ACTIVE_SKILL_FACT_EXTRACTION (дополнение активного skill)
<содержимое ## Fact Extraction Prompt>

DOMAIN_SCHEMA_HINT
<список entity_type и полей data>
```

Смысл этого блока: skill объясняет, **какие факты полезны именно в этом домене**, а schema объясняет, **в какой форме
они должны быть записаны**. Это разные уровни. Например, skill может сказать «не сохраняй одноразовые цены конкретного
поиска», а schema задаёт `preferred_airport`, `baggage_preference`, `budget_constraint`.

Второй проход остаётся строгим: если кандидат имеет `entity_type`, то `data` и `entity_key` заполняются по закрытой
схеме сущности. Если у skill нет схемы, предметные факты с `entity_type` не должны сохраняться как полуструктурированный
мусор; их нужно либо перевести в свободный факт без `entity_type`, либо отклонить на merge-этапе.

## 10. Схема доменной памяти

В целевой модели схема — это часть skill, а не запись в БД. Файл `domain-schema.json` лежит рядом со `SKILL.md` и
является **единственным источником истины** для структуры доменной памяти:

```text
skills/
  math-tutor/
    SKILL.md
    domain-schema.json
```

В `SKILL.md`:

```yaml
memory:
  schema: domain-schema.json
```

Маленькую схему можно встроить прямо в блок `## Domain Schema`. Runtime при старте читает схему из skill, валидирует её
тем же мета-валидатором, который сейчас используется в `src/schema/meta.js`, и держит активные схемы в in-memory
registry. Извлечение и merge запрашивают схему из этого реестра по `domain_key`, а не из таблицы. Таблица
`mem.domain_schemas` как runtime-источник в целевой модели **не нужна** — её можно дропнуть или оставить пустой legacy.

> **Зачем `domain-schema.json` вообще существует.** Это закрытая JSON-схема полей `data` и правило канонизации
> `entity_key` для каждой сущности домена (подробно слой описан в
> [11-per-domain-schema.md](../../../docs/ai-bot-with-memory/11-per-domain-schema.md)). Она превращает поле `data jsonb`
> из «мешка произвольных ключей» в проверяемый контракт и делает `entity_key` стабильным идентификатором: извлечение
> заполняет ровно объявленные поля, дедупликация по `(entity_type, entity_key)` надёжна, а выборка фильтрует прямо
> внутри `data` через оператор `@>`. Раньше этот контракт хранился в `mem.domain_schemas`; теперь он переезжает рядом
> со skill, потому что схема — такая же часть поведения домена, как prompt и список инструментов.

Переходный режим (только на время миграции) описан в разделе 14: пока `extract.js`/`merge.js` ещё читают схему через
`loadDomainDefinition(domainKey)` из `mem.domain_schemas`, команда `skills sync` синхронизирует туда схему из skill.
Это временные леса, а не постоянная двухпутёвость: финальная цель — читать схему напрямую из skill registry.

## 11. Инструменты и справочники

`buildToolDefs(ctx)` должен учитывать активный skill:

- базовые системные инструменты доступны всегда, если разрешены глобальными флагами и правами;
- предметные инструменты доступны только если они перечислены в `tools.allowed` активного skill;
- если `tools.allowed` отсутствует, для первого этапа можно оставить старое поведение, но лучше логировать warning;
- `candidate_tools` классификатора не должен сам включать инструмент, которого нет в `tools.allowed`.

Нужен инструмент `skill_read_reference`:

- доступен только при активном skill;
- принимает относительный путь внутри `references/**`;
- запрещает абсолютные пути и `..`;
- возвращает содержимое файла с лимитом размера;
- не читает справочники других skills.

Это даёт прогрессивное раскрытие: роутер видит короткое описание, основной prompt видит компактный `# Skill Prompt`,
а тяжёлые материалы читаются только по явной необходимости.

## 12. Хранение сведений в разрезе доменных skills

Для памяти и знаний нужно придерживаться одного правила: **все доменные сведения адресуются `domain_key`, который
принадлежит активному skill**.

Что хранится по `domain_key`:

- пользовательская доменная память (`memory_items.scope = 'domain'`);
- secure records, если они относятся к предметной области;
- темы диалога в companion/proactive контуре;
- глобальные факты и глобальная база знаний, релевантные только этому домену.

Схема (`domain-schema.json`) формально привязана к `domain_key`, но **хранится и редактируется как часть skill**, а не
в БД. В целевой модели `mem.domain_schemas` как источник схемы не участвует.

Что хранится по `skill_name`:

- prompt-блоки;
- правила классификации;
- список разрешённых инструментов;
- ссылки на справочники;
- настройки моделей;
- версия файлового skill.

Если два skills разделяют один `domain_key`, они видят общую доменную память. Это допустимо только при явной настройке,
иначе можно случайно смешать факты разных предметных областей.

## 13. Предлагаемые изменения в коде

### 13.1. Конфигурация

Добавить раздел:

```js
skills: {
  enabled: flag(env.SKILLS_ENABLED, false),
  dir: env.SKILLS_DIR || 'skills',
  syncSchemasOnStartup: flag(env.SKILLS_SYNC_SCHEMAS_ON_STARTUP, false),
  switchThreshold: Number(env.SKILLS_SWITCH_THRESHOLD || 0.65),
  referenceMaxBytes: Number(env.SKILL_REFERENCE_MAX_BYTES || 50000),
}
```

Флаг `SKILLS_ENABLED` — **миграционный, а не постоянная развилка**. Пока он `false`, поведение остаётся прежним
доменным; на время этапов 1–3 он позволяет включать skill-путь и сравнивать с прежним. На этапе 4 флаг и весь
fallback-путь выпиливаются, остаётся единственный путь — skills. То же относится к `syncSchemasOnStartup`: это
временный мост к `mem.domain_schemas`, который снимается вместе с таблицей-источником.

### 13.2. Registry

Добавить `src/pipeline/skills/registry.js` и тесты на:

- успешную загрузку минимального skill;
- ошибку при отсутствии `domain_key`;
- ошибку при невалидной schema;
- запрет выхода из `references/**`;
- корректное извлечение `# Skill Prompt` и `## Fact Extraction Prompt`.

### 13.3. Классификатор

`classifyIntent()` строит prompt из `listSkillRoutes()`, возвращает `skill_name` и вычисляет `domain_key` через
registry. Чтение списка доменов из `mem.agent_domains` через `listDomains()` остаётся только как fallback на время
миграции и удаляется на этапе 4 вместе с флагом.

### 13.4. Agent

`handleMessage()` должен:

- передать в классификатор текущий `skillName` или вывести его из текущего `domainKey`;
- получить `activeSkill`;
- установить `ctx.skillName` и `ctx.domainKey`;
- добавить `ACTIVE_SKILL_CONTEXT`;
- вызвать `buildToolDefs(ctx)` с учётом skill;
- вызвать `extractCandidates({ skillName, domainKey, ... })`.

### 13.5. Extract

`extractCandidates()` должен подмешивать `getFactExtractionPrompt(skillName)`. В целевой модели схемный второй проход
берёт закрытую схему сущности из skill registry (`getDomainSchema(skillName)`). На время миграции допустимо оставить
старый путь через `loadDomainDefinition(domainKey)`, пока `skills sync` синхронизирует schema в `mem.domain_schemas`;
на этапе 4 этот путь заменяется чтением из registry.

### 13.6. Tools

`buildToolDefs(ctx)` должен фильтровать предметные инструменты по `activeSkill.tools.allowed`. Добавить
`skill_read_reference` в базовый набор только при наличии активного skill и `references.allowed = true`.

### 13.7. Schema CLI

Текущий `src/schema/cli.js` можно оставить, но добавить второй путь:

- `skills validate` — проверить все `SKILL.md`;
- `skills sync` — записать домены и схемы из skills в `mem.agent_domains` и `mem.domain_schemas`;
- `skills list` — показать активные skills, их `domain_key`, инструменты и источник schema.

## 14. Миграция

Этап 1: каркас за флагом.

- Добавить registry и конфиг.
- Создать `skills/general/SKILL.md`.
- Создать один пилотный предметный skill, лучше `flight-search`.
- Роутить по skills только при `SKILLS_ENABLED=true`.
- Не менять существующие таблицы, кроме автосоздания строк в `mem.agent_domains`.

Этап 2: prompt-блоки и инструменты.

- Подмешивать `# Skill Prompt` в основной ответ.
- Подмешивать `## Fact Extraction Prompt` в извлечение.
- Ограничить предметные инструменты через `tools.allowed`.
- Добавить `skill_read_reference`.

Этап 3: схемы рядом со skills.

- Перенести содержимое `schemas/*.json` в соответствующие `skills/*/domain-schema.json` или `## Domain Schema`.
- Добавить `skills sync`, который пишет активные версии в `mem.domain_schemas`.
- Зафиксировать, что `schemas/*.json` больше не источник новых доменов, а legacy-черновики.

Этап 4: снести леса, оставить один механизм.

- Удалить флаг `SKILLS_ENABLED` и весь fallback-путь: остаётся единственный путь — skills.
- Перевести чтение схемы на skill registry (`getDomainSchema`); прекратить использование `mem.domain_schemas` как
  источника. Таблицу можно дропнуть или оставить пустой legacy. Снять `syncSchemasOnStartup` и команду `skills sync`.
- Удалить `listDomains()` из классификатора.
- `mem.agent_domains` оставить только как тонкий справочник `domain_key` → `domain_id`, заполняемый из skills
  автоматически. Поля-описания удалить миграцией или игнорировать; руками таблица не редактируется.
- Документацию `domains-explained.md` обновить: домен становится namespace памяти, определяемым skill.

Спорный момент: можно вообще отказаться от таблицы-мэппинга и генерировать `domain_id` детерминированно из
`domain_key`. Но это убирает контроль целостности через внешние ключи, поэтому тонкую таблицу `mem.agent_domains`
лучше сохранить.

## 15. Критерии готовности

Функциональность можно считать реализованной, когда:

- добавление каталога `skills/<name>/SKILL.md` делает skill видимым роутеру после запуска или reload;
- роутер выбирает `skill_name` по `classification.when_to_use`, а не по жёсткому списку признаков;
- выбранный skill задаёт `domain_key`, по которому читается и пишется доменная память;
- основной ответ получает `# Skill Prompt` только активного skill;
- извлечение фактов получает `## Fact Extraction Prompt` только активного skill;
- schema skill валидирует и структурирует доменные факты;
- предметные инструменты не доступны вне skills, где они явно разрешены;
- справочники skill читаются только через `skill_read_reference` и только из каталога активного skill;
- схема читается напрямую из skill registry, а не из `mem.domain_schemas`;
- от старых доменов остаётся только технический слой: `domain_id` как ключ хранения и тонкий справочник
  `domain_key` → `domain_id`; редакционного реестра, runtime-источника схем и доменного роутинга больше нет.

На время миграции (этапы 1–3) добавляется временный критерий: при выключенном `SKILLS_ENABLED` старое доменное
поведение сохраняется. После этапа 4 этот критерий снимается вместе с флагом.

## 16. Главный архитектурный результат

После изменения добавление новой предметной области должно выглядеть как добавление одного пакета:

```text
skills/
  nutrition-coach/
    SKILL.md
    domain-schema.json
    references/
      products.md
```

В этом пакете находится всё, что раньше было размазано по доменам, схемам и prompt-правкам: классификация, доменный
namespace памяти, prompt ответа, prompt извлечения, schema, tools и справочники. Это сохраняет существующий слой памяти
и `domain_id`, но убирает ограничение строгого доменного реестра как главного источника поведения.
