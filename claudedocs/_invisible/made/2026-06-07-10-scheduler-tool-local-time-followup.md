# Follow-up: возвращать локальное время из `scheduler_create_task`

## Проблема

Сейчас `scheduler_create_task` возвращает `next_run_at` как `Date` из `pg`. В tool result для LLM это сериализуется в
ISO 8601 UTC, например `2026-06-08T06:00:00.000Z`. Для cron/RRULE это технически правильно, но неудобно для ответа
пользователю: локальное намерение могло быть «каждый будний день в 09:00 Europe/Moscow», а модель видит только `06:00Z`.

LLM может помнить аргументы tool call, но в результате инструмента нет явных `timezone`, локального времени и исходного
расписания. Из-за этого ответ пользователю может быть менее точным или выглядеть как UTC-время вместо локального.

## Что сделать

Расширить результат `scheduler_create_task`, не ломая существующее поле `next_run_at`:

- оставить `next_run_at` как ISO UTC для машинного использования;
- добавить `timezone`, сохранённый в задаче после нормализации;
- добавить `next_run_at_local` как человекочитаемое локальное время в timezone задачи;
- добавить `schedule_kind`, `cron_expr`, `rrule`, если они полезны LLM для подтверждения регулярного расписания.

Пример желаемого результата:

```json
{
  "task_id": "…",
  "title": "Будний отчёт",
  "schedule_kind": "cron",
  "timezone": "Europe/Moscow",
  "cron_expr": "0 9 * * 1-5",
  "rrule": null,
  "next_run_at": "2026-06-08T06:00:00.000Z",
  "next_run_at_local": "2026-06-08 09:00:00 Europe/Moscow"
}
```

## Улучшить `MEMORY_CONTEXT` для запланированных задач

Сейчас при запросе scope `reminder` LLM видит активные задачи в блоке `Активные напоминания и задачи`, но формат слишком
бедный:

```text
Активные напоминания и задачи:
- Будний отчёт (срок: 2026-06-08T06:00:00.000Z)
```

Такой блок не показывает timezone, локальное время, тип расписания, cron/RRULE и регулярность. Для пользователя это
особенно плохо после добавления полноценного cron/RRULE: LLM видит только ближайший UTC-момент и не может уверенно
ответить, что задача стоит «каждый будний день в 09:00 по Москве».

Нужно расширить выборку и форматирование активных задач в `retrieveMemory()`:

- выбирать `instruction`, `schedule_kind`, `timezone`, `cron_expr`, `rrule`, `next_run_at`;
- показывать `next_run_at` в UTC и локальном виде;
- для `cron` и `rrule` добавлять исходное выражение, чтобы LLM понимала регулярность;
- держать формат компактным, потому что блок попадает в prompt.

Пример желаемого блока:

```text
Активные напоминания и задачи:
- Будний отчёт — report; следующее: 2026-06-08 09:00:00 Europe/Moscow
  (UTC: 2026-06-08T06:00:00.000Z; cron: 0 9 * * 1-5)
```

## Где менять

- `src/pipeline/agent-tools/scheduler-create-task.js`: расширить объект, который возвращает handler.
- При необходимости вынести форматирование локального времени в маленькую helper-функцию рядом с scheduler/tool кодом.
- `src/pipeline/retrieve.js`: расширить выборку и формат блока `Активные напоминания и задачи`.
- `tests/run.js`: обновить проверку 9e, чтобы она проверяла `timezone` и `next_run_at_local` в результате tool.
- `tests/run.js`: добавить проверку, что `MEMORY_CONTEXT` содержит локальное время и cron/RRULE для активной задачи.
- `docs/ai-bot-with-memory/10-operations.md`: зафиксировать, что tool result содержит UTC и локальное представление.

## Критерии готовности

- LLM получает и UTC, и локальное время следующего запуска.
- Пользовательский ответ может подтверждать расписание в локальном времени без самостоятельной конвертации из UTC.
- При вопросе о запланированных задачах LLM видит не только ближайший UTC-дедлайн, но и timezone, локальное время,
  тип расписания и cron/RRULE-выражение.
- Старые клиенты, которые читают `next_run_at`, не ломаются.
