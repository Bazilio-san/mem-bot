# План: генерация картинок по запросу пользователя с отправкой в Telegram

## Цель

Пользователь пишет боту фразу вида «сгенерируй картинку кота-космонавта на Марсе», бот обращается к внешнему
API генерации изображений, получает готовую картинку и присылает её в чат Telegram как фотографию (с подписью —
кратким текстом ответа модели).

## Как работает внешний API (проверено реальным запросом)

Конечная точка (адрес запроса): `POST https://ml.bazilio.ru/img-gen/api`, тело в формате JSON.

Запрос:

```json
{
  "prompt": "a red cube on white background",
  "negative_prompt": "blurry, low quality",
  "width": 1024,
  "height": 1024,
  "model": "auto",
  "seed": -1
}
```

Поля запроса:
- `prompt` — описание желаемой картинки (обязательное).
- `negative_prompt` — что нужно исключить (можно пустую строку).
- `width`, `height` — размеры в пикселях. На тесте работали 512 и 1024.
- `model` — имя модели или `"auto"` (сервер сам выберет; в ответе вернулась реальная модель `flux`).
- `seed` — зерно генератора. Значение `-1` означает случайное зерно; в ответе приходит фактически
  использованное число, что позволяет повторить ту же картинку.

Ответ — JSON, приходит синхронно и быстро (картинка уже готова к моменту ответа):

```json
{
  "imageUrl": "https://ml.bazilio.ru/img-gen/img/1781190425_93858d4272fbd460.png",
  "model": "flux",
  "seed": 3073944919
}
```

Важные факты, установленные при проверке:
- `imageUrl` — публичный https-адрес готового файла PNG (на тесте около 100 КБ, тип `image/png`,
  отдаётся nginx с заголовками кэширования). Дополнительной авторизации для скачивания не требуется.
- Поскольку адрес публичный и https, Telegram умеет скачать его сам: в методе `sendPhoto` можно передать
  не файл, а строку с URL. Это самый простой путь доставки — промежуточная загрузка файла на нашей стороне
  не нужна.

## Как это ложится на текущую архитектуру

Конвейер агента (`src/agent.js`) канал-независим: он формирует объект результата (`result`) с полями `answer`,
`toolsUsed`, `replyMode` и прочими, а конкретный канал доставки решает, как это показать пользователю. Telegram-адаптер
(`src/telegram/bot.js`) в функции `deliverAgentResult` уже умеет доставлять текст, голос (`sendVoice`) и кнопки
мини-приложений по содержимому `result.toolsUsed` (функция `sendWidgetButtons`). Картинку встроим по тому же образцу.

Решение состоит из двух частей:

1. **Инструмент агента `generate_image`** — вызывает API, возвращает модели и каналу данные о созданной картинке.
   Живёт в `src/pipeline/agent-tools/` рядом с остальными инструментами, регистрируется в `index.js`. Инструмент сам
   ничего в Telegram не отправляет (он работает и в командной строке, и на сервере — то есть остаётся канал-независимым).
2. **Доставка в Telegram** — `bot.js` после ответа агента смотрит в `result.toolsUsed`, находит результаты
   инструмента `generate_image` и отправляет каждую картинку методом `sendPhoto` по полученному `imageUrl`.

Так инструмент остаётся универсальным, а знание о Telegram (метод `sendPhoto`, ограничения подписи) остаётся в адаптере.

## Изменения по файлам

### 1. Новый инструмент: `src/pipeline/agent-tools/image/generate_image.js`

Экспортирует объект инструмента по образцу существующих (например, `scheduler/scheduler_create_task.js`):

- `name: 'generate_image'`
- `title: 'Генерирую картинку...'` — показывается пользователю как статус выполнения.
- `isEnabled: (ctx, config) => config.imageGen.enabled` — инструмент доступен только при включённой настройке.
- `definition` — описание функции для модели. Параметры:
  - `prompt` (строка, обязателен) — в описании просим модель переводить запрос на английский, если модель
    генерации лучше работает с английским (уточнить по факту качества), и обогащать описание деталями стиля,
    освещения, качества.
  - `negative_prompt` (строка либо null) — что исключить.
  - `width`, `height` (целые либо null) — со значениями по умолчанию из конфигурации, с ограничением допустимых
    размеров (например, 512 или 1024), чтобы модель не запросила слишком большое изображение.
- `handler(ctx, args)`:
  - читает базовый адрес и таймаут из `config.imageGen`;
  - выполняет `fetch` методом POST с телом JSON, оборачивает в таймаут (превышение времени ожидания) через
    `AbortController` — генерация может затянуться, нельзя зависать бесконечно;
  - при ошибке сети, ненулевом коде ответа или отсутствии `imageUrl` возвращает `{ error: '...' }` —
    конвейер уже умеет показывать такие ошибки и не падает;
  - при успехе возвращает структуру, из которой канал доставки достанет картинку. Предлагаемая форма
    (по аналогии с виджетами, которые лежат в `structuredContent.widget`):

    ```js
    return {
      // короткое поле для модели — она увидит его в истории и сможет осмысленно ответить текстом
      ok: true,
      model: data.model,
      seed: data.seed,
      // артефакт для канала доставки
      structuredContent: {
        image: {
          url: data.imageUrl,
          prompt: args.prompt,
          model: data.model,
          seed: data.seed,
        },
      },
    };
    ```

  Поле `structuredContent.image` — это сигнал каналу: «есть картинка для отправки». В `result.toolsUsed`
  оно сохраняется автоматически (см. `agent.js`, массив `toolsUsed`).

### 2. Регистрация инструмента: `src/pipeline/agent-tools/index.js`

Добавить импорт `imageGenerateTool` и включить его в массив `allTools`. Так инструмент попадает в реестр
(`src/pipeline/tools.js`) и становится базовым (доступным при любом активном навыке).

### 3. Доставка картинки: `src/telegram/bot.js`

- Добавить функцию `sendPhoto(chatId, imageUrl, caption)` по образцу `sendVoice`. Простейший вариант — JSON-запрос
  через существующий helper `tg('sendPhoto', { chat_id, photo: imageUrl, caption, parse_mode: 'HTML' })`, потому что
  Telegram скачает картинку по публичному URL сам. Подпись (`caption`) ограничена 1024 символами — длинный текст
  обрезать или отправить отдельным текстовым сообщением.
- Добавить функцию `sendGeneratedImages(chatId, result)` по образцу `sendWidgetButtons`: пройти по
  `result.toolsUsed`, отобрать `t.result?.structuredContent?.image`, для каждого вызвать `sendPhoto`.
  Зарегистрировать ссылки на отправленные сообщения через `saveSentRefs(..., 'image')`.
- Вызвать `sendGeneratedImages` внутри `deliverAgentResult` — после отправки основного ответа (текст или голос).
  Подпись к первой картинке — это `result.answer` (краткий комментарий модели); если картинок несколько,
  у остальных подписи нет. Уточнить порядок: вероятно, лучше отправить картинку, а текст-комментарий уже как подпись,
  чтобы не задваивать сообщения.
- Показать индикатор `sendChatAction` с действием `upload_photo`, пока идёт генерация и загрузка (по образцу
  `sendVoiceAction`), чтобы ожидание выглядело осмысленным.

### 4. Конфигурация: `config/default.yaml` и `src/config.js`

В `config/default.yaml` добавить блок (по образцу `voiceOutput`), с поясняющими комментариями на русском:

```yaml
#> Генерация изображений по запросу пользователя через внешний API.
imageGen:
  #> Включить инструмент генерации картинок.
  enabled: true
  #> Базовый адрес API генерации.
  apiUrl: 'https://ml.bazilio.ru/img-gen/api'
  #> Модель по умолчанию ('auto' — сервер выбирает сам).
  model: 'auto'
  #> Размеры картинки по умолчанию (пиксели).
  width: 1024
  height: 1024
  #> Максимальное время ожидания ответа API в миллисекундах.
  timeoutMs: 120000
```

При необходимости добавить в `src/config.js` валидацию (например, проверку допустимых размеров), по образцу
проверок блока `voiceOutput`.

## Краевые случаи и риски

- **Ожидание генерации.** На тесте ответ пришёл быстро, но это не гарантия. Обязателен таймаут (превышение времени
  ожидания) на стороне инструмента и индикатор `upload_photo` в чате, иначе пользователь решит, что бот завис.
- **Ошибки API.** Сеть, код ответа не 200, пустой `imageUrl` — инструмент возвращает `{ error }`; модель сообщит
  пользователю об ошибке текстом. Бот не должен падать.
- **Длина подписи.** Подпись к фотографии в Telegram ограничена 1024 символами (текстовое сообщение — 4096).
  Длинный ответ модели обрезаем для подписи либо шлём отдельным сообщением.
- **Содержимое запросов.** В тестовом файле `tests/http/1.ignore.http` есть запрос с откровенным содержимым (NSFW).
  Если бот публичный, стоит решить вопрос модерации запросов отдельно — это вне рамок текущей задачи, но риск
  нужно зафиксировать. Само API ограничений, судя по тестовым данным, не накладывает.
- **Несколько картинок за ход.** Модель теоретически может вызвать инструмент несколько раз. Доставка перебирает
  все результаты в `toolsUsed`, так что несколько картинок отправятся корректно.
- **Другие каналы.** Командная строка (`src/cli.js`) и веб-сервер не умеют показывать фото — они просто покажут
  текстовый ответ модели и проигнорируют `structuredContent.image`. Это допустимо: инструмент остаётся
  канал-независимым, доставку реализует только Telegram.

## Тестирование

- **Ручная проверка через Playwright** — навык `/test-telegram-bot`: перезапустить бота, отправить «сгенерируй
  картинку рыжего кота», убедиться, что в чат пришла фотография с осмысленной подписью.
- **Юнит-проверка инструмента** — отдельный тест (по образцу файлов в `tests/`) с заглушкой `fetch`,
  проверяющий: корректное тело запроса, разбор успешного ответа (поле `structuredContent.image`),
  поведение при ошибке (возврат `{ error }`) и срабатывание таймаута.
- **Проверка доставки** — тест функции `sendGeneratedImages`: что из `toolsUsed` извлекаются картинки
  и для каждой вызывается `sendPhoto`.

## Порядок реализации (шаги)

1. Добавить блок `imageGen` в `config/default.yaml` (+ валидация в `src/config.js` при необходимости).
2. Создать инструмент `src/pipeline/agent-tools/image/generate_image.js` с вызовом API и таймаутом.
3. Зарегистрировать инструмент в `src/pipeline/agent-tools/index.js`.
4. В `src/telegram/bot.js` добавить `sendPhoto`, `sendGeneratedImages` и индикатор `upload_photo`,
   подключить вызов в `deliverAgentResult`.
5. Написать юнит-тесты инструмента и доставки.
6. Прогнать ручной сценарий через Playgright-навык `/test-telegram-bot`.
7. Отразить изменения в документации (`docs/ai-bot-with-memory/` и `docs/telegram/telegram-bot.md`)
   согласно правилам из соответствующих `00-documentation-principles.md`.
