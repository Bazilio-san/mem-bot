# Proposal: буст по сущностям в retrieveMemory (реанимация поля entities классификатора)

Статус: предложение, не реализовано. Дата: 2026-06-12.

## Предыстория

Поле `entities` в схеме классификатора (`src/pipeline/classify.js:31`) — осиротевший хвост от сущностной модели
памяти первой версии (коммит `7202ca0`). Тогда таблица памяти имела колонки `entity_type`/`entity_key`, извлечённые
классификатором сущности уходили в `retrieveMemory` как `entityKeys`, и совпадение `entity_key` давало фактy надбавку
`(entity_match ? 1 : 0) * 0.07` в формуле скоринга, а также служило ключом дедупликации фактов. В коммите `609bfd7`
(«flat user_facts memory store») сущностная модель была сознательно упрощена до плоской таблицы `mem.user_facts`,
и `retrieveMemory` потерял параметр `entityKeys` вместе со слагаемым `entity_match`. Вызывающую сторону при этом не
почистили: `src/agent.js:511` до сих пор собирает `entityKeys` из `intent.entities` и передаёт их в функцию, которая
этот параметр игнорирует (`src/pipeline/retrieve.js:58` деструктурирует только `{ userId, domainKey, query, scopes }`).

Дополнительная проблема: `entities` объявлен как свободный объект (`additionalProperties: true`), из-за чего
`prepareJsonSchema` в `src/llm.js` понижает ВСЮ схему классификатора до `strict: false` — провайдер не гарантирует
соответствие ответа схеме на уровне декодера.

## Суть предложения

Факты, в тексте которых встречается упомянутая пользователем сущность, должны получать гарантированно высокую
релевантность при поиске памяти — даже если ни векторный, ни полнотекстовый поиск по всей фразе их не подняли.
Пользователь упомянул машину — факт «ездит на Nissan Teana» всплывает в `MEMORY_CONTEXT`.

## Какие реальные проблемы это решает

Текущая релевантность (`src/pipeline/retrieve.js:107`) — это `max(вектор, полнотекст, 0.15)` по целой фразе. Два
слабых места:

1. **Разбавление сигнала.** В длинной фразе «слушай, я тут подумала, может в выходные съездить куда-нибудь на
   машине» сущность «машина» — одно слово из пятнадцати. Эмбеддинг всей фразы будет про «планы на выходные», и факт
   про Nissan Teana может не попасть в топ ранжирования.
2. **Дыра в отборе кандидатов.** Шаг 1 (`retrieve.js:64-74`) берёт только 100 фактов по
   `confidence DESC, last_confirmed_at DESC`. Старый, редко подтверждаемый факт про конкретную сущность может вообще
   не попасть в пул кандидатов — и тогда никакое ранжирование его не спасёт. Предложение закрывает и это: сущностные
   совпадения добираются в пул кандидатов отдельным запросом.

Побочный выигрыш: после замены свободного объекта на строгий массив в схеме классификатора не остаётся свободных
полей, схема становится `strict: true`, и провайдер начинает гарантировать соответствие ответа схеме.

## Изменение 1: схема классификатора (`src/pipeline/classify.js`)

Свободный объект заменяется строгим массивом пар «тип — значение»:

```js
entities: {
  type: 'array',
  maxItems: 8,
  items: {
    type: 'object',
    additionalProperties: false,
    required: ['type', 'value'],
    properties: {
      type: { type: 'string' },   // person, place, vehicle, topic, product…
      value: { type: 'string' },  // «Берлин», «Nissan Teana», «мама»
    },
  },
},
```

В системный промпт добавляется инструкция: «Сущности возвращай массивом пар type/value; value — в начальной форме
(именительный падеж), без лишних слов; не более 8; не включай местоимения и общие слова». Начальная форма важна,
потому что полнотекстовый индекс `search_tsv` построен с конфигурацией `'simple'` — без русского стемминга, то есть
«Берлином» и «Берлин» для него разные токены.

## Изменение 2: передача сущностей (`src/agent.js:511`)

```js
entityKeys: (Array.isArray(intent.entities) ? intent.entities : [])
  .map((e) => (typeof e?.value === 'string' ? e.value.trim() : ''))
  .filter((v) => v.length >= 3),
```

Фильтр «не короче трёх символов» — защита от мусорных сущностей вроде «я», «он», которые совпали бы с половиной
памяти. Запасной вариант в `catch` (`agent.js:482`) меняется с `entities: {}` на `entities: []`.

## Изменение 3: буст в `retrieveMemory` (`src/pipeline/retrieve.js`)

Сигнатура снова принимает `entityKeys = []`. Два механизма, оба нужны.

### 3а. Добор кандидатов (recall)

Между шагами 1 и 2 — дополнительный запрос: факты, в которых встречается хотя бы одна сущность, добираются в пул,
даже если не прошли в топ-100. Один SQL-запрос на все сущности через `websearch_to_tsquery`, который понимает
оператор OR и кавычки для фраз:

```js
let entityIds = new Set();
if (entityKeys.length) {
  const orQuery = entityKeys.map((v) => `"${v.replace(/"/g, '')}"`).join(' OR ');
  const { rows } = await query(
    `SELECT id, domain_key, fact_type, fact_text AS memory_text, fact_text, confidence,
            evidence_count, last_confirmed_at, updated_at, source, persistent
       FROM mem.user_facts
      WHERE user_id = $1 AND status = 'active'
        AND (expires_at IS NULL OR expires_at > now())
        AND domain_key IN ('general', $2)
        AND search_tsv @@ websearch_to_tsquery('simple', $3)
      LIMIT 20`,
    [userId, domainKey, orQuery],
  );
  entityIds = new Set(rows.map((r) => r.id));
  const known = new Set(candidates.map((c) => c.id));
  for (const r of rows) {
    if (!known.has(r.id)) candidates.push(r); // факт вне топ-100 всё равно попадает в ранжирование
  }
}
```

Запрос дешёвый — идёт по тому же GIN-индексу, что и существующий полнотекстовый шаг 3, и выполняется только когда
сущности есть.

### 3б. Пол релевантности в ранжировании

Совпадение по сущности — это и есть релевантность: факт буквально о том предмете, который пользователь только что
упомянул. Поэтому вместо отдельного слагаемого в формуле весов (вариант первой версии, требовавший перенормировки
всех коэффициентов) используется гарантированный минимум релевантности — тот же приём, что уже применяется для
`CORE_TYPES` в `scoreFact` (`retrieve.js:42`):

```js
// Шаг 4, вместо текущей строки 107:
const entityFloor = entityIds.has(it.id) ? 0.7 : 0;
const relevance = Math.max(vecScores.get(it.id) ?? 0, textScores.get(it.id) ?? 0, entityFloor, 0.15);
```

Значение 0.7 ставит сущностный факт выше «среднесемантических» совпадений, но не выше точного попадания вектора
(0.85–0.95) — точные совпадения по смыслу всей фразы остаются главнее. Формула весов `scoreFact` не меняется,
перекалибровка не нужна.

### 3в. Подстрочная пометка по загруженным кандидатам (морфология)

Конфигурация `'simple'` не знает русских словоформ: сущность «Берлин» не совпадёт с «живёт в Берлине» через
`tsquery`. Дешёвая страховка — дополнительно к SQL-добору помечать совпадения подстрокой по уже загруженным
кандидатам: `it.fact_text.toLowerCase().includes(entity.toLowerCase())`. Для русского это работает неплохо, потому
что начальная форма обычно является префиксом словоформы («берлин» входит в «берлине», «ниссан» в «ниссане»).
Ноль дополнительных запросов; помеченные так факты получают тот же пол релевантности 0.7.

## Изменение 4: наблюдаемость

В лог события агента (стадия `memory`, `src/agent.js:501-505`) добавить извлечённые сущности и количество фактов,
добранных по ним. Без этого невозможно понять, работает ли буст и не шумит ли дешёвая модель-классификатор. В
веб-просмотрщике логов ничего менять не нужно — данные просто появятся в `data` события.

## Риски и смягчения

- **Шумные сущности от дешёвой модели** (gpt-5.4-nano может извлечь «выходные» как сущность). Смягчение тройное:
  лимит `maxItems: 8` в схеме, фильтр коротких значений в `agent.js`, и сам дизайн буста — пол 0.7 лишь поднимает
  факт в ранжировании, жёсткие лимиты групп (`LIMITS`) никуда не деваются.
- **Частые слова** («дом» совпадёт со множеством фактов). Частично закрывается фильтром длины; стоп-список заранее
  не городим (YAGNI) — добавим, только если на практике будет шуметь.
- **Лишний SQL-запрос на каждое сообщение.** Один индексированный запрос с `LIMIT 20`, только при наличии сущностей.
  На фоне двух существующих поисковых запросов и вызова `embed` — незаметно.

## Объём работ

Четыре файла кода, без миграций и изменений хранилища: `classify.js` (схема + промпт), `agent.js` (две строки),
`retrieve.js` (доборный запрос + пол релевантности + подстрочная пометка), тест на `retrieveMemory` с сущностями.
Плюс обновление документации (последний пункт плана).

## План реализации

1. `src/pipeline/classify.js` — строгая схема `entities` (массив пар type/value) и инструкция в системном промпте
   про начальную форму и лимит. Проверить, что схема классификатора стала `strict: true` в `prepareJsonSchema`.
2. `src/agent.js` — сбор `entityKeys` из массива пар, фильтр длины, `entities: []` в запасном варианте `catch`.
3. `src/pipeline/retrieve.js` — параметр `entityKeys = []`, доборный запрос через `websearch_to_tsquery`,
   подстрочная пометка по кандидатам, пол релевантности 0.7 в шаге 4.
4. Наблюдаемость — сущности и число добранных фактов в `data` события стадии `memory`.
5. Тесты — `tests/` на `retrieveMemory` с сущностями: добор факта вне топ-100, пол релевантности, фильтр коротких
   сущностей, пустой массив сущностей (поведение не меняется).
6. **Обновление документации `docs/ai-bot-with-memory/` строго по правилам
   `docs/ai-bot-with-memory/00-documentation-principles.md`**: новая механика вписывается как единственное текущее
   состояние системы (настоящее время, без «было/стало» и без упоминания изменений), согласованно во всех затронутых
   местах — минимум `04-architecture.md` (схема классификации, вызов retrieval, цитаты кода с `entityKeys`) и
   `08-prompts-and-models.md` (схема `skill_classification`, системный промпт классификатора, режим strict).

## Что остаётся сделать на будущее (вне рамок этого предложения)

- **Русская морфология в полнотекстовом поиске**: перевести `search_tsv` на конфигурацию `'russian'` или добавить
  вторую tsv-колонку с ней. Отдельная миграция и отдельное решение; пока компенсируется подстрочной пометкой (3в).
- **Стоп-список частых слов** для сущностей — только если практика покажет шум, которого не гасят фильтр длины и
  лимиты групп.
- **Судьба полей `needs_tools` и `candidate_tools`** в схеме классификатора: они обязательны в схеме, но не
  используются нигде в коде. Решить — удалить или начать использовать (например, для предварительной фильтрации
  набора инструментов основного вызова).
- **Использование `type` сущности**: сейчас тип извлекается, но в бусте участвует только `value`. В перспективе тип
  может пригодиться для группировки фактов или взвешивания (имена людей важнее тем).
- **Второе свободное поле из исходного списка**: `state_json` и `facts_to_memory` в
  `src/pipeline/history-compress.js:39-52` — отдельная задача, этим предложением не покрывается.
