# Как бот ведёт себя в проактивности

## Коротко

Бот больше не работает по принципу «проснулся каждые пять минут, нашёл повод и написал». Теперь у него есть два слоя
решения:

1. **Триггер** говорит: «появился повод подумать об отправке».
2. **Contact policy** решает: «уместно ли сейчас писать этому пользователю».

LLM не решает, писать или не писать. Это важно: проверка уместности алгоритмическая, дешёвая и работает по счётчикам,
времени, состоянию пользователя и типу сообщения. Модель вызывается только если алгоритм уже разрешил отправку, и только
для формулировки текста.

## Главное правило

Молчание пользователя считается ответом.

Если бот написал сам, а пользователь не ответил, бот не ищет новую тему и не пытается зайти с другого повода. Он
переходит в более осторожный режим. После повторного молчания бот замолкает до возвращения пользователя или до окончания
периода тишины.

Пример плохого поведения, которое теперь должно подавляться:

```text
09:00 — Можно продолжить тему памяти.
09:30 — Ещё можно сделать emoji-реакции.
10:00 — Доброе утро.
10:30 — Напомню про анализ логов.
```

Пример ожидаемого поведения:

```text
09:00 — Можно продолжить тему памяти, если будет удобно.
Пользователь молчит.
Бот молчит и не начинает новые темы.
```

## Режимы контакта

### `active`

Пользователь недавно отвечал или ещё не игнорировал инициативные сообщения. В этом режиме бот может отправить мягкую
инициативу, если сработал подходящий триггер и не превышены лимиты.

Пример:

```text
Пользователь вчера активно обсуждал задачу.
Прошли сутки без сообщений.
Сработал inactivity.
Бот может один раз мягко написать: «Если захочешь, можем вернуться к памяти ассистента».
```

### `cautious`

Бот уже отправил одно мягкое инициативное сообщение, а пользователь не ответил. Новые обычные идеи, check-in и внешние
поводы блокируются. Возможен только высоковажный follow-up после большой паузы.

Пример:

```text
Бот: «Можно позже аккуратно добить схему памяти».
Пользователь молчит.
Через час сработал другой повод.
Бот не пишет, потому что предыдущее инициативное сообщение ещё не закрыто ответом.
```

### `quiet`

Пользователь проигнорировал несколько мягких инициатив или активен `quiet_until`. Бот не пишет сам мягкие сообщения.
Явные пользовательские напоминания остаются отдельным классом и могут доставляться в своих лимитах.

Пример:

```text
Бот один раз написал сам.
Потом был разрешён один важный follow-up.
Пользователь снова молчит.
Бот уходит в quiet и ждёт входящего сообщения пользователя.
```

### `welcome_back`

Это не фоновое сообщение. Бот не пишет «рад видеть тебя снова» просто потому, что пользователь долго молчал. Возврат
появляется только когда пользователь сам написал после паузы.

Пример:

```text
Пользователь молчал два дня.
Бот ничего не пишет.
Пользователь: «Вернулся, давай продолжим».
Бот отвечает мягко: «Рад видеть. Можем продолжить с памяти или с emoji-реакций — как удобнее».
```

## Какие триггеры есть сейчас

### `inactivity`

**Что означает:** пользователь давно не писал.

**Параметр по умолчанию:** `PROACTIVE_INACTIVITY_MIN = 1440`, то есть 24 часа.

**Поведение:**

- срабатывает, если с последнего сообщения пользователя прошло не меньше заданного порога;
- не срабатывает, если этот же триггер уже срабатывал недавно;
- считается мягкой инициативой `soft_proactive`;
- проходит через общий дневной и недельный лимит;
- блокируется, если у пользователя уже есть незакрытое инициативное сообщение.

Пример:

```text
Пользователь писал вчера в 09:00.
Сейчас сегодня 09:30.
Прошло больше 24 часов.
inactivity готов к отправке.

Но если вчера бот уже сам писал и пользователь не ответил, contact policy запретит новое сообщение.
```

### `daily_checkin`

**Что означает:** наступил заданный час ежедневного check-in.

**Параметр по умолчанию:** `PROACTIVE_CHECKIN_HOUR = 10`.

**Поведение сейчас:**

- технически триггер проверяется в 10:00;
- не чаще одного раза в день;
- классифицируется как `social_proactive`;
- фоновая отправка социальных сообщений запрещена contact policy.

Иными словами, сам по себе ежедневный «доброе утро» не должен улетать пользователю фоном. Социальный тон уместен, когда
пользователь сам пришёл в диалог.

Пример:

```text
10:00.
daily_checkin готов по времени.
Пользователь молчит.
Contact policy отвечает: social_requires_incoming_user_message.
Бот не пишет.
```

### `goal_reminder`

**Что означает:** у пользователя есть цель в памяти, и пришло время мягко спросить о прогрессе.

**Параметр по умолчанию:** `PROACTIVE_GOAL_INTERVAL_MIN = 2880`, то есть 48 часов.

**Поведение:**

- срабатывает, если в таблице `mem.user_facts` есть активный (`status = 'active'`, не истёкший) факт типа
  `fact_type = 'goal'`;
- не срабатывает чаще заданного интервала;
- считается мягкой инициативой `soft_proactive`;
- проходит через общий фильтр уместности;
- если пользователь молчит после прошлого proactive, обычный goal reminder будет заблокирован.

Пример:

```text
В памяти есть цель: «дописать инструмент анализа логов».
Прошло 48 часов с прошлого goal reminder.
Пользователь недавно отвечал.
Бот может написать: «Если актуально, можем сегодня коротко добить анализ логов».
```

Другой пример:

```text
В памяти есть цель.
Но последнее сообщение в диалоге — инициативное сообщение бота, на которое пользователь не ответил.
Goal reminder не отправляется, потому что это стало бы давлением.
```

### `welcome_back`

**Что означает:** пользователь сам вернулся после паузы.

**Параметр по умолчанию:** `PROACTIVE_WELCOME_GAP_MIN = 60`.

**Поведение сейчас:**

- в списке дефолтных триггеров он ещё есть как настраиваемый повод;
- фоновый `shouldFire` для него всегда возвращает `false`;
- реальное поведение живёт во входящем пайплайне `handleMessage`;
- welcome-back контекст добавляется в prompt, когда выполнены оба условия: пауза с прошлого сообщения пользователя
  не меньше `PROACTIVE_WELCOME_GAP_MIN`, и при этом либо бот ждал ответа (есть незакрытая инициатива или активен
  quiet), либо пауза превысила порог неактивности `PROACTIVE_INACTIVITY_MIN` (24 часа).

Пример:

```text
Бот вчера написал мягкую инициативу, пользователь не ответил.
Пользователь сегодня: «Я вернулся».
Пауза больше часа, и бот ждал ответа.
Бот получает welcome-back контекст и отвечает спокойнее, без списка накопленных идей.
```

Контрпример:

```text
Пользователь молчал 3 часа, но бот сам ничего не писал.
Пользователь: «Я вернулся».
Пауза меньше суток и незакрытой инициативы нет — welcome-back контекст не добавляется.
Бот отвечает как на обычное сообщение.
```

## Внешние события

Есть отдельный контур внешних событий, например новости из заглушки. Он включается флагом `PROACTIVE_EVENTS_ENABLED`.

Поведение:

- берётся одно событие за проход;
- для каждого пользователя сначала проверяется contact policy;
- если пользователь в quiet или уже не ответил на прошлую инициативу, LLM-релевантность даже не вызывается;
- только после алгоритмического разрешения модель оценивает релевантность события;
- если событие релевантно и выше порога `NEWS_RELEVANCE_THRESHOLD`, бот формирует сообщение.

Порог по умолчанию:

```text
NEWS_RELEVANCE_THRESHOLD = 0.6
```

Пример:

```text
Вышла новость про локальные LLM.
У пользователя в памяти есть интерес к AI.
Пользователь в active-режиме.
Contact policy разрешает проверку.
Модель оценивает релевантность на 0.8.
Бот может один раз коротко предложить обсудить новость.
```

Пример блокировки:

```text
Та же новость.
Но пользователь не ответил на прошлое инициативное сообщение.
Contact policy запрещает контакт.
Модель релевантности не вызывается.
Сообщение не отправляется.
```

## Явные напоминания пользователя

Напоминания, которые пользователь сам попросил создать, считаются не мягкой инициативой, а `requested_reminder`.

Параметр по умолчанию:

```text
PROACTIVE_REQUESTED_REMINDER_DAILY_LIMIT = 2
```

Поведение:

- такие напоминания не считаются новой идеей бота;
- они не сбрасывают quiet-режим для мягкой проактивности;
- у них отдельный дневной лимит;
- если лимит превышен, задача не теряется, а откладывается на час (`next_run_at = now() + 1 час`), причина отказа
  записывается в metadata задачи.

Пример:

```text
Пользователь попросил: «Напомни завтра в 10 проверить цены».
В 10:00 планировщик доставляет напоминание.
Это не означает, что бот снова получил право писать мягкие идеи.
```

## Общие лимиты

Параметры по умолчанию:

```text
PROACTIVE_SOFT_DAILY_LIMIT = 1
PROACTIVE_SOFT_WEEKLY_LIMIT = 3
PROACTIVE_MIN_SOFT_PAUSE_MIN = 360
PROACTIVE_QUIET_AFTER_UNANSWERED = 2
PROACTIVE_QUIET_HOURS_AFTER_IGNORES = 24
PROACTIVE_INTERVAL_MS = 300000
```

Расшифровка:

- мягкая инициатива — максимум 1 раз в день;
- мягкая инициатива — максимум 3 раза в неделю;
- между мягкими инициативами минимум 6 часов;
- после 2 инициатив без ответа бот уходит в quiet;
- quiet после игноров длится 24 часа, если пользователь сам не вернулся раньше;
- воркер проверяет фоновые поводы каждые 5 минут.

## Как выбирается одно сообщение

За один проход пользователь может получить максимум одно proactive-сообщение. Если одновременно готовы несколько
поводов, бот выбирает лучший разрешённый.

Приоритет:

```text
critical
scheduled_reminder
goal_reminder
event
inactivity
daily_checkin
```

Важно: `daily_checkin` имеет низкий приоритет и как социальный фоновый повод всё равно блокируется contact policy.

Пример:

```text
Одновременно готовы inactivity и goal_reminder.
Пользователь в active-режиме.
Оба проходят policy.
Бот выбирает goal_reminder, потому что он приоритетнее.
```

## Что происходит после отправки

Если бот отправил мягкую инициативу:

- сохраняется `last_proactive_sent_at`;
- сохраняется `last_soft_proactive_sent_at`;
- увеличивается `unanswered_proactive_count`;
- увеличивается счётчик игнорируемых мягких инициатив `ignored_soft_count_7d`;
- увеличиваются дневной и недельный счётчики мягкой инициативы;
- запоминается последний тип триггера и тема;
- при достижении порога игноров выставляется `quiet_until`.

Если пользователь потом написал:

- `unanswered_proactive_count` сбрасывается в 0;
- `quiet_until` очищается;
- режим снова становится `active`;
- если пауза была длинной, в текущий ответ добавляется welcome-back контекст.

## Показательные сценарии

### Сценарий 1: обычная пауза

```text
День 1, 09:00.
Пользователь активно общался.

День 2, 09:30.
Прошло больше 24 часов.
inactivity проходит.
Бот пишет одно мягкое сообщение.

День 2, 10:00.
Пользователь молчит.
Другой повод готов.
Бот не пишет, потому что есть незакрытая инициатива.
```

### Сценарий 2: пользователь игнорирует два раза

```text
Бот отправил мягкую инициативу.
Пользователь молчит.

Позже разрешён один high follow-up.
Бот отправил его.
Пользователь снова молчит.

Состояние становится quiet.
Обычные идеи, check-in, события и goal reminder больше не отправляются.
```

### Сценарий 3: пользователь вернулся

```text
Пользователь молчал несколько часов или дней.
Бот не писал фоновое welcome-back.

Пользователь: «Давай продолжим».
Бот сбрасывает quiet, видит паузу и отвечает мягко:
«Рад, что ты вернулся. Можем продолжить с анализа логов или сначала закрыть память».
```

### Сценарий 4: явное напоминание

```text
Пользователь заранее попросил напомнить: «Завтра в 10 проверить цены».
Даже если soft-проактивность в quiet, планировщик может доставить requested reminder.
Но после этого бот не начинает новые идеи сам.
```

## Как это устроено технически

### Таблицы

`mem.proactive_triggers` хранит сами поводы:

- `trigger_type`;
- `config`;
- `enabled`;
- `last_fired_at`.

`mem.proactive_contact_state` хранит состояние отношений с пользователем:

- `mode`;
- `last_proactive_sent_at`;
- `last_soft_proactive_sent_at`;
- `last_user_reply_after_proactive_at`;
- `unanswered_proactive_count`;
- `ignored_soft_count_7d`;
- `daily_soft_count`;
- `daily_requested_reminder_count`;
- `weekly_soft_count`;
- `counters_day` и `counters_week` — даты, к которым привязаны дневные и недельные счётчики (по ним счётчики
  сбрасываются при смене дня и недели);
- `quiet_until`;
- `last_trigger_type`;
- `last_topic_key`.

### Основной модуль

Главный алгоритм находится в:

```text
src/pipeline/proactiveContactPolicy.js
```

Ключевые функции:

- `classifyTriggerCandidate` превращает триггер в класс сообщения;
- `classifyEventCandidate` классифицирует внешнее событие;
- `classifyReminderCandidate` классифицирует задачу планировщика;
- `evaluateContactPolicy` принимает алгоритмическое решение `allowed/denied`;
- `recordProactiveSent` обновляет состояние после отправки;
- `recordUserInboundForContactPolicy` сбрасывает тишину на входящем сообщении;
- `chooseBestAllowed` выбирает один лучший разрешённый повод.

### Проход фонового воркера

Фоновая проактивность идёт так:

```text
1. Найти пользователей с включённой проактивностью и включёнными поводами.
2. Для каждого пользователя загрузить contact state.
3. Проверить условия триггеров через shouldFire.
4. Превратить сработавшие триггеры в candidates.
5. Прогнать candidates через evaluateContactPolicy.
6. Выбрать максимум один лучший candidate.
7. Только теперь вызвать LLM для текста.
8. Сохранить сообщение в историю и notification_outbox.
9. Обновить last_fired_at и proactive_contact_state.
```

### Почему это дешёво

До LLM используются только:

- SQL-запросы;
- timestamps;
- счётчики;
- тип сообщения;
- простые правила.

Если пользователь в quiet или уже не ответил на прошлую инициативу, expensive-часть не запускается. Это касается и
внешних событий: сначала contact policy, потом LLM-релевантность.

### Где находится welcome back

Welcome back живёт не в фоновом воркере, а во входящем сообщении:

```text
src/agent.js
```

Когда пользователь пишет, `handleMessage` вызывает `recordUserInboundForContactPolicy`. Если выполнено условие
возвращения (пауза не меньше `PROACTIVE_WELCOME_GAP_MIN` и при этом бот ждал ответа либо пауза превысила порог
неактивности), в prompt добавляется `WELCOME_BACK_CONTEXT`. Это не отдельная доставка, а настрой текущего ответа.

### Где происходит доставка

Ядро не зависит от Telegram. Оно кладёт сообщение в:

```text
mem.notification_outbox
```

Конкретный канал уже забирает очередь и доставляет сообщение пользователю. Сегодня это может быть Telegram, завтра —
другой адаптер.
