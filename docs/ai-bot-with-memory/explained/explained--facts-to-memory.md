# facts_to_memory: как факты из сжимаемой истории попадают в долговременную память

Подробное объяснение «для чайника»: что это за поле, зачем оно нужно, какой путь проходят данные и что
даёт перевод его схемы на строгий режим. Код: `src/pipeline/history-compress.js` (производитель),
`src/pipeline/facts.js` (потребитель).

## Контекст: сжатие истории в двух словах

История диалога делится на «горячее окно» (последние сообщения, передаются модели дословно) и «холодную
зону» (всё, что старше). Когда холодная зона разрастается сверх порога, отдельный вызов дешёвой модели —
суммаризатор — пересказывает её кратко. Сырые сообщения после этого в контекст больше не попадают.

Здесь возникает риск: вместе с сырыми сообщениями погибли бы и ценные сведения о пользователе. Пример
куска диалога недельной давности:

> — Хочу спланировать поездку в Берлин на сентябрь.
> — Отлично! На сколько дней?
> — Дней на пять. Кстати, я не люблю ранние вылеты.
> — Учту. Отели смотрим в центре?
> — Да. И ещё: я вегетарианка, рестораны подбирай соответствующие.

«Не любит ранние вылеты» и «вегетарианка» пригодятся через месяц в совсем другом разговоре. Поле
`facts_to_memory` — это спасательная шлюпка для таких сведений: суммаризатор выносит их из сжимаемой
истории в долговременную память бота.

## Что возвращает суммаризатор

Массив фактов фиксированной формы — ровно той же, что у основного конвейера извлечения фактов
(`EXTRACT_SCHEMA` в `src/pipeline/facts.js`):

```json
"facts_to_memory": [
  { "type": "preference", "fact_text": "Пользователь не любит ранние вылеты", "confidence": 0.9, "ttl_days": null },
  { "type": "profile",    "fact_text": "Пользователь — вегетарианка",         "confidence": 0.95, "ttl_days": null }
]
```

Четыре поля:

- `type` — тип факта из словаря десяти типов (`FACT_TYPES`): profile, preference, habit, goal,
  emotional_pattern, activity_rhythm, communication_style, open_loop, topic_energy, discovery_seed;
- `fact_text` — короткая фраза от третьего лица («Пользователь …»);
- `confidence` — уверенность модели от 0 до 1;
- `ttl_days` — срок жизни факта в днях; `null` — бессрочный. Нужен для временных сведений: например,
  у незакрытой линии (`open_loop`, «собиралась к врачу, обещала рассказать») срок по умолчанию 30 дней.

## Путь данных по шагам

### Шаг 1. Приведение формы — factsToCandidates (`history-compress.js`)

Функция-прихожая: подставляет значения по умолчанию (нет типа — `profile`, нет уверенности — 0.7) и
выкидывает явный мусор (пустой текст, упоминания паспортов и карт по эвристике `f_isSensitive`). При
строгой схеме ответа эти значения по умолчанию — страховка для запасного нестрогого режима, в строгом
режиме модель физически не может прислать факт без обязательного поля.

### Шаг 2. Сохранение с дедупликацией — saveFacts → saveFact (`facts.js`)

Каждый факт проходит конвейер по очереди:

**2а. Проверки на входе.** Тип из известного списка, текст непустой, уверенность не ниже порога
`config.facts.minConfidence`. Не прошёл — факт пропускается с результатом `skipped`.

**2б. Выбор домена.** «Общечеловеческие» типы (профиль, предпочтения, привычки и другие из
`GENERAL_TYPES`) кладутся в домен `general` — они видны из любого разговора. Привязанные к теме (`goal`,
`open_loop`) — в домен текущего разговора.

**2в. Срок жизни.** Из `ttl_days` вычисляется дата забывания `expires_at`; `null` — факт бессрочный.

**2г. Поиск двойника.** Для текста факта вычисляется эмбеддинг (числовой вектор смысла), в
`mem.user_facts` ищется ближайший по смыслу существующий факт того же типа. Три исхода:

- **Очень похож** (сходство ≥ `confirmSimilarity`) — тот же факт. Новая строка НЕ создаётся: у
  существующей продлевается свежесть и растёт счётчик подтверждений `evidence_count`. Важно: у
  источников фактов есть ранги доверия (`SOURCE_RANK`), и `history_summary` — самый слабый (ранг 0).
  Факт из сжатия истории может подтвердить существующий факт, но не может переписать формулировку факта,
  пришедшего из прямой реплики пользователя.
- **Похож, но значение другое** (сходство ≥ `replaceSimilarity`; та же тема: «живёт в Москве» против
  «переехала в Казань») — замещение: старая строка архивируется, вставляется новая. Но опять по рангам:
  слабый `history_summary` не имеет права заместить факт из прямой реплики — он лишь продлевает его
  свежесть. Это защита от ситуации, когда модель при сжатии что-то напутала и пытается испортить
  надёжный факт.
- **Не похож ни на что** — создаётся новая строка в `mem.user_facts` с эмбеддингом, доменом, сроком
  жизни и источником `history_summary`.

### Шаг 3. Возвращение факта в разговор (через день, месяц, год)

Через месяц пользователь пишет: «Посоветуй ресторан на вечер». Поиск памяти (`retrieveMemory` в
`src/pipeline/retrieve.js`) поднимает факт «Пользователь — вегетарианка» по семантической близости, факт
попадает в блок `MEMORY_CONTEXT` промпта — и бот советует вегетарианские рестораны, не переспрашивая.
Ради этого вся механика и существует: сырые сообщения давно сжаты и недоступны, но факт пережил сжатие.

## Что даёт строгая схема на этом пути

В нестрогом режиме модель может вернуть, например, `{"text": "вегетарианка"}` — поле названо
неправильно. Код подставит тип по умолчанию, увидит пустой `fact_text` и тихо пропустит факт. Факт
потерян, никто не заметил. В строгом режиме такой ответ невозможен технически: каждый элемент массива
обязан иметь ровно четыре правильных поля, а `type` — только значение из словаря `FACT_TYPES`. Сам
дальнейший путь данных (шаги 1–3) при этом не меняется ни на строку — меняется только гарантия формы на
входе.

## Сводка

| Вопрос | Ответ |
|---|---|
| Откуда берётся | Суммаризатор истории выносит устойчивые факты из сжимаемой холодной зоны |
| Куда пишется | Таблица `mem.user_facts` через обычный конвейер `saveFacts` (пороги, дедупликация, ранги) |
| Кто читает | Код: поиск памяти `retrieveMemory` → блок `MEMORY_CONTEXT` будущих запросов |
| Горизонт жизни | Долгий: месяцы и годы (с учётом `ttl_days`), переживает любые разговоры |
| Ранг доверия | Минимальный (`history_summary`): подтверждает чужие факты, но не переписывает их |
