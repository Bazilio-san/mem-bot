# Предложение: поджатие истории диалога для чат-бота с памятью

Да. По текущему документу **полноценной функции поджатия истории сейчас нет**.

Сейчас сделано так:

1. В запрос к модели подмешиваются **последние 8 сообщений** через `getRecentMessages(conversation.id, 8)`.
2. Есть таблица `conversation_summaries`, но она **пока не наполняется**: в документе прямо сказано, что сжатие длинной истории в короткое резюме — это кандидат на доделку.
3. Память уже минимизируется отдельно: в промпт попадает не вся память, а ограниченный набор фактов, например `profile: 7`, `dialog: 5`, `domain: 12`, общий предел `30`.
4. В текущей схеме Chat Completions историю действительно надо передавать вручную: состояние диалога сохраняется через чередование сообщений `user` и `assistant`, которые приложение само добавляет в следующий запрос.

Моё предложение: добавить отдельный слой **сжатой истории диалога**, который работает между сырой историей сообщений и долговременной памятью.

## Рекомендуемая схема

Я бы ввёл такие параметры:

```js
const RAW_KEEP_MESSAGES = 8;          // последние 8 сообщений не трогаем
const HISTORY_MAX_SIZE = 9000;        // порог старой истории в токенах
const HISTORY_SHRINKED_SIZE = 2500;   // целевой размер сжатой истории
const HISTORY_MIN_COMPRESS_GAIN = 0.35;
```

Тут **токены** — это условные кусочки текста, которыми модель считает размер запроса. В среднем русское слово может занимать больше одного токена, поэтому лучше считать именно токены, а не символы.

Почему такие значения:

| Параметр | Рекомендация | Логика |
|---|---:|---|
| `RAW_KEEP_MESSAGES` | `8` | Уже соответствует текущему коду. Это примерно 4 пары «пользователь → бот», если считать по сообщениям. |
| `HISTORY_MAX_SIZE` | `8000–12000`, по умолчанию `9000` | Старую историю не надо сжимать слишком рано, иначе будет лишняя стоимость на поджатие. |
| `HISTORY_SHRINKED_SIZE` | `2000–3500`, по умолчанию `2500` | Обычно достаточно, чтобы сохранить решения, ограничения, текущую задачу и важные повороты диалога. |
| Соотношение | `SHRINKED_SIZE ≈ 25–35% от MAX_SIZE` | История реально уменьшается, но не превращается в бесполезные 5 строк. |

Для MVP я бы взял именно:

```js
HISTORY_MAX_SIZE = 9000
HISTORY_SHRINKED_SIZE = 2500
```

Важно: `MAX_SIZE` и `SHRINKED_SIZE` должны относиться **не ко всему промпту**, а только к старой части истории, то есть к сообщениям **до последних 8**.

## Как сделать «ближние фразы ужимаются меньше, дальние — больше»

Нужна не одна плоская сводка, а несколько слоёв истории.

```text
[Системный промпт]
[MEMORY_CONTEXT: профиль, доменная память, задачи]
[HISTORY_CONTEXT: сжатая история]
[последние 8 сообщений без изменений]
[новое сообщение пользователя]
```

Сжатую историю лучше делить так:

| Слой | Что входит | Как сжимать |
|---|---|---|
| `raw` | последние 8 сообщений | вообще не трогать |
| `near` | сообщения перед последними 8, например ещё 12–20 сообщений | сжимать мягко: сохранить детали, формулировки, выбранные варианты |
| `middle` | более старая середина диалога | сжимать сильнее: решения, причины, ограничения, открытые вопросы |
| `far` | совсем старая часть | сжимать максимально: только итоговые выводы, важные решения, ссылки на сущности |

Пример итогового блока:

```text
HISTORY_CONTEXT

Недавний контекст до последних 8 сообщений:
- Пользователь выбирал схему памяти для универсального бота.
- Важное требование: специализация бота может быть любой, поэтому доменная память должна быть универсальной.
- Пользователь хочет конкретную реализацию в PostgreSQL, а не абстрактное описание.

Средний контекст:
- Было согласовано 5 видов памяти: краткосрочная, профильная, доменная, защищённая, память задач.
- Основной критерий качества: в модель попадет только малый релевантный фрагмент.

Дальний контекст:
- Проект связан с ботом с долговременной памятью и проверяемой архитектурой.
```

## Как не дублировать память

Это критично. История и память должны решать разные задачи.

**Память** хранит устойчивые факты:

```text
Пользователь предпочитает русский язык и понятные объяснения без лишних англицизмов.
```

**Сжатая история** хранит ход текущего разговора:

```text
В этом диалоге пользователь уточняет, как реализовать поджатие истории поверх уже существующей памяти.
```

Перед созданием сводки надо передавать суммаризатору уже выбранную память:

```json
{
  "active_memory": [
    {
      "scope": "profile",
      "memory_text": "Пользователь предпочитает понятный русский язык без лишних англицизмов."
    },
    {
      "scope": "domain",
      "memory_text": "Бот проектируется как универсальный агент с пятью видами памяти."
    }
  ],
  "messages_to_summarize": []
}
```

И в промпте суммаризатора написать правило:

```text
Не дублируй в summary_text факты, которые уже есть в active_memory.
Если факт уже есть в памяти, не повторяй его в истории.
В истории оставляй только то, что нужно для понимания текущего хода диалога:
решения, незакрытые вопросы, выбранные варианты, причины изменений.
```

Так мы избегаем ситуации, когда одно и то же хранится и в `memory_items`, и в `conversation_summaries`.

## Что добавить в базу

Таблица `conversation_summaries` уже есть, но я бы расширил её:

```sql
ALTER TABLE mem.conversation_summaries
ADD COLUMN IF NOT EXISTS layer text NOT NULL DEFAULT 'middle'
  CHECK (layer IN ('near','middle','far','full')),

ADD COLUMN IF NOT EXISTS covered_from_message_id uuid,
ADD COLUMN IF NOT EXISTS covered_to_message_id uuid,

ADD COLUMN IF NOT EXISTS source_message_count integer NOT NULL DEFAULT 0,
ADD COLUMN IF NOT EXISTS source_token_count integer NOT NULL DEFAULT 0,
ADD COLUMN IF NOT EXISTS summary_token_count integer NOT NULL DEFAULT 0,

ADD COLUMN IF NOT EXISTS memory_dedupe jsonb NOT NULL DEFAULT '{}'::jsonb,
ADD COLUMN IF NOT EXISTS summary_version integer NOT NULL DEFAULT 1;
```

Зачем:

| Поле | Зачем |
|---|---|
| `layer` | понимать, это ближняя, средняя или дальняя сводка |
| `covered_from_message_id` / `covered_to_message_id` | знать, какие сообщения уже покрыты сводкой |
| `source_token_count` | понимать, сколько было до сжатия |
| `summary_token_count` | проверять, что реально ужали до лимита |
| `memory_dedupe` | хранить, какие факты не попали в историю, потому что уже есть в памяти |
| `summary_version` | потом менять формат сводки без хаоса |

## Алгоритм после каждого ответа

```text
1. Сохрани user-сообщение и assistant-сообщение.
2. Посчитай токены старой части истории:
   все сообщения диалога, кроме последних RAW_KEEP_MESSAGES.
3. Если старая часть меньше HISTORY_MAX_SIZE — ничего не делай.
4. Если больше HISTORY_MAX_SIZE:
   4.1. Возьми старые сообщения, которые ещё не покрыты актуальной сводкой.
   4.2. Возьми активную память пользователя и домена.
   4.3. Отправь в дешёвую модель-суммаризатор.
   4.4. Получи JSON:
        summary_text,
        state_json,
        facts_to_memory,
        dropped_because_in_memory.
   4.5. Сохрани summary в conversation_summaries.
   4.6. Кандидаты facts_to_memory отправь в обычный контур памяти.
5. В следующий ответ модели передавай:
   MEMORY_CONTEXT,
   HISTORY_CONTEXT,
   последние 8 сообщений,
   новое сообщение пользователя.
```

Поджатие лучше делать **после ответа пользователю**, как сейчас уже делается извлечение фактов в память. Это не будет тормозить основной ответ.

## Промпт для суммаризатора

```text
Ты сжимаешь старую часть истории диалога для чат-бота с долговременной памятью.

Твоя задача:
1. Сохранить только то, что нужно для продолжения текущего диалога.
2. Не трогать последние сообщения — они не переданы тебе и будут добавлены отдельно.
3. Не дублировать факты, которые уже есть в active_memory.
4. Ближний контекст описывать подробнее, дальний — короче.
5. Не превращать историю в память профиля. Устойчивые факты вынеси в facts_to_memory.
6. Не сохранять секретные данные в открытом виде.
7. Не сохранять мусор: приветствия, повторы, эмоции без последствий, одноразовые фразы.

Верни строго JSON.
```

Схема ответа:

```json
{
  "type": "object",
  "required": [
    "summary_text",
    "state_json",
    "facts_to_memory",
    "dropped_because_in_memory"
  ],
  "additionalProperties": false,
  "properties": {
    "summary_text": {
      "type": "string"
    },
    "state_json": {
      "type": "object",
      "additionalProperties": true,
      "properties": {
        "current_goal": { "type": ["string", "null"] },
        "decisions": { "type": "array", "items": { "type": "string" } },
        "open_questions": { "type": "array", "items": { "type": "string" } },
        "constraints": { "type": "array", "items": { "type": "string" } },
        "next_steps": { "type": "array", "items": { "type": "string" } }
      }
    },
    "facts_to_memory": {
      "type": "array",
      "items": {
        "type": "object",
        "additionalProperties": true
      }
    },
    "dropped_because_in_memory": {
      "type": "array",
      "items": { "type": "string" }
    }
  }
}
```

## Как собирать итоговый контекст для ответа

Сейчас у тебя:

```js
const history = await getRecentMessages(conversation.id, 8);

const messages = [
  { role: 'system', content: MAIN_SYSTEM },
  { role: 'system', content: memoryContext },
  ...history,
  { role: 'user', content: userMessage }
];
```

Я бы сделал так:

```js
const historySummary = await getConversationHistoryContext({
  conversationId: conversation.id,
  userId: user.id,
  domainKey: effectiveDomain,
  query: userMessage,
  maxTokens: HISTORY_SHRINKED_SIZE,
});

const history = await getRecentMessages(conversation.id, RAW_KEEP_MESSAGES);

const messages = [
  { role: 'system', content: MAIN_SYSTEM },
  { role: 'system', content: memoryContext },
  { role: 'system', content: historySummary },
  ...history.map(toMessage),
  { role: 'user', content: userMessage },
];
```

Порядок важен. Для выгоды от кэширования статичные части промпта лучше держать в начале, а переменные части — ближе к концу.

## Важное правило при конфликте

Приоритет должен быть такой:

```text
1. Текущее сообщение пользователя
2. Последние 8 сырых сообщений
3. Сжатая история
4. Долговременная память
5. Старые дальние сводки
```

То есть если в старой сводке написано «пользователь выбрал вариант А», а в последних 8 сообщениях он выбрал вариант Б, модель должна считать актуальным вариант Б.

Это хорошо ложится на уже существующее правило проекта: текущий запрос важнее старой памяти, а `MEMORY_CONTEXT` подаётся как справка, не как команда.

## Минимальный план реализации

Я бы добавил 3 файла:

```text
src/pipeline/history-compress.js
src/pipeline/history-context.js
src/pipeline/token-counter.js
```

### `history-compress.js`

Отвечает за:

```text
- проверить размер старой истории;
- выбрать сообщения до последних 8;
- вызвать модель-суммаризатор;
- сохранить запись в conversation_summaries;
- отправить устойчивые факты в контур памяти.
```

### `history-context.js`

Отвечает за:

```text
- достать актуальные сводки;
- выбрать ближние подробнее, дальние короче;
- убрать дубли с MEMORY_CONTEXT;
- собрать HISTORY_CONTEXT.
```

### `token-counter.js`

Отвечает за:

```text
- примерно считать токены сообщений;
- обновлять conversation_messages.token_count;
- проверять MAX_SIZE и SHRINKED_SIZE.
```

## Как проверить тестами

Добавить проверки:

| Тест | Что проверяет |
|---|---|
| История меньше `MAX_SIZE` | суммаризатор не вызывается |
| История больше `MAX_SIZE` | создаётся запись в `conversation_summaries` |
| Последние 8 сообщений | попадают в промпт без изменений |
| Сжатая история | не превышает `SHRINKED_SIZE` |
| Ближний слой | содержит больше деталей, чем дальний |
| Дальний слой | содержит только решения и важные выводы |
| Дубли с памятью | факты из `memory_items` не повторяются в `summary_text` |
| Конфликт | последние сообщения побеждают старую сводку |
| Секретные данные | не попадают в открытую сводку |
| Удаление памяти | после удаления памяти сводка не раскрывает тот же факт обратно |

Итог: текущая система уже хорошо подготовлена к этому — таблица для сводок есть, `token_count` у сообщений есть, последние 8 сообщений уже выделены. Нужно только добавить недостающий этап: **автоматическое поджатие старой истории с дедупликацией относительно памяти**.
