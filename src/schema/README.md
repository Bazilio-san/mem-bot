# Слой per-domain схем памяти

Этот модуль превращает свободное поле `data` каждого факта в проверяемый контракт для конкретного домена и
приводит `entity_key` к стабильному виду. Благодаря этому `data` становится машиночитаемым (его можно фильтровать
в SQL по конкретным полям), а дедупликация фактов по сущности — надёжной, потому что ключ канонизирован.

**Схема обязательна.** Предметный факт (тот, у которого задан `entity_type`) сохраняется только если у домена есть
активная схема, в ней объявлена эта сущность, и `data` проходит валидацию. Любое нарушение приводит к отклонению
факта — он не сохраняется. Факт без `entity_type` (например свободное предпочтение профиля) схемой не описывается
и проходит без изменений: это не сущность домена, и валидировать в нём нечего.

## Где хранятся схемы

Схемы живут в двух местах, и это разделение важно понимать. Каталог `schemas/` в корне репозитория хранит схемы как
исходный текст в виде файлов `*.json` — их удобно ревьюить, держать в git и править руками. Таблица `mem.domain_schemas`
(создаётся миграцией `migrations/006_domain_schemas.sql`) хранит схему как источник истины во время выполнения. Рантайм
читает только таблицу; файлы из `schemas/` он не открывает. Файл становится действующей схемой лишь после команды `save`,
которая записывает его в таблицу новой активной версией.

## Как добавить новый домен со схемой

Поток состоит из четырёх шагов: сгенерировать черновик, отревьюить его руками, сохранить в реестр и при желании
проверить результат. Команды ниже показаны и через npm-скрипты, и через прямой вызов `node` — они эквивалентны.

### Шаг 1. Сгенерировать черновик (LLM)

Генератор по названию домена предлагает набор сущностей с закрытыми схемами `data` и правилами формирования ключа.
Ключ домена (`--key`) обязателен; описание (`--desc`) и примеры реплик (`--sample`, можно повторять) помогают модели
точнее угадать сущности и поля. Результат пишется в файл `schemas/<ключ>.draft.json`.

```bash
node src/schema/cli.js generate "Поиск и покупка авиабилетов" --key flights \
  --desc "перелёты, маршруты, пассажиры" --sample "ищу билет из Казани" --sample "не люблю ночные рейсы"
```

### Шаг 2. Отревьюить и поправить черновик

Откройте `schemas/<ключ>.draft.json` и проверьте предложенные сущности, поля `data` и словари `entity_key`. Генератор —
помощник, а не финальный авторитет: он может предложить неполный или избыточный набор полей, поэтому ручная проверка
обязательна. После правки файл можно оставить как `*.draft.json` либо переименовать в `<ключ>.json` — команда `save`
принимает любой путь.

### Шаг 3. Сохранить схему в реестр

Команда `save` валидирует определение по мета-схеме, вычисляет следующую версию, переводит прежнюю активную версию в
архив и активирует новую. Если домена ещё нет в таблице `mem.agent_domains`, он заводится туда автоматически. Так что
отдельно регистрировать домен не нужно — достаточно сохранить его схему.

```bash
node src/schema/cli.js save schemas/flights.json
```

### Шаг 4. Проверить результат

```bash
node src/schema/cli.js list          # домены с активными схемами и их версии
node src/schema/cli.js show flights  # активная схема конкретного домена
```

Те же действия доступны через npm-скрипты: `npm run schema:generate -- "<название>" --key <ключ>`, затем
`npm run schema:save -- schemas/<файл>`, а также `npm run schema:list` и `npm run schema:show -- <ключ>`.

## Что происходит дальше, при записи каждого факта

После сохранения схемы ничего дополнительно подключать не нужно. Контур записи памяти (`src/pipeline/merge.js`,
функция `processCandidate`) на каждый предметный факт вызывает `validateAndCanonicalize` из `validate.js`. Этот шаг
проверяет `data` по закрытой схеме сущности и приводит `entity_key` к каноническому виду. Перед тем как признать `data`
невалидным, выполняется дешёвая кодовая нормализация: отбрасываются лишние ключи, одиночное значение оборачивается в
массив там, где схема ждёт массив, строка-число приводится к числу, отсутствующие поля заполняются `null`. Если и после
нормализации `data` не сходится, факт отклоняется. Номер версии схемы и замечания канонизации записываются в поле
`metadata` строки факта, чтобы было видно, по какому контракту он сохранён.

Этап извлечения (`src/pipeline/extract.js`) работает в два прохода, когда у домена есть схема. Первый проход определяет,
что вообще стоит запомнить, и какой это тип сущности — в его промпт подставляется перечень сущностей домена и их полей.
Второй проход для каждого предметного кандидата, чья сущность объявлена в схеме, перезаполняет `data` и `entity_key`
строго по закрытой схеме именно этой сущности: модели отправляется схема, где `data` — это `data_schema` сущности, а
`entity_key` для режима `fixed_vocab` ограничен словарём. Так модель заполняет ровно нужные поля с точными типами и
значениями, и на запись приходит уже валидный кандидат. Если строгий проход почему-то не удался, кандидат идёт дальше
как есть, а итоговый контроль всё равно срабатывает в `validateAndCanonicalize` при записи.

## Формат определения домена

Полное описание формы объекта `definition` задано мета-схемой в `meta.js` (константа `DEFINITION_META_SCHEMA`) и
проверяется функцией `validateDefinition`. Кратко: у определения есть `domain_key`, `title`, необязательное `description`,
список допустимых видов памяти `allowed_memory_kinds` и массив `entities`. Каждая сущность задаёт `entity_type`, правило
`entity_key` и закрытую `data_schema`. Закрытость означает `additionalProperties: false`, непустой `required` со всеми
полями и конкретные типы — именно это делает `data` строго проверяемым.

Правило `entity_key` поддерживает два режима. Режим `fixed_vocab` требует, чтобы ключ был из словаря `vocabulary`;
присланные синонимы из `synonyms` приводятся к каноническому ключу, а близкие по смыслу значения подбираются по
эмбеддингу, если их близость выше порога. Режим `slug` нормализует ключ в латинский slug: транслитерация кириллицы,
нижний регистр и дефисы (например «Нижний Новгород» становится `nizhniy-novgorod`).

## Версионирование

Каждое сохранение создаёт новую версию и активирует её, а прежняя активная версия уходит в архив. На домен в любой
момент приходится не более одной активной версии (это гарантирует частичный уникальный индекс в миграции). Факты,
записанные по старой версии, остаются прослеживаемыми через поле `schema_version` в их `metadata`.

## Файлы модуля

- `meta.js` — мета-схема определения домена, общий экземпляр валидатора `ajv` и функция `validateDefinition`.
- `registry.js` — чтение активной схемы с кэшем, сохранение новой версии, список доменов, `getEntitySpec`.
- `generate.js` — генератор черновика определения по названию домена через LLM.
- `validate.js` — `validateAndCanonicalize`: валидация `data`, кодовая нормализация и канонизация `entity_key`.
- `cli.js` — командная строка: `generate`, `save`, `list`, `show`.
