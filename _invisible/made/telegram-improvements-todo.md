# Telegram-бот: задачи на доработку

Документ описывает две доработки Telegram-адаптера (`src/telegram.js`): событийную доставку исходящих
сообщений через механизм PostgreSQL `LISTEN/NOTIFY` и параллельную обработку входящих сообщений
с сохранением порядка внутри одного чата. Обе задачи нацелены на снижение задержек и переход
от опроса по таймеру к реакции по событию там, где это возможно.

## Задача 1. Событийная доставка из очереди через PostgreSQL LISTEN/NOTIFY

### Зачем

Сейчас функция `drainOutbox()` вызывается в `workerLoop` по таймеру — каждые `WORKER_INTERVAL_MS`
(по умолчанию 5 секунд). Это даёт две проблемы. Во-первых, исходящее сообщение (проактивное или
напоминание планировщика) ждёт в очереди `mem.notification_outbox` до пяти секунд, прежде чем уйдёт
пользователю. Во-вторых, опрос таблицы происходит вхолостую даже когда очередь пуста — лишние запросы
к базе данных раз в пять секунд при любом числе ботов и пользователей.

Механизм `LISTEN/NOTIFY` в PostgreSQL позволяет базе данных сама уведомлять приложение о событии.
При добавлении записи в очередь база посылает уведомление, и адаптер опустошает очередь немедленно,
а не по таймеру. Задержка доставки падает с величины «до пяти секунд» до десятков миллисекунд.

### Что сделать

1. **SQL-триггер на таблице очереди.** Добавить новую миграцию (например, `migrations/004_outbox_notify.sql`),
   которая создаёт функцию и триггер: при вставке строки со статусом `pending` база вызывает
   `pg_notify('outbox_new', NEW.id::text)`. Канал уведомления — `outbox_new`, полезная нагрузка —
   идентификатор записи (не обязателен для логики, но удобен для отладки и точечной доставки).

   ```sql
   CREATE OR REPLACE FUNCTION mem.notify_outbox() RETURNS trigger AS $$
   BEGIN
     IF NEW.status = 'pending' THEN
       PERFORM pg_notify('outbox_new', NEW.id::text);
     END IF;
     RETURN NEW;
   END;
   $$ LANGUAGE plpgsql;

   DROP TRIGGER IF EXISTS trg_notify_outbox ON mem.notification_outbox;
   CREATE TRIGGER trg_notify_outbox
     AFTER INSERT ON mem.notification_outbox
     FOR EACH ROW EXECUTE FUNCTION mem.notify_outbox();
   ```

2. **Отдельное подключение-слушатель в `src/telegram.js`.** Уведомления приходят на конкретное
   соединение, а не через пул, поэтому нельзя использовать общий хелпер `query` из `src/db.js`.
   Нужно взять выделенного клиента (`getPool().connect()`), выполнить на нём `LISTEN outbox_new`
   и подписаться на событие `notification`. По уведомлению — сразу вызвать `drainOutbox()`.

   ```js
   import { getPool } from './db.js';

   async function startOutboxListener() {
     const client = await getPool().connect();
     client.on('notification', () => { drainOutbox().catch((e) =>
       console.error('Ошибка событийного слива очереди:', e.message)); });
     client.on('error', (e) => console.error('Ошибка соединения-слушателя очереди:', e.message));
     await client.query('LISTEN outbox_new');
     return client;
   }
   ```

3. **Защита от потери уведомлений.** Уведомления `LISTEN/NOTIFY` не переживают обрыв соединения:
   если слушатель отвалился и переподключился, события за время простоя потеряны. Поэтому таймерный
   проход `drainOutbox` в `workerLoop` оставить как страховку, но сделать редким — раз в 30–60 секунд
   вместо пяти. Это гарантирует доставку даже при пропущенном уведомлении или после перезапуска,
   почти не нагружая базу.

4. **Переподключение слушателя.** На событии `error`/закрытия соединения слушателя — переоткрывать
   его с небольшой паузой (например, 3 секунды) и заново выполнять `LISTEN`. Сразу после
   переподключения один раз вызвать `drainOutbox()`, чтобы забрать накопившееся за время простоя.

5. **Защита от гонок.** Событие и таймер могут вызвать `drainOutbox` одновременно. Добавить флаг
   «слив уже идёт» (простой `let draining = false`) и при повторном входе выходить сразу, либо
   полагаться на то, что выборка использует `FOR UPDATE SKIP LOCKED` (см. ниже).

6. **Конкурентная безопасность выборки (желательно).** Если процессов-доставщиков может быть больше
   одного, заменить `SELECT … WHERE status = 'pending'` на `SELECT … FOR UPDATE SKIP LOCKED`, чтобы
   два процесса не отправили одно и то же сообщение дважды. В рамках одного процесса достаточно флага
   из пункта 5.

### Критерии готовности

- Проактивное сообщение уходит в Telegram в течение долей секунды после попадания в очередь.
- При остановленном слушателе сообщения всё равно доставляются страховочным таймером (проверить:
  отключить слушатель, вставить запись вручную, убедиться, что она ушла на следующем таймерном проходе).
- Обрыв соединения слушателя приводит к переподключению и доставке накопленного, без дублей.

## Задача 2. Параллельная обработка входящих сообщений

### Текущее поведение (проблема)

В `pollLoop` обновления обрабатываются строго последовательно:

```js
for (const update of updates) {
  offset = update.update_id + 1;
  if (update.message) await handleUpdate(update.message);
}
```

Параллелизма нет вообще, даже между разными пользователями. Пока агент обрабатывает одно сообщение
(а это секунды, основная часть — ответ LLM), все остальные сообщения ждут, и даже следующий вызов
`getUpdates` откладывается до конца пакета. При нескольких активных пользователях это заметная задержка.

### Целевая модель

- **Между разными чатами — параллельно.** Сообщения разных пользователей независимы: у них разные
  `user_id`, между ними нет гонок записи в память и нет смысловой зависимости. Их можно и нужно
  обрабатывать одновременно.
- **Внутри одного чата — строго последовательно.** Два подряд идущих сообщения одного пользователя
  нельзя обрабатывать одновременно: возникнут гонки записи в общую память этого пользователя, а второе
  сообщение часто смыслово зависит от результата первого. Поэтому на каждый чат — своя последовательная
  цепочка, и порядок ответов сохраняется автоматически.

Такая модель снимает необходимость в «массиве результатов с упорядоченным сливом»: порядок реплик
внутри чата обеспечивается самой последовательной цепочкой, а между чатами порядок не важен.

### Реализация: очередь-цепочка на каждый чат

Держать `Map<chatId, Promise>` — «хвост» последовательной цепочки этого чата. Новое сообщение
подвешивается за хвост: оно начнёт обрабатываться только после завершения предыдущего сообщения
того же чата, но независимо от других чатов.

```js
const chatChains = new Map();   // chatId -> Promise (хвост последовательной цепочки чата)

function enqueue(chatId, message) {
  const prev = chatChains.get(chatId) || Promise.resolve();
  // Следующее звено стартует после предыдущего ИГНОРИРУЯ его ошибку, чтобы один сбой не рвал цепочку.
  const next = prev.catch(() => {}).then(() => handleUpdate(message));
  chatChains.set(chatId, next);
  // Очистка: когда цепочка дошла до этого звена и оно последнее — убрать ключ, чтобы Map не рос.
  next.finally(() => { if (chatChains.get(chatId) === next) chatChains.delete(chatId); });
  return next;
}
```

В `pollLoop` больше не ждём обработку — только ставим в очередь и сразу продолжаем опрос:

```js
for (const update of updates) {
  offset = update.update_id + 1;
  if (update.message) enqueue(update.message.chat.id, update.message);
}
```

Теперь `getUpdates` вызывается без задержки на обработку, разные чаты идут параллельно, а внутри
чата — по порядку.

### Ограничение параллелизма (обязательно)

Без ограничения сотня одновременных чатов даст сотню параллельных запросов к LLM-прокси и положит его.
Нужен предел одновременных тяжёлых обработок (например, 5–10). Простейший вариант — семафор (счётчик
свободных слотов), который захватывается на время вызова `handleMessage` внутри `handleUpdate`.
Альтернатива — готовая маленькая библиотека вроде `p-limit`, но ради одной функции можно обойтись
самописным семафором на промисах, чтобы не добавлять зависимость. Значение предела вынести в
переменную окружения (например, `TELEGRAM_MAX_CONCURRENCY`, по умолчанию 5).

Важно: предел общий по всем чатам, но он ограничивает только одновременность; порядок внутри чата
по-прежнему гарантируется цепочкой из `chatChains`.

### Опционально на будущее: упорядоченная выдача при параллелизме внутри чата

Если когда-нибудь понадобится распараллелить тяжёлые подэтапы внутри одного чата (например, считать
несколько вариантов ответа), порядок реплик придётся восстанавливать явно: собирать результаты в
буфер по порядковому номеру сообщения и отправлять строго по возрастанию номера, не раньше, чем готов
очередной по порядку. Это и есть тот самый «массив с упорядоченным сливом сверху вниз». Сейчас он не
нужен и в задачу не входит — фиксируем как возможное расширение, чтобы не усложнять базовую версию.

### Замечание про самую большую задержку

Параллелизм убирает задержку от очереди обработки, но **основная задержка — это сам LLM** (ответ
gpt-5.4-mini занимает порядка 5–10 секунд на текущем прокси). Если нужен быстрый отклик, отдельно
стоит рассмотреть переключение `MAIN_MODEL=gpt-4o-mini` (порядка 1.2 секунды по заметкам проекта)
и потоковую отправку ответа частями через `editMessageText`. Это вне рамок данной задачи, но важно
для понимания: распараллеливание ускоряет пропускную способность при многих пользователях, а не сам
ответ одному пользователю.

### Критерии готовности

- Сообщения разных пользователей обрабатываются одновременно (проверить: два чата пишут почти
  одновременно — оба получают ответы, не дожидаясь друг друга).
- Два сообщения одного чата подряд обрабатываются по порядку, ответы приходят в исходном порядке.
- Число одновременных обращений к LLM не превышает заданный предел `TELEGRAM_MAX_CONCURRENCY`.
- Ошибка обработки одного сообщения не рвёт цепочку чата: следующее сообщение этого чата всё равно
  обрабатывается.
- `Map` цепочек не растёт неограниченно: завершённые чаты удаляются из неё.
