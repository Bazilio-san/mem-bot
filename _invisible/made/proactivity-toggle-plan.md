# План: управление проактивностью (глобально + на пользователе + потриггерно)

## Текущее состояние (как работает сейчас)

- **Глобальный рубильник.** Флаг `config.proactive.enabled` читается из переменной окружения `PROACTIVE_ENABLED`
  (по умолчанию выключено), `src/config.js:62`. Фоновый воркер вызывает проверку триггеров только при включённом
  флаге (`src/scheduler-run.js:22`, `src/telegram.js:282`), и сама `checkProactiveTriggers` делает ранний выход при
  выключенном флаге (`src/pipeline/proactive.js:86`). Требование «глобально выключено — не работает ни у кого»
  уже выполнено.
- **Срабатывание у пользователя.** Единого признака «проактивность включена у пользователя» в базе нет. Есть только
  потриггерный флаг `enabled` в `mem.proactive_triggers` (по умолчанию `true`, `migrations/002_proactive.sql:29`).
  При включённом глобальном контуре на каждое сообщение в `src/agent.js:44-52` идемпотентно создаётся набор из
  четырёх триггеров, и все они создаются включёнными. Значит новый пользователь сейчас получает проактивность
  включённой — это противоречит требованию.
- **Выборка получателей.** `listUsersWithTriggers` (`src/repo.js:171`) берёт всех, у кого есть хотя бы один
  включённый триггер; признака на уровне пользователя в условии нет.
- **Меню телеграм-бота.** Команды заданы статической константой `BOT_COMMANDS` (`src/telegram.js:79`) и
  регистрируются один раз глобально (`src/telegram.js:325`): `/start`, `/help`, `/domain`, `/proactive` (ручной
  запуск для отладки). Команд `/proactivity-on` и `/proactivity-off` нет; меню не зависит ни от глобального флага,
  ни от состояния пользователя.

## Целевая модель поведения

Два уровня управления, оба хранятся в БД:

1. **Глобальный рубильник** — `config.proactive.enabled` (как сейчас). Выключен глобально — не работает ни у кого.
2. **Признак на пользователе** — новая колонка `mem.users.proactivity_enabled` (по умолчанию `false`).
   Мастер-переключатель конкретного пользователя.
3. **Потриггерный признак** — существующий `mem.proactive_triggers.enabled`, но теперь триггеры создаются
   выключенными (`false`) и именно в момент включения проактивности пользователем, а не на первое сообщение.

Состояния пользователя при глобально включённой проактивности:

- мастер `false` → проактивности нет, триггеров в БД может не быть вовсе;
- мастер `true`, все триггеры `false` → пользователь включил контур, но ни один повод не активирован (ничего не шлётся);
- мастер `true`, часть триггеров `true` → работают только включённые поводы.

Отдельная мастер-колонка нужна: состояние «включил, но все поводы выключил» нельзя отличить от «выключил» без неё.

## Изменения по файлам

### 1. Миграция `migrations/007_proactivity_flag.sql` (новая)

- `ALTER TABLE mem.users ADD COLUMN IF NOT EXISTS proactivity_enabled boolean NOT NULL DEFAULT false;`
- Идемпотентно, в стиле миграции 005.

### 2. `src/repo.js`

- `ensureDefaultTriggers` — создавать триггеры с `enabled = false` (сейчас полагается на табличный `DEFAULT true`;
  нужно вставлять `false` явно). Вариант: параметр `enabled` со значением по умолчанию `false`, чтобы сид-скрипт мог
  по-прежнему создавать включённые.
- `listUsersWithTriggers` — добавить в условие `u.proactivity_enabled = true` (рубильник пользователя над
  потриггерным `enabled`).
- Новые функции:
  - `setUserProactivity(externalId, enabled)` — переключает мастер-флаг; при включении создаёт набор триггеров
    (все `false`).
  - `getProactivityState(externalId)` — возвращает мастер-флаг и список триггеров с их `enabled` (для подменю).
  - `setTrigger(externalId, triggerType, enabled)` — переключение одного триггера.

### 3. `src/agent.js`

- Убрать авто-создание включённых триггеров на каждое сообщение (`src/agent.js:44-52`). Создание переносится в
  момент команды `/proactivity-on`. Иначе требование «по умолчанию выключено» нарушается.

### 4. `src/pipeline/proactive.js` и `src/pipeline/events.js`

- Логику срабатывания не трогаем — она берёт пользователей из `listUsersWithTriggers`, который теперь сам отфильтрует
  по мастер-флагу. Проверено, что `events.js` использует ту же выборку (`src/pipeline/events.js:113`).

### 5. `src/telegram.js` — команды и динамическое меню

- Включить приём `callback_query` в длинном опросе (`allowed_updates: ['message','callback_query']`, сейчас только
  `message`, `src/telegram.js:307`) и добавить ветку обработки нажатий инлайн-кнопок.
- Команды:
  - `/proactivity-on` — ставит мастер `true`, создаёт триггеры (все `false`), обновляет меню чата.
  - `/proactivity-off` — ставит мастер `false`, обновляет меню чата.
  - `/proactivity` — открывает экранное подменю (инлайн-клавиатура): кнопка на каждый тип триггера с галочкой
    состояния (нажатие переключает `enabled`) и отдельная кнопка «Выключить проактивность» (мастер `false`).
    Нажатия приходят как `callback_query`, обрабатываются через `answerCallbackQuery` и перерисовку клавиатуры
    `editMessageReplyMarkup`.
- Динамическое меню per-chat через `setMyCommands` со `scope: { type: 'chat', chat_id }`, пересчитывается после
  каждого сообщения и после переключений:
  - глобально выключено → базовые команды без проактивности;
  - глобально включено, мастер пользователя `false` → базовые + `/proactivity-on`;
  - глобально включено, мастер `true` → базовые + `/proactivity-off` + `/proactivity`.
- Глобальный `setMyCommands` при старте (`src/telegram.js:325`) оставить как запасной набор по умолчанию
  (базовые + `/proactivity-on`, если глобально включено).
- Отладочную команду `/proactive` (ручной запуск триггера) предлагается оставить как есть для тестов; при желании
  убрать из видимого меню.

## Утверждённые решения по деталям

- Подписи кнопок подменю — отображаемые русские имена вместо технических ключей триггеров: `inactivity` →
  «Неактивность», `daily_checkin` → «Ежедневное приветствие», `goal_reminder` → «Напоминание о цели»,
  `welcome_back` → «Возвращение». Технические ключи остаются в БД и в `callback_data`, пользователю показываются
  только русские подписи (с галочкой текущего состояния).
- Формат `callback_data` — короткие коды: `pa:t:<type>` для переключения одного триггера (например,
  `pa:t:inactivity`) и `pa:off` для выключения мастер-флага. Коды заведомо укладываются в лимит Telegram в 64 байта.
