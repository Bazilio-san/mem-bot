# Предложение: доделка cron- и rrule-расписаний в планировщике

## Проблема

В планировщике (`src/pipeline/scheduler.js`) вычисление следующего запуска задачи выполняет функция
`calculateNextRun` (строки 81–89). Для разовых и интервальных задач она работает корректно, но для расписаний вида
`cron` и `rrule` стоит заглушка — фиксированный суточный шаг:

```js
// cron/rrule в этом MVP упрощены до суточного шага (в проде — croniter/rrule).
return new Date(Date.now() + 86400000);
```

Из-за этого произвольные расписания не отрабатывают как заявлено. Например, просьба «напоминать каждый будний день в
9:00» приведёт к тому, что задача будет срабатывать просто раз в сутки от момента предыдущего запуска, без привязки к
конкретному времени, дням недели и часовому поясу пользователя.

Важно: схема базы данных к полноценной поддержке уже готова. В таблице `mem.scheduled_tasks` (миграция
`migrations/001_init.sql`) есть колонки `cron_expr`, `rrule` и `timezone` (по умолчанию `Europe/Moscow`), а тип
`mem.task_schedule_kind` включает значения `cron` и `rrule`. Не хватает только вычислительной части в коде.

## Цель

Сделать так, чтобы задачи с `schedule_kind = 'cron'` и `schedule_kind = 'rrule'` вычисляли следующий запуск по
настоящему выражению расписания, с учётом часового пояса пользователя из поля `timezone`.

## Что предлагается сделать

### 1. Подключить библиотеки разбора расписаний

В проекте сейчас нет ни одной зависимости для разбора cron/rrule (`package.json` содержит только `dotenv`, `openai`,
`pg`). Предлагается добавить две библиотеки на чистом JavaScript, без нативных сборок:

- **`cron-parser`** — разбор cron-выражений с поддержкой часовых поясов (опция `tz`). Нужна для `schedule_kind='cron'`.
- **`rrule`** — разбор и развёртывание правил повторения формата iCalendar RRULE. Нужна для `schedule_kind='rrule'`.

Обе зависят только от JavaScript и совместимы с текущим окружением Node.js. Часовые пояса cron-parser берёт из
встроенной в Node поддержки Intl, отдельной базы данных временных зон ставить не требуется.

### 2. Переписать вычисление следующего запуска

Превратить `calculateNextRun` в функцию, которая разбирает реальное выражение и считает первый запуск строго **после
заданного момента** (опорной точкой берётся `last_run_at` или текущее время), с учётом `timezone` задачи:

- для `cron` — разобрать `cron_expr` через `cron-parser` с опцией часового пояса и взять ближайшую будущую дату;
- для `rrule` — разобрать строку `rrule` и через метод `after(date)` получить следующую дату;
- если выражение пустое или не разбирается — не падать молча: вернуть безопасный отложенный запуск и записать ошибку
  разбора, чтобы задача не зацикливалась на невалидном расписании.

Аналогично доработать `computeFirstRun` (строки 72–78), которая сейчас для cron/rrule отдаёт `new Date()` (немедленный
запуск): первый запуск тоже должен считаться по выражению расписания, а не «сразу».

### 3. Учесть часовой пояс на всём пути

Сейчас `timezone` сохраняется в задаче, но в вычислении следующего запуска не используется. После доработки часовой
пояс пользователя должен участвовать в расчёте и для cron, и для rrule, чтобы «9:00» означало девять утра по времени
пользователя, а не по времени сервера.

### 4. Согласовать с извлечением задач

Функция `extractTask` (строки 39–53) и описание инструмента `scheduler_create_task` в `src/pipeline/tools.js` уже
позволяют модели вернуть `cron_expr`/`rrule`. Нужно убедиться, что модель получает понятную инструкцию формировать
корректные cron- и rrule-выражения, и при необходимости дополнить системный промпт примерами (например, «каждый будний
день в 9:00» соответствует cron-выражению `0 9 * * 1-5`).

## Тесты

Расширить `tests/run.js`:

- cron-задача с конкретным временем (`cron_expr` вида `0 9 * * 1-5`) вычисляет следующий запуск именно на ближайший
  будний день в 9:00 по заданному часовому поясу, а не «через сутки»;
- rrule-задача вычисляет следующий запуск по правилу повторения;
- невалидное выражение не приводит к падению воркера и не зацикливает задачу (срабатывает безопасный откат);
- проверка часового пояса: одна и та же cron-строка при разных `timezone` даёт разные моменты запуска в UTC.

## Оценка объёма

Изменения локализованы в одном модуле (`src/pipeline/scheduler.js`), плюс две новые зависимости и тесты. Схема базы и
интерфейс инструмента менять не требуется — поля уже есть. Основной риск — корректная работа с часовыми поясами и
крайними случаями расписаний, поэтому тесты на временные зоны обязательны.
