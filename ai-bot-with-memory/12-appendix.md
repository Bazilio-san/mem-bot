## Упрощения и доделки

Эта таблица фиксирует места, где документация описывает уже заложенную архитектуру, но текущий MVP реализует её проще.
Пункты не ломают базовые 36 проверок и слой проактивности, но важны для честного статуса реализации.

| Что | Текущий статус | Где продолжать |
|-----|----------------|----------------|
| Поджатие истории диалога | реализовано: модули `history-*`, миграция `003`, слой тестов `layerHistory`; остаются точный токенизатор (`tiktoken`) и опциональное послойное досжатие | [13-history-compression.md](13-history-compression.md) |
| Очередь записи памяти | `memory_jobs` создана, но запись идёт промисом внутри процесса ответа | отдельный memory worker |
| Отправитель уведомлений | `notification_outbox` наполняется, внешней доставки в Telegram/email/push пока нет | transport worker |
| `cron` и `rrule` | поля есть, но расписание сведено к суточному шагу | разбор cron/rrule в `scheduler.js` |
| Сложное слияние фактов | конфликт решается правилами, без отдельной модели `MergeDecision` | модельное решение конфликта |
| Per-domain-схемы | слой описан как следующий контракт для `data jsonb`, но не является частью MVP | `mem.domain_schemas`, `ajv` |
| Источник событий | события проактивности стартуют с встроенной заглушки новостей | внешний API событий |
| Тест событий | проверяет смоук-проход и дедупликацию, не детерминирует релевантность модели | ручной сценарий или фикстуры |
| Триггер возврата | веб-события нет; `welcome_back` работает по паузе, команде или проходу воркера | сигнал web-клиента |
| Воркер проактивности | совмещён с `src/scheduler-run.js`, отдельного процесса нет | отдельный процесс при масштабе |
| Оценка вовлечённости | `user_engagement` выдаёт модель и сглаживание; метрика грубая | дополнительные эвристики |
| Потоковая передача ответа | обычный ответ без streaming | streaming в `agent.js` и `llm.js` |
| Кэширование системного промпта | стабильный промпт отделён, но кэширование не подключено | кэш неизменной части промпта |
| JSON-логирование | есть `DEBUG`-трассировка и журналы таблиц, но нет единого JSON-логгера | `LOG_LEVEL`, correlation ids |
| Очистка памяти по расписанию | регулярной задачи `memory_cleanup` нет; устаревшие и давно неиспользуемые факты не архивируются | задача `memory_cleanup` в `scheduler.js` |
| Связь безопасного факта с секретом | таблица `memory_secure_links` создана, но пока не используется для явной связи безопасного факта с секретной записью | заполнение `memory_secure_links` в `secure.js` |
