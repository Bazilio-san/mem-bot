# Чат-бот с долговременной памятью

Рабочий агентский чат-бот с памятью на PostgreSQL, реализованный по архитектуре из
`_prompts/agent_memory_architecture.md`. Поддерживает пять видов памяти, приватность
чувствительных данных, инструменты, планировщик напоминаний и асинхронную запись фактов.


## Что умеет бот

Бот реализует все двенадцать критериев готовности из задания. Ниже — где это в коде.

| Критерий | Реализация |
|----------|------------|
| Пять видов памяти (диалог, профиль, предметная, защищённая, напоминания) | таблицы `mem.*` в `migrations/001_init.sql` |
| Не сохраняет мусор | порог важности и уверенности в `src/pipeline/merge.js` (`passesAutoSave`) |
| Не раздувает промпт (10–30 фактов) | жёсткие лимиты в `src/pipeline/retrieve.js` (`LIMITS`) |
| Новое сообщение важнее старой памяти | правило в системном промпте `src/agent.js` (`MAIN_SYSTEM`) |
| Обновление фактов без дублей | `decideMerge`/`updateMemory`/`archiveMemory` в `src/pipeline/merge.js` |
| Факт / намерение / задача разделены | `scope`/`memory_kind` в памяти, отдельный планировщик `scheduled_tasks` |
| Чувствительные данные — с подтверждением | `src/pipeline/secure.js` (шифрование AES-256-GCM, согласие) |
| Не раскрывает лишние данные | в промпт идёт только `redacted_summary`, полное значение — через инструмент |
| Вызов инструментов | `src/pipeline/tools.js` + цикл инструментов в `src/agent.js` |
| Управление памятью в диалоге | инструменты `memory_list`, `memory_forget_entity`, `memory_forget_all` (`src/pipeline/tools.js`) поверх `src/pipeline/admin.js` |
| Планировщик | `src/pipeline/scheduler.js` (захват задач, повторы, перепланирование) |
| Устойчивость к вредной памяти | блок `MEMORY_CONTEXT` подаётся как справочные данные, а не команды |
| Скорость | классификация → быстрая выборка → ответ → асинхронная запись фактов после ответа |

## Требования

- Node.js 22+
- PostgreSQL 16 с расширениями `pgvector` и `pgcrypto` (проверено: оба доступны)
- Заполненный `.env` (строка подключения к БД и `OPENAI_API_KEY`; `OPENAI_BASE_URL` нужен только для прокси)

## Установка и запуск

```bash
npm install            # установка зависимостей (openai, pg, dotenv)
npm run migrate        # создаёт базу agent_mem, схему mem, все таблицы и индексы
npm run chat           # интерактивный чат в терминале
npm run scheduler      # отдельный воркер планировщика напоминаний
npm test               # полный прогон проверок (36 проверок, см. ниже)
```

В интерактивном чате доступны команды: `/domain <ключ>` — сменить специализацию
(`general`, `flight_search`, `joke_teller`, `math_tutor`), `/tick` — прогнать планировщик
вручную, `/exit` — выход.

## Используемые модели

Модели соответствуют рекомендациям архитектуры и проверены скриптом `tests/check-llm.js`
(все отвечают: чат, JSON, вызов инструментов, эмбеддинги):

- основной агент и извлечение фактов — `gpt-5.4-mini`;
- классификация запроса — `gpt-5.4-nano`;
- эмбеддинги для смыслового поиска — `text-embedding-3-small` (1536 измерений).

Любую модель можно переопределить через переменные окружения `MAIN_MODEL`,
`AUX_MODEL`, `EXTRACT_MODEL`, `EMBED_MODEL`. Если `OPENAI_BASE_URL` задан, запросы идут в OpenAI-совместимый
прокси по этому адресу; если не задан, используется прямой OpenAI API. Например, для самого быстрого отклика
можно задать `MAIN_MODEL=gpt-4o-mini` (на проверенном прокси отвечает примерно за 1.2 с).

Проверить доступность и скорость любой модели: укажи её в константе `MODEL` в
`tests/check-llm.js` и запусти `npm run check:llm`.

## Структурный вывод модели

Структурный вывод реализован через режим `json_object` с описанием JSON Schema прямо
в системном промпте (`src/llm.js`, функция `chatJSON`). Строгий режим `json_schema`
сам по себе поддерживается провайдером, но требование OpenAI strict mode — `additionalProperties:false`
у всех вложенных объектов — несовместимо с нашими свободными полями `data` и `entities`
(произвольные ключи под конкретный домен). Поэтому строгий режим для этих схем не используется.
Это проверяется тестом «свободное поле в strict отвергается» в `tests/check-llm.js`.

## Пайплайн ответа

1. Классификация входящего сообщения (`pipeline/classify.js`).
2. Выборка минимального релевантного и безопасного набора фактов (`pipeline/retrieve.js`).
3. Сборка компактного `MEMORY_CONTEXT` (отдельным системным сообщением).
4. Ответ модели с циклом вызова инструментов (`agent.js`).
5. Сохранение сообщений диалога.
6. Асинхронное извлечение новых фактов и их слияние с памятью (`extract.js` + `merge.js`).

## Проверки

`npm test` прогоняет 36 проверок по слоям: структура БД, извлечение фактов на наборе
`tests/memory_cases.json`, двенадцать обязательных тестов задания, приватность
защищённых данных и полный сценарий диалога с репетитором. Тесты используют реальную
БД и реальные модели через выбранный OpenAI-compatible endpoint.

## Структура проекта

```
migrations/001_init.sql     схема памяти (13 таблиц, индексы, типы)
src/config.js               конфигурация и выбор моделей
src/db.js                   пул подключений PostgreSQL
src/llm.js                  клиент LLM (чат, структурный вывод, эмбеддинги)
src/migrate.js              бутстрап базы и применение миграций
src/repo.js                 пользователи, диалоги, сообщения, журнал инструментов
src/agent.js                главный пайплайн ответа
src/cli.js                  интерактивный чат
src/scheduler-run.js        отдельный воркер планировщика
src/pipeline/classify.js    классификация запроса
src/pipeline/retrieve.js    выборка памяти, приватность, минимизация
src/pipeline/extract.js     извлечение кандидатов в память
src/pipeline/merge.js       дедупликация, обновление, запись фактов
src/pipeline/secure.js      защищённая память (шифрование, согласие)
src/pipeline/scheduler.js   создание задач, планировщик, повторы
src/pipeline/tools.js       инструменты агента
src/pipeline/admin.js       просмотр и удаление памяти пользователем
tests/run.js                комплексная проверка
tests/memory_cases.json     набор кейсов извлечения
```
